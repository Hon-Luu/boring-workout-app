Section 1 — Must-Pass Content Criteria
🗣 1. Voice & Tone Foundation 8 tests ▼
CV-001
No Cheerleading
Zero instances of: "Amazing!", "Crushing it!", "You're killing it!", "So proud of you!", "Beast mode!", "You've got this!", "Keep going champ!" anywhere in phrase_bank.json, HONMessageLibrary, or any Swift string. Search all user-facing string sources.
Pass: Full-text search returns zero matches for these phrases and their common variants across all source files.
CV-002
Specificity Requirement
No coaching phrase can be fully true for every user in every state. Each phrase must reference at least one data point: exercise name, weight, reps, trend direction, gap duration, or readiness level. Phrases that work for any user in any state are generic and must be revised.
Pass: Randomly sample 20 phrases from phrase_bank.json; all 20 reference at least one data-specific element. Zero fully-generic phrases in the sample.
CV-003
Direct Statement Rule
Coaching language uses declarative statements, not questions. Not "How did that feel?" but "Note how you feel." Not "Are you ready to push?" but "Push when ready." Questions shift the dynamic from coach to companion — H.O.N. coaches, it does not ask permission.
Pass: Zero coaching phrases end with a question mark across phrase_bank.json and all coaching output strings.
CV-004
No Clinical Framing
Zero uses of: "compliance", "adherence", "protocol", "regimen", "optimization", "metrics", "performance indicator", "biometric" in user-facing text. Clinical language creates distance between the athlete and the work. H.O.N. speaks like a coach, not a research paper.
Pass: Search across all user-facing strings returns zero matches for clinical vocabulary. Developer comments and internal code identifiers are excluded.
CV-005
Identity-Affirming Language
The user is referred to as doing the work, not being monitored. "You trained 4 days this week" not "Your training frequency was 4 sessions." Active voice, athlete as subject. The difference is whether the user is an actor in their own story or a data source being reported on.
Pass: Sample 10 coaching notes across different engine outputs; all use active voice with the user as grammatical subject. Zero passive constructions where the system is the actor.
CV-006
Sentence Length
No coaching sentence exceeds 20 words. Short sentences create impact; long sentences feel like reports. This applies to every sentence in every phrase in phrase_bank.json, HONMessageLibrary, ReadinessEngine notes, and TrainerTabView plan introductions. Count compound sentences separately.
Pass: The longest sentence in any coaching output is ≤20 words when measured by simple word count. No exceptions.
CV-007
No Hedging
Coaching language does not hedge with "try to", "might want to", "perhaps", "maybe", "could be". H.O.N. says what it means. Hedging language suggests uncertainty — the coaching should be confident and direct even when recommending a cautious approach ("Ease in today" not "Maybe try going lighter today").
Pass: Zero hedging phrases found in any coaching output across all engines and phrase banks. Search includes "try to", "might want", "perhaps", "maybe consider", "could be".
CV-008
Consistent Register
The coaching voice does not shift between formal and casual within the same session. If the app says "You've hit 5 reps" in one note, it does not say "Nailed it" in the next. Inconsistent register breaks the persona and signals that different people wrote different parts of the app — even if they did.
Pass: Read 5 consecutive coaching outputs from a single session aloud; no register shift detected. A neutral reviewer should not be able to tell that different phrase categories were assembled.
📄 2. Phrase Bank Quality 8 tests ▼
CV-009
Minimum Phrase Count
Every phrase category in phrase_bank.json has at least 8 variants. The rotation window is 8 uses — a category with fewer than 8 phrases will always repeat within the window, defeating the purpose of rotation. This is a hard floor, not a target.
Pass: All JSON keys with phrase arrays have length ≥8. Any key with fewer than 8 phrases is a blocking issue.
CV-010
No Duplicate Meaning
Within a single phrase category, no two phrases communicate the same meaning in different words. Synonyms are not variants — they are duplicates. "You hit all your reps" and "You completed every rep" are the same phrase with different words. True variants communicate meaningfully different information or perspectives.
Pass: Review each category; no two phrases are paraphrases of each other. A reviewer should be unable to compress any two phrases into a single statement without losing meaning.
CV-011
Phrase Specificity Gradient
Within each category, phrases range from data-light (works with minimal context) to data-heavy (requires rich historical data). The engine picks based on data availability. A new user with one session should get a meaningful phrase, not a fallback error or a phrase that references non-existent history.
Pass: At least one phrase per category is usable for a new user with minimal data. At least one phrase per category references rich historical data. The gradient is intentional, not accidental.
CV-012
Rotation Window Effectiveness
The pick() rotation must prevent the same phrase from appearing more than once in 8 consecutive uses. This is the core mechanism preventing coaching text from feeling repetitive. Verify that the UserDefaults tracking correctly advances the index and wraps at the category length.
Pass: Simulate 24 consecutive uses of a phrase category; no phrase repeats within any 8-use window. If a category has exactly 8 phrases, the first phrase reappears only at position 9, 17, etc.
CV-013
Tone Consistency Across Categories
The "history" key, "weight" key, "connective" key, and "whats_next" key all sound like the same coach. WorkoutNarrativeEngine assembles phrases from multiple categories into a single coaching output — if the categories have different registers or voices, the assembled text will feel stitched together rather than authored.
Pass: Pull one phrase from each key; read them as a single assembled statement. No tonal inconsistency. A neutral reader should not detect that the phrases came from separate JSON keys.
CV-014
Empty Category Handling
If a phrase category is exhausted or unavailable (e.g., the JSON key is missing or empty), the engine must fail gracefully — omitting the slot rather than showing a placeholder, empty string, or raw key name. The user should never see "[MISSING]", a blank coaching note, or a crash.
Pass: Empty a phrase category in phrase_bank.json; verify that the assembled coaching output omits that slot silently. No "[MISSING]", no blank string, no crash. The remaining phrase slots assemble correctly.
CV-015
Cardio Phrase Parity
Cardio coaching phrases — end-of-session summaries and in-session EMOM transition cues — must match strength coaching in voice, quality, and quantity. Cardio is not a second-class feature. A user who trains cardio exclusively should receive coaching that feels as considered as what a strength-focused user receives.
Pass: Cardio phrase section exists in phrase_bank.json with ≥10 end-of-session phrases and ≥6 EMOM/transition phrases. All pass CV-001 through CV-008 independently.
CV-016
Phrase Freshness
Phrases must not reference specific dates, years, seasons, or real-world events that would become stale. "This winter" will be wrong in six months. "2026" will be wrong next year. "After the holiday season" excludes users in the southern hemisphere. Coaching text should be timeless.
Pass: No phrase contains "this year", "2026", "winter", "summer", "holiday", or any calendar-specific reference. Phrases read as true regardless of when they appear.
🎯 3. Coaching Output Coverage 8 tests ▼
CV-017
Post-Set Coaching — Full State Coverage
Every set-completion state must have distinct coaching copy: hit all reps, missed reps, exceeded reps, first time this exercise, return after long gap, return after short gap, stalled progression, drop set completed, to-failure set. Nine states — all nine require at least one distinct phrase. A state that falls through to a blank is a content gap.
Pass: Each of the 9 states has at least one distinct phrase in phrase_bank.json. No state falls through to an empty coaching output. Trigger all 9 states in sequence and verify output in each.
CV-018
Readiness Coaching — Tier Coverage
Every readiness tier must have a substantively different coaching note: high (≥80), moderate-high (65–79), moderate (50–64), low (<50). A coaching note that works for all four tiers is not tier-specific — it is generic. The note for score 42 should not be usable at score 85. ReadinessEngine.compute() must branch into distinct outputs per tier.
Pass: Generate coaching notes for scores 85, 72, 58, 42. All four are tier-appropriate and substantively different from each other. No two notes share more than one sentence.
CV-019
Trainer Plan Coaching — Plan-Specific Introduction
Every generated workout plan in TrainerTabView must have an introductory sentence explaining why this plan was suggested — based on readiness, last performance, or identified gaps. A generic "Here's your workout" is a content failure. The introduction should tell the user something the plan itself does not show them.
Pass: Generate plans at high readiness (≥80) and low readiness (<50). Both plans have distinct, plan-specific introductions. Neither introduction is usable for the other plan.
CV-020
Session Summary Coaching — Session-Specific Reference
Every workout completion in CardioSessionSummaryView or the strength session summary must reference at least one specific element of that session: an exercise name, set count, PR achieved, readiness alignment, or volume number. A generic "Good session" tells the user nothing about what they actually did.
Pass: Complete 3 different workouts (different exercises, durations, volumes). All 3 session summaries reference at least one session-specific data element. No two summaries are identical.
CV-021
Milestone Coaching — Weight Proportionality
Sessions 1, 10, 25, 50, 100, 200, and 365 each have a distinct milestone message in HONMessageLibrary. Messages must be proportional to the milestone size — Session 1 and Session 100 should feel like completely different events. Session 365 is a year of consistent training; treating it like Session 10 would be a content failure.
Pass: All 7 milestones have distinct messages. Session 100 message is notably more significant in tone and content than Session 1. Session 365 message reads as the most significant of all seven.
CV-022
Comeback Coaching — Gap-Appropriate Tone
Gap durations of 7–13 days, 14–29 days, and 30+ days each require distinct comeback messages in HONHabitEngine. No message may reference the gap duration numerically — "you've been away for 23 days" is clinical and potentially shaming. Each message must be tonally calibrated to the size of the gap without naming it.
Pass: Three gap durations produce three distinct messages. None says "you've been away for X days" or any numeric gap reference. All three are warm, non-judgmental, and action-oriented.
CV-023
Empty State Copy — On-Brand Across All Screens
Every screen that can have an empty state must have an on-brand, one-line H.O.N.-voice message. Screens to verify: Progress with no sessions, Trainer with no plan, Insights with no data, cardio with no logs, StrengthLab with no e1RM data. No empty state may say "No data", "Nothing here", "N/A", or leave the space blank.
Pass: All empty states across all tabs show on-brand copy that tells the user what to do next in one sentence. No generic system placeholder text anywhere in the app.
CV-024
Error Message Copy — Human and Actionable
All form validation errors must be in plain, non-technical language. No error may say "Invalid input", "Value out of range", "Null reference", or any developer-facing message. Errors must name the specific problem and tell the user what to do: "Weight must be greater than zero" not "Invalid weight value".
Pass: Trigger every validation error in the app (0 weight, 0 reps, invalid session name, etc.). All messages are human-readable, non-technical, and actionable. None could be mistaken for a system error.
✅ 4. Copy Completeness Audit 7 tests ▼
CV-025
Tab Bar Labels — Unambiguous Single Nouns
All 5 tab labels must be single nouns that communicate function without ambiguity. A user who has never seen the app should be able to predict what they'll find on each tab from the label alone. Labels like "More" or "Dashboard" that don't name the content fail this test.
Pass: Show the tab bar to 3 people unfamiliar with H.O.N. All correctly identify each tab's purpose from the label alone, without seeing the tab's content. 15/15 correct identifications required.
CV-026
NavigationTitle Clarity
Every primary screen must have a NavigationTitle that describes the screen's purpose in 1–2 words. No NavigationTitle may be "View", a company name, or a generic term that doesn't identify the screen. NavigationTitles that appear on multiple screens must be distinct from each other.
Pass: Navigate all 5 tabs and any nested primary screens. All NavigationTitles are descriptive, distinct, and 1–2 words. Zero instances of "View", "H.O.N.", or a repeated title on different screens.
CV-027
Button Label Specificity
All CTA buttons must use action verbs + object ("Log Set", "End Session", "Save Routine") not generic labels. "OK" does not tell the user what they're confirming. "Submit" does not tell the user what they're submitting. "Continue" does not tell the user where they're going. Every button should be self-descriptive.
Pass: No button label in the app is "OK", "Submit", "Yes", "No", or "Continue" without a qualifying object. Every button label describes the specific action it performs.
CV-028
Confirmation Dialog Copy
All confirmation dialogs for destructive actions must name the specific action and its consequence, not ask a generic question. "Are you sure?" is not a confirmation — it's a stall. "Delete this workout session? This cannot be undone." gives the user the information they need to decide.
Pass: All destructive-action confirmation dialogs reviewed. Each names the specific thing being deleted, ended, or lost. None uses "Are you sure?" without a qualifying clause that specifies the consequence.
CV-029
Onboarding Copy — Plain Language
Onboarding labels for goal, equipment, and fitness level must use plain language that requires no prior fitness knowledge. "Mesocycle optimization" fails. "Hypertrophy phase" fails. "Build strength" passes. A user who has never trained before must be able to select their options confidently without needing to Google a term.
Pass: Walk a non-fitness-expert through onboarding labels only (no interaction). They can explain each option in their own words. Zero labels require prior fitness knowledge to interpret correctly.
CV-030
Settings Labels — User-Centric Framing
All settings options must be labeled in terms of what the user gets, not what the system does. "Weight in kilograms" not "KG mode enabled." "Show previous session weights" not "Display historical performance data." The user should be able to predict the effect of each setting from its label without enabling it first.
Pass: All settings labels reviewed for user-centric framing. No label describes a system state; all describe a user outcome. A reviewer should be able to explain each setting's effect from the label alone.
CV-031
Notification Copy — Context-Specific
Any push notification templates must follow voice guidelines and name a specific context relevant to the user. "Time to work out!" is generic and indistinguishable from any other fitness app. "Your squat is ready for its next attempt" tells the user something only H.O.N. could know. Notifications that don't use user data should not be sent.
Pass: All notification templates reviewed. None are generic fitness reminders. Each references user-specific context (exercise, readiness state, or streak element). All pass CV-001 through CV-008.
🌐 5. Localization Readiness 6 tests ▼
CV-032
Strings Externalized
All user-facing strings must be in Localizable.strings or an equivalent externalized format — not hardcoded in Swift view files. Hardcoded strings cannot be translated. This includes button labels, empty state messages, error messages, coaching text assembled in views (as opposed to engines), and NavigationTitles.
Pass: Grep for hardcoded English strings in iOSApp/Views/ returns zero results (excluding developer comments, identifiers, and format specifiers). All user-visible text is externalized.
CV-033
No String Concatenation
Coaching phrases must not be built by concatenating translated fragments. Word order varies by language — "You lifted 100 kg" in English becomes "100 kg gehoben hast du" in German if fragments are concatenated in English order. All coaching outputs must use full-phrase strings with interpolation for data values, not subject + verb + object assembly.
Pass: All coaching outputs use complete sentence templates with %@ or equivalent interpolation for data values. No coaching string is assembled from separately translated subject, verb, and object fragments.
CV-034
Date and Number Formatting
All dates and numbers must use locale-aware formatters. No hardcoded "lbs" or "kg" in string literals — use the formatter output. No hardcoded date formats like "MM/dd/yyyy" — use DateFormatter with the user's locale. This applies to all displayed values: weights, distances, durations, set counts, and dates in the log.
Pass: Change device locale to German. All dates display in locale-appropriate format (e.g., "30. Mai 2026"). All units display using the locale's preferred unit system. Zero hardcoded "lbs" or "kg" visible in any UI string.
CV-035
Character Length Budget
UI strings must have a character budget that accounts for 40% expansion when translated to German or French. A button label that is 12 characters in English may be 17 characters in German — the button must accommodate this without truncation or layout breaking. This is a layout constraint, not just a copy guideline.
Pass: The longest English string in any fixed-width UI container (buttons, chips, labels) × 1.4 still fits in its container at the same font size without truncation or overflow. Verify on iPhone SE (smallest supported screen).
CV-036
Cultural Neutrality
No coaching phrase may assume a cultural context specific to one region. Sports idioms that are opaque outside their home culture ("knocked it out of the park", "hat trick"), references to local events, or assumptions about what "normal" training looks like in a specific country all fail this test. Coaching must be interpretable by a user anywhere.
Pass: All phrases reviewed for cultural neutrality. Zero region-specific sports idioms, cultural references, or assumptions about regional norms found. A reviewer from three different countries can interpret every phrase correctly.
CV-037
RTL Layout Preparedness
The app layout must remain usable when the system language is set to a right-to-left language (Arabic, Hebrew). SwiftUI natively handles most RTL mirroring, but custom layouts, absolute positioning, and icon directions must be verified. A broken RTL layout is a blocker for any market that uses RTL scripts.
Pass: Enable RTL in iOS Simulator (use a pseudolanguage or Hebrew locale). All 5 tabs render correctly. No text truncation from wrong-direction layout. No icon pointing the wrong direction. Navigation gestures work correctly.
📜 6. Voice Governance Process 5 tests ▼
CV-038
Voice Gate Defined
A written voice checklist must exist with a minimum of 5 questions that every new phrase must answer before being added to phrase_bank.json or HONMessageLibrary. Without a formal gate, voice drift happens incrementally — each individual addition seems fine but the cumulative effect dilutes the voice. The checklist makes the standard explicit.
Pass: A voice gate checklist document exists, is accessible to all contributors (in repo or linked from CLAUDE.md), and contains at least 5 distinct questions a phrase must answer. The checklist is not aspirational — it blocks non-compliant phrases.
CV-039
Approval Authority
One named person must have final voice approval authority for all new phrases. Approval by committee produces incoherent voice — every person approves phrases they personally find acceptable, and individual standards vary. A single approver produces a consistent standard. No phrase goes to production without their explicit, timestamped sign-off.
Pass: A sign-off log exists (can be a simple file, PR comment thread, or spreadsheet). Every phrase currently in production has a corresponding approval entry with a timestamp and approver identity.
CV-040
Phrase Review Cadence
The phrase bank must be reviewed quarterly for freshness, specificity, and voice drift. Phrases that were excellent at launch may become stale, may drift from the established register, or may reference features that have changed. A quarterly review catches drift before it compounds. The review calendar must be scheduled, not ad hoc.
Pass: A review calendar entry exists. The most recent phrase bank review occurred within the last 90 days. Review notes or a change log from the last review are accessible to the team.
CV-041
Contributor Guidelines
Any engineer or designer who might add copy — button labels, empty states, error messages, tooltips — must have access to the voice guide before writing. This prevents accidental voice violations introduced by contributors who are writing functionally correct copy but tonally inconsistent copy. The guide should be discoverable without asking.
Pass: The voice guide is accessible in the repo (CLAUDE.md or a linked document). It is referenced in contribution guidelines. A new contributor can find and read it without being told where to look.
CV-042
Copy Regression Process
When a feature is changed or removed, its copy must be reviewed for continued accuracy. A phrase that references "your last cardio session" becomes stale if cardio is removed. A phrase that references a specific exercise by name becomes stale if that exercise is removed from the database. Stale copy that survives feature changes is a copy regression.
Pass: The feature change checklist (in PR templates or engineering process) includes an explicit copy review step. No shipped build contains coaching copy that references removed features, old UI patterns, or renamed data fields.
♾ 7. Accessibility & Inclusion in Copy 4 tests ▼
CV-043
Plain Language — Grade 8 or Below
All instructional copy — onboarding, settings descriptions, error messages, help text — must be readable at a Grade 8 reading level or below on the Flesch-Kincaid scale. This is not about dumbing down content; it is about removing unnecessary complexity. Grade 8 is the threshold at which text becomes broadly accessible to adult readers.
Pass: Flesch-Kincaid readability score for all instructional copy is ≤ Grade 8. Coaching phrases are exempt (they are by nature short and direct). Test instructional copy specifically: onboarding explanations, settings descriptions, error messages.
CV-044
Ability-Neutral Language
Coaching language must not assume physical capability. "Stand up straight" assumes the user can stand and has full spinal mobility. "Complete the movement" is ability-neutral. "Keep your chest up" assumes a specific posture. "Maintain control through the movement" is neutral. Users with physical limitations should not encounter coaching that presupposes capabilities they may not have.
Pass: All coaching phrases reviewed for ability-neutral language. Zero phrases assume standing, bilateral limb use, full range of motion, or any specific physical capability that varies by user. Phrases describe movement principles, not specific physical positions.
CV-045
Gender-Neutral Language
All coaching language must be gender-neutral. No "he/him" or "she/her" pronouns in coaching copy. No gendered assumptions about training goals ("for a lean look" carries gendered assumptions). No gendered archetypes ("strong like a man"). H.O.N. coaches the individual — the individual has no assumed gender.
Pass: Search returns zero gendered pronouns (he, him, his, she, her, hers) in all user-facing coaching strings. Zero phrases carry implicit gendered assumptions about training goals or body ideals.
CV-046
Non-Judgmental Weight and Body Language
No copy may reference body weight as a positive or negative outcome. "Your training weight" is acceptable when it means the weight on the bar. "Your body weight goal" conflates fitness with body composition and is not acceptable. H.O.N. tracks performance, not body transformation. All weight references must be unambiguously about the load being lifted, not the user's body.
Pass: Search for "body weight", "losing weight", "gaining weight", "body fat", "lean", "tone", "slim" in all coaching strings returns zero results in user-facing text. "Training weight" and "bar weight" are acceptable where exercise-context-specific.

Section 3 — Definition of Done for Content

Content is done when every phrase, label, error message, empty state, confirmation dialog, and coaching note in the shipped build has been authored against the Voice & Tone Foundation (CV-001 through CV-008), reviewed against the phrase bank quality standards (CV-009 through CV-016), and explicitly approved by the designated voice authority with a timestamped sign-off entry. Done means every coached state has copy — no state falls through to a blank or a generic system string. Done means the phrase bank has at least 8 variants per category and those variants are meaningfully distinct, not paraphrases. Done means all instructional copy reads at Grade 8 or below, uses ability-neutral and gender-neutral language, and makes no reference to body composition or body ideals. Done means the governance process is operational: the voice gate is written, the sign-off log is current, the quarterly review is scheduled, and contribution guidelines point to the voice guide. A build is not content-done if any of the 5 pre-launch gates remain open. Content done is a prerequisite for ship-ready — it is not a post-launch cleanup task.
