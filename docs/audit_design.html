BLOCK
HM-1 · Activity Heatmap
HomeView → StreakHeatMap
P1 Visual earns its place P7 Context before data All states
The heatmap renders as a constrained block rather than filling the full card width. It lacks a legend explaining what the color intensity represents (volume? sessions? streaks?). A new user sees a grid of faded dots with zero explanation. An experienced user has no scale reference — a "very active" day vs a "normal" day are visually ambiguous. Simulating 12 weeks of data: the heatmap fills roughly 70% of card width and the trailing white-space looks broken.
📊 Empty state: empty grey grid — no label, no "start logging" hint. Rich state (50 sessions): busy grid with no color scale — unclear if purple = volume or streak.
BLOCK → Fix: (1) Set `.frame(maxWidth: .infinity)` on heatmap container. (2) Add a legend row: "Less ●●●●● More" with color ramp. (3) Add a subtitle "Workout frequency past 12 weeks" so intent is immediate.
BLOCK
HM-2 · Momentum Chip
HomeView → MomentumChip
P6 Honest affordances P2 Tap reveals not repeats Growing, Rich
The momentum chip displays a "streak" or "momentum" badge but has no visual affordance that it is tappable. MomentumDetailView exists in the codebase but the chip does not navigate to it. Users who tap it receive no feedback — no animation, no navigation, nothing. This violates honest affordances: the chip looks decorative but was designed to be interactive.
📊 Rich state: chip shows "8 week streak 🔥" — no chevron, no tap response, no hover state.
BLOCK → Fix: Wrap MomentumChip in a Button that presents MomentumDetailView as a sheet. Add a subtle chevron or tactile highlight on press state. Alternatively, if navigation is intentionally deferred, remove the chip until it is wired.
FIX
HM-3 · "Start Workout" CTA positioning and labeling
HomeView → TodaysPlanSection → primary CTA
P8 Serves the habit P4 Legible hierarchy All states with plan
When a plan is scheduled for today, the primary "Start Workout" CTA is not visible above the fold — it sits below the heatmap, readiness card, and momentum chip. The button also does not display the routine name; it's generic. The most important action in the app (start today's session) requires scrolling to find. Simulating a user with a morning routine and a morning check-in: they see the readiness score, scroll past heatmap, and only then find the CTA below the fold.
📊 With plan: "Start Workout" appears at ~70% scroll depth. Without plan: correct — no CTA shown.
FIX → Move TodaysPlanSection above the heatmap when a plan exists today. Change button label from "Start Workout" to "Start [Routine Name]" e.g. "Start Push Day A". This single change makes intent unmistakable.
FIX
HM-4 · Development Tier vs adjacent card — mismatched heights
HomeView → BeginnerProgressCard / DevelopmentTierCard
P3 Consistent size language Minimal, Growing
The Development Tier card and the card immediately to its left (Readiness or another metric card) in the horizontal pair do not share the same height. The shorter card leaves visible whitespace at the bottom of the row, communicating "these cards are not related." Side-by-side cards in an HStack should share height via `.frame(maxHeight: .infinity)` + alignment, but one card has a fixed height while the other is dynamic.
📊 Minimal (1–3 sessions): Development Tier card is shorter by ~30px. Rich (10+ sessions): heights equalize naturally but tier card may grow taller.
FIX → Add `.frame(maxHeight: .infinity, alignment: .top)` to both cards in the HStack pair and wrap in an HStack with `.alignmentGuide`. Or use LazyVGrid with `.adaptive` columns for the card pair so SwiftUI equalizes heights automatically.

DC-1 · Expand/collapse icon — inconsistent across cards
Multiple cards — ExpandableChartCard, CollapsibleDashSection, ReadinessCoachCard
P3 Consistent size language P6 Honest affordances All
Expand icons vary: some cards use `arrow.up.left.and.arrow.down.right` (fullscreen), some use `chevron.down`/`chevron.up` (collapse), some use custom icons. The user can't build a consistent mental model of what "tap to expand" means because the icon shape differs by card type. A chevron on one card means "this section collapses" and on another means "this card goes full-screen."
FIX → Define two icons system-wide: (1) `chevron.up/down` = section collapse/expand (never full-screen), (2) `arrow.up.left.and.arrow.down.right` = full-screen. Apply consistently across all expandable surfaces.
