1. Unit Testing 8 tests · QA-001–QA-008 0/8 ▼
QA-001
e1RM formula routes correctly by rep count
SetRecord.e1RM(weight:reps:) must route to three distinct formulas depending on rep count: Epley for 2–10 reps, Mayhew for 11–15 reps, exact weight for 1 rep, and 0 (excluded) for any rep count above 15. Test boundary values: 1, 2, 10, 11, 15, 16 reps. Confirm that a single-rep set returns the raw weight with no multiplier applied.
Pass: Formula branch confirmed at each boundary. 1 rep returns exact weight. 16 reps returns 0. Epley and Mayhew produce numerically distinct results for the same weight/reps inputs in the 11–15 range.
QA-002
Equipment.effectiveWeight doubles dumbbells, floors barbells at 20 kg
Equipment.effectiveWeight(_:) applies two distinct adjustments. Dumbbell entries represent a per-hand weight — the effective bilateral total is weight multiplied by 2. Barbell entries floor at 20 kg (bar weight) regardless of the stored value. Verify both rules with explicit inputs: a 15 kg dumbbell entry should return 30, a 0 kg barbell entry should return 20.
Pass: Dumbbell 15 kg input returns 30. Barbell 0 kg input returns 20. Other equipment types return the stored value unchanged. No integer overflow or rounding error.
QA-003
ReadinessEngine produces score 0–100 with valid confidence and coachingNote
ReadinessEngine.compute(log:) must return a ReadinessState with a score clamped to the 0–100 range, a non-empty coachingNote, a valid confidence value (0.0–1.0), and a non-nil delta. Test with: empty log, single-entry log, 7-day dense log, and a log where all sessions are older than 30 days.
Pass: Score is within [0, 100] for all four input scenarios. coachingNote is non-empty in all cases. Confidence is within [0.0, 1.0]. No crash or nil result for any input.
QA-004
WorkoutNarrativeEngine phrase rotation avoids repeats across consecutive sets
The pick() function in WorkoutNarrativeEngine uses a UserDefaults rotation index per phrase-bank key to avoid returning the same phrase on consecutive calls. Simulate 10 consecutive calls to the same phrase-bank key and confirm no two adjacent results are identical. The rotation index must persist to UserDefaults and survive a simulated app relaunch.
Pass: No two adjacent results from pick() are identical for the same key, given a bank with 3+ phrases. Rotation index is written to UserDefaults before the function returns. Index increments correctly and wraps at bank length.
QA-005
SetRecord drop set fields persist correctly through encode/decode
SetRecord has three drop-set fields: isDropCompleted, dropWeight, and dropReps. Construct a SetRecord with all three fields set to non-default values, encode to JSON via JSONEncoder, decode via JSONDecoder, and verify field equality. Also verify a SetRecord with no drop set values round-trips without error.
Pass: Encoded/decoded SetRecord has identical isDropCompleted, dropWeight, and dropReps values. No data loss or type coercion. Nil/false defaults survive round-trip without becoming unexpected values.
QA-006
strengthRetentionFactor: no decay for first 14 days, then -0.7%/day, floor 50%
The strength decay model in StrengthLabView applies peakE1RM * strengthRetentionFactor(daysSince). The retention factor must be 1.0 for 0–14 days, decrease by 0.007 per day from day 15 onward, and never fall below 0.5. Test at days 0, 14, 15, 50, 100, and 200.
Pass: Days 0 and 14 return factor 1.0. Day 15 returns 0.993. Day 100 returns max(0.5, 1.0 - 0.007*(100-14)) = max(0.5, 0.398) = 0.5. Day 200 also returns 0.5 (floor). No value below 0.5 at any input.
QA-007
HomeCache builds in a single O(n) log pass; both lookup caches populated
HomeCache.buildExerciseCaches(log:) must populate both lastPerformanceCache and exerciseHistoryCache in a single traversal of the workout log. Verify with a synthetic log of 50 sessions across 10 exercises: both caches must be populated after a single call, all 10 exercises must have entries, and the function must not call any other cache-building function internally.
Pass: After one call to buildExerciseCaches, both caches are non-empty. All exercise UUIDs present in the log appear as keys in both caches. No duplicate log traversal detectable via instrumentation.
QA-008
WorkoutPlanEngine debounce token: only last in-flight result applied
refreshAnalytics() mints a new UUID token on each call. The background block only commits its result if the token still matches when it returns to the main thread. Simulate three rapid consecutive calls and verify that only the result from the third call is applied. Earlier results must be discarded silently, not queued.
Pass: Only the last result is committed when three calls fire in rapid succession. No intermediate result is observable on the main thread. No race condition or partial state from an earlier call leaks through.

Section 3 — Definition of Done for QA

A release is QA-complete when all of the following are true

  • 1. All 56 test cases in Section 1 have a status of PASS or SKIP. No test case may be left in TODO, FAIL, or BLOCKED state at release time. SKIP requires a written justification recorded in the QA run notes (e.g., "feature not included in this release scope").
  • 2. All 6 pre-release gates in Section 2 are confirmed passing. Gate confirmation must be signed off by the QA lead with a timestamp. Self-certification by the engineer who wrote the code is not acceptable for G-2 through G-5.
  • 3. xcodebuild -scheme workout -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build exits with code 0 and "BUILD SUCCEEDED" on the exact commit SHA being submitted for release. The build must not include any uncommitted changes.
  • 4. The UAT inbox (docs/UAT_INBOX.md) has been reviewed and all unresolved UAT notes older than 7 days have been triaged: either resolved, converted to a backlog item, or explicitly deferred with a reason. No UAT note may remain "unreviewed" at release.
  • 5. The phrase_bank.json key audit (QA-026) has been run against the current engine source and confirmed clean. This audit must be re-run whenever WorkoutNarrativeEngine.swift or phrase_bank.json is modified, not just at release time.
  • 6. Regression tests QA-039 through QA-044 have been run on the release candidate build and all returned PASS. Results must be logged with the build number and run date. "It passed last time" does not substitute for a current run.
  • 7. The QA run is documented: test case statuses are saved (this page's localStorage state or an exported snapshot), the run date, tester name, build number, and simulator/device configuration are recorded alongside the results.
