Root Cause
The weight step button multiplies delta by lbsPerKg (2.20462), producing 2.5 × 2.20462 = 5.5115….   • After any step in lbs mode, round the displayed value to 1 decimal before storing back
  • Apply same protection to drop set weight binding path (editingDropWeight)

B-008a P1 Confirmed Open
"Short gap" / "Long gap" message fires on exercise rotation, not on missed workout days — trigger logic is wrong
What the User Saw
A coaching phrase appeared implying the user "came back after a short rest." The user had worked out the previous day. The message felt wrong and broke trust in the coaching system.
Root Cause
WorkoutNarrativeEngine computes historyKey per-exercise: if that specific exercise hasn't been done in 7–13 days, it's "Short Gap." This is exercise-level recency, not session-level recency. A user on a normal push/pull/legs split will bench press every 7 days by design — every bench session will trigger "Short Gap" even though they've never missed a training day. A user who trains yesterday but rotates to a new movement today will see a gap message for that movement. The trigger is exercise rotation, not absence. It should only fire when the user actually skipped workout days.
Correct Behaviour
"Short Gap" and "Long Gap" should only appear when the user has genuinely missed training days — i.e., there were days they would have trained but didn't. The gap is about the user's attendance, not the exercise's rotation frequency.
Proposed Solution
  • Primary fix — session-gap gate: In assembleSingle(), before applying the history key, check daysSinceLastAnySession. If the user has trained within the last 3 days (i.e., they are actively training and simply rotating exercises), force historyKey = "Normal" and suppress the gap phrase entirely. The exercise gap is irrelevant if the user is showing up.
  • Threshold logic: Gap phrases should only fire when BOTH conditions are true: (a) the exercise gap is ≥7 days, AND (b) the user's last workout session was ≥5 days ago. This ensures the message reflects real absence, not normal exercise rotation.
  • Secondary fix — phrase audit: Any phrase in phrase_bank.json under history keys "Short Gap" / "Long Gap" that implies a general workout return ("You're back", "Welcome back") must be rewritten to be exercise-specific: "It's been a while since you've done this one" — not a session-return message.
  • Pass daysSinceLastAnySession into ExerciseAnalysis or as a parameter to the assembly function so the gate has access to session-level context.

Combine Advanced analytics (StrengthLabView) and Insights into a single unified view — two analytics tabs create user confusion
Problem
The current 6-tab layout has both a Progress tab (workout history, consistency, PRs) and an Advanced/Insights tab (StrengthLabView: composite score, e1RM analytics, decay model). Two tabs answering overlapping questions about "how am I doing" creates ambiguity: users don't know which tab to open, and data is split across both. The result is confusion and underuse of both views.
Proposed Unification
  • Merge StrengthLabView content and any standalone Insights content into a single view
  • The unified view is the analytics depth level reachable from Progress via "Full Analytics →" (see NAV-001) — it is not a separate tab
  • Content hierarchy: Progress (overview, 5 categories) → Full Analytics (StrengthLabView merged with Insights, all depth content)
  • Decide what "Insights" means explicitly before merging: if Insights = coach-generated observations (from EmergentInsightEngine), those belong in Full Analytics as a dedicated section
  • Coordinate with NAV-001 — this item defines the content of the unified analytics view that NAV-001 makes a drill-down
Acceptance Criteria
  • No standalone Insights tab and no standalone Advanced tab in the final 5-tab layout
  • All StrengthLabView content and all Insights content accessible from a single drill-down view reached via Progress
  • Clear section separation within the unified view: "Strength Analytics", "Insights & Coaching", "History"
  • No analytics content orphaned or removed — only reorganized
