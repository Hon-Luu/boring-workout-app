B-001 P1 Confirmed Open
Decimal overflow in weight field (lbs mode)
Root Cause
The weight step button multiplies delta by lbsPerKg (2.20462), producing 2.5 × 2.20462 = 5.5115…. The display is formatted via .weightFormatted (max 1 decimal), but floating-point accumulation over multiple presses, combined with kg↔lbs round-trip conversion, causes drift. The stored kg value may not convert back to a clean lbs display — e.g. 45.3592 kg → 99.9999… lbs. Drop set fields use a different binding path without the same formatting protection.
Proposed Solution
  • Round the lbs step to nearest 0.5 lbs before adding: (round(cur * 2) / 2) + 2.5
  • After any step in lbs mode, round the displayed value to 1 decimal before storing back
  • Apply same protection to drop set weight binding path (editingDropWeight)

B-004 P1 Confirmed Open
All sets get the same suggested weight — warm-up sets ignored
What the User Saw
When weight suggestions are applied, every set of an exercise is set to the same weight (e.g. 82.5 kg × 4 sets). In practice, users typically warm up — set 1 is lighter, working weight starts at set 2 or 3. The equal-weight suggestion forces the user to manually adjust set 1 every single session.
Root Cause
applyWeightSuggestion(exerciseId:weightKg:) iterates all sets for that exercise and applies the same targetWeight uniformly. There is no concept of warm-up sets in the data model — every SetRecord is treated as a working set.
Proposed Solution
  • Apply suggested weight only to sets whose previous targetWeight matched the last working weight (i.e. skip sets that were previously below working weight)
  • Add a isWarmup: Bool flag to SetRecord — warm-up sets retain last session's weight unchanged
  • In the suggestion sheet, show a preview of how weights will be distributed across sets so the user can see before accepting
  • Infer warm-up sets from history: if set 1 weight was consistently 60–70% of set 2–3 weight over last 3 sessions, mark set 1 as warm-up automatically

B-006 P0 Confirmed Open
RPE: numeric scale is confusing; should use words; last set complete button should be the RPE selector (one tap for both)
What the User Saw
The RPE row shows a horizontal scroll of numeric chips: 6 · 6.5 · 7 · 7.5 · 8 · 8.5 · 9 · 9.5 · 10. Most users don't know what RPE means numerically. The row appears after every single set — asking for effort rating 3–4 times per exercise. The user does not want to log a set AND then separately log an RPE as two distinct actions.
Root Cause
RPERow uses a numeric scale from 6.0 to 10.0 in 0.5 steps (9 chips + clear). It is rendered as part of every completed set row with no logic to suppress it on non-final sets. Completing a set and rating effort are treated as two independent actions with no connection between them.
Proposed Solution — One Tap for Both
For sets 1 through N-1: normal circle checkmark, completes set only, no RPE prompt.

For the last set (all prior sets completed, this one is not): replace the checkmark with 4 inline word chips:
[ Easy ]  [ Hard ]  [ Very Hard ]  [ Max ]   ✓ skip
  • Tapping any chip calls completeSet() + updateRPE() atomically — one tap, both logged
  • The ✓ skip button does a plain complete with no RPE for users who don't want to rate
  • Last set detection: isLastSet = setIndex == sets.count - 1 && !record.isCompleted && allPreviousSetsCompleted — pass as Bool parameter into SetRow
  • Word → RPE mapping stored internally: Easy → 6.5 · Hard → 8.0 · Very Hard → 9.0 · Max → 10.0 — no changes to the data model or analytics pipeline
  • Remove RPERow from completed set rows entirely — it only ever needs to appear once, on the last set, as part of the complete action
  • Apply the same pattern in GuidedWorkoutSessionView for guided sessions

T-012 P2 Confirmed Open
EmergentInsight outputs not connected to Trainer — 6 advanced signals computed but never used for planning
Available Signals and Trainer Use
The following EmergentInsight types are computed each analytics cycle but never reach the Trainer:
  • trueAdaptationState — if "maintaining" (not adapting), flag to the user: "Gains have plateaued across the board — consider a stimulus change this week"
  • imbalanceTrajectory — if worsening, route the plan to correct it (feeds into T-005)
  • loadTolerance — if low, cap intensity at moderate regardless of readiness score
  • feelArchetype — if "moodDependent", add a note: "Your performance varies with how you feel going in — rate yourself before starting"
  • strongestDay — if today matches the user's strongest day of week, surface it: "You consistently perform best on [day] — good day to push"
  • programCalibration — if frequency is too low for stated goal, surface it in coach note
Requirements
  • Depends on T-010 (analytics passed into plan engine)
  • In generatePlans(), iterate insights array and apply the relevant routing rules listed above
  • loadTolerance = low overrides intensity: force moderate cap
  • strongestDay match: add a coachNotePrefix string that gets prepended to the plan's coach note
  • trueAdaptationState = maintaining for ≥3 consecutive weeks: add stimulus-change hint to coach note
  • All routing rules are skipped if the insight's inputs prerequisites are not met (insight is in placeholder state)

Strongest day routing — high-performance day not flagged in Trainer when it would most affect load decision
Description
EmergentInsightEngine.strongestDay() identifies the day of week with the highest average session volume. If today is that day and readiness is moderate-to-high, the plan should nudge the user toward the higher end of the intensity band. This is a distinct signal from the readiness score — it's about day-of-week performance patterns, not recovery state.
Requirements

  • If today matches the strongest day and readiness ≥ 65: add a one-line callout above the card deck — "Your strongest day of the week. Go for it." — in HONTheme.positive
  • Callout is a single text line, not a card — it must not compete visually with the plan cards
  • Callout only appears when the insight has sufficient data (insight is not in placeholder state)
  • Callout is dismissed after the user starts a session (tracked via @State)

T-014 P2 Open
Per-exercise intelligence not shown on plan cards — slopePerWeek and isPlateau invisible to user
Description
Each exercise in a plan card shows name + sets × reps + weight. The plan engine has (or will have after T-010) access to slopePerWeek and isPlateau per exercise. Surfacing this on the card gives the user a reason to trust the selection — and flags when something unusual was prescribed.
Requirements
  • Add an optional performanceTag: String? field to GuidedExercise — populated during plan generation when analytics are available
  • Tag values: "↑ gaining" (slope ≥ 1.0 kg/wk) · "→ holding" (0.2–1.0 kg/wk) · "⟳ variation" (isPlateau, exercise was swapped) · nil (no history)
  • In WorkoutCard exercise list rows, show the tag as a small coloured label after the exercise name — max 10 characters, caption2 font
  • Colour: gaining = HONTheme.positive, holding = .secondary, variation = HONTheme.warning
  • Tag is hidden when performanceTag == nil — no blank space shown
Acceptance Criteria
  • An exercise progressing at +1.5 kg/wk shows "↑ gaining" tag on its row in the plan card
  • An exercise swapped due to plateau shows "⟳ variation" tag
  • New users with no history see no tags — card layout unchanged
