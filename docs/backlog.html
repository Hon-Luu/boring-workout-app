B-001 P1 Confirmed Open
Decimal overflow in weight field (lbs mode)
Root Cause
The weight step button multiplies delta by lbsPerKg (2.20462), producing 2.5 × 2.20462 = 5.5115…. The display is formatted via .weightFormatted (max 1 decimal), but floating-point accumulation over multiple presses, combined with kg↔lbs round-trip conversion, causes drift. The stored kg value may not convert back to a clean lbs display — e.g. 45.3592 kg → 99.9999… lbs. Drop set fields use a different binding path without the same formatting protection.
Proposed Solution
  • Round the lbs step to nearest 0.5 lbs before adding: (round(cur * 2) / 2) + 2.5
  • After any step in lbs mode, round the displayed value to 1 decimal before storing back
  • Apply same protection to drop set weight binding path (editingDropWeight)

B-002 P1 Confirmed Open
"FAIL" pill overflows reps fraction field layout
Root Cause
The RepsFractionField HStack contains: 48px reps field + slash + "FAIL" pill + 14px toggle button. The containing ExerciseCard row also has a 22px set number, a 98px weight stepper, and a 44px complete button — leaving the reps fraction with minimal allocated space. When toFailure = true, the "FAIL" pill replaces the target number field; its padding and text can clip against the complete button or exceed the available width, especially on smaller devices or in lbs mode where the weight field label is wider.

  • Constrain RepsFractionField with .fixedSize(horizontal: true, vertical: false) and ensure the containing HStack uses explicit Spacer(minLength: 0)
  • Add a minimum width floor to the reps column in ExerciseCard

B-003 P0 Confirmed Open
Coaching note contradicts weight suggestion (readiness vs. progression systems not coordinated)
What the User Saw
Readiness score 50–65 → coaching note: "Clean execution over big numbers today." Simultaneously, the weight suggestion sheet appeared offering +2.5 kg on every exercise. The app told the user to hold back and to push more in the same session open.
Root Cause
ReadinessEngine coaching note is based purely on the readiness score. SeedStore.computeWeightSuggestions() is based purely on whether the last session hit ≥90% of target reps. Neither system knows the other exists. A user can have a moderate readiness score while simultaneously having hit all reps last session — both conditions fire independently.
Proposed Solution
  • Pass readiness score into computeWeightSuggestions(): suppress suggestions entirely when score < 60
  • When score 60–72, label suggestions as optional: "You hit all reps, but readiness is moderate — try this only if you feel good"
  • Add a readinessContext field to WeightSuggestion so the sheet can display readiness-aware framing
  • Long term: merge these into a single pre-session brief that gives one coherent recommendation, not two conflicting ones

B-004 P1 Confirmed Open
All sets get the same suggested weight — warm-up sets ignored
What the User Saw
When weight suggestions are applied, every set of an exercise is set to the same weight (e.g. 82.5 kg × 4 sets). In practice, users typically warm up — set 1 is lighter, working weight starts at set 2 or 3. The equal-weight suggestion forces the user to manually adjust set 1 every single session.
Root Cause
applyWeightSuggestion(exerciseId:weightKg:) iterates all sets for that exercise and applies the same targetWeight uniformly. There is no concept of warm-up sets in the data model — every SetRecord is treated as a working set.
Proposed Solution
  • Apply suggested weight only to sets whose previous targetWeight matched the last working weight (i.e. skip sets that were previously below working weight)
  • Add a isWarmup: Bool flag to SetRecord — warm-up sets retain last session's weight unchanged
  • In the suggestion sheet, show a preview of how weights will be distributed across sets so the user can see before accepting
  • Infer warm-up sets from history: if set 1 weight was consistently 60–70% of set 2–3 weight over last 3 sessions, mark set 1 as warm-up automatically

B-006 P0 Confirmed Open
RPE: numeric scale is confusing; should use words; last set complete button should be the RPE selector (one tap for both)
What the User Saw
The RPE row shows a horizontal scroll of numeric chips: 6 · 6.5 · 7 · 7.5 · 8 · 8.5 · 9 · 9.5 · 10. Most users don't know what RPE means numerically. The row appears after every single set — asking for effort rating 3–4 times per exercise. The user does not want to log a set AND then separately log an RPE as two distinct actions.
Root Cause
RPERow uses a numeric scale from 6.0 to 10.0 in 0.5 steps (9 chips + clear). It is rendered as part of every completed set row with no logic to suppress it on non-final sets. Completing a set and rating effort are treated as two independent actions with no connection between them.
Proposed Solution — One Tap for Both
For sets 1 through N-1: normal circle checkmark, completes set only, no RPE prompt.

For the last set (all prior sets completed, this one is not): replace the checkmark with 4 inline word chips:
[ Easy ]  [ Hard ]  [ Very Hard ]  [ Max ]   ✓ skip
  • Tapping any chip calls completeSet() + updateRPE() atomically — one tap, both logged
  • The ✓ skip button does a plain complete with no RPE for users who don't want to rate
  • Last set detection: isLastSet = setIndex == sets.count - 1 && !record.isCompleted && allPreviousSetsCompleted — pass as Bool parameter into SetRow
  • Word → RPE mapping stored internally: Easy → 6.5 · Hard → 8.0 · Very Hard → 9.0 · Max → 10.0 — no changes to the data model or analytics pipeline
  • Remove RPERow from completed set rows entirely — it only ever needs to appear once, on the last set, as part of the complete action
  • Apply the same pattern in GuidedWorkoutSessionView for guided sessions

B-007 P2 Confirmed Open
Finish screen message has insufficient variety — same message repeats weekly
What the User Saw
The session celebration screen shows the same phrase every week for the same number of sessions. A user who consistently does 3 sessions per week sees "Three sessions in. The habit is forming." every single week indefinitely. The message pool has exactly 7 entries — one per possible weekly session count — with no variation or context awareness.
Root Cause
SessionCelebrationView.feedbackText selects from a hardcoded 7-item array indexed purely by sessionDays.count. No randomization, no contextual routing (PR in this session, volume record, first session back, streak milestone), and no rotation to avoid repetition.
Proposed Solution
  • Expand to a phrase bank with 4–6 options per session count (28–42 total) — store in a JSON file matching the WorkoutNarrativeEngine pattern
  • Add context-aware routing: if a PR was set this session → PR variant; if volume record → volume variant; if streak milestone → milestone variant
  • Use the same UserDefaults rotation pattern from WorkoutNarrativeEngine.pick() to avoid consecutive repeats
  • Add a second message tier that references the streak count when it's ≥4 weeks: "X weeks straight. Habits don't break this easily."

B-008a P1 Confirmed Open
"Short gap" message fires for exercise-specific gap, but reads as a session-level gap to the user
What the User Saw
A coaching phrase appeared implying the user "came back after a short rest." The user had worked out the previous day. The message felt wrong and broke trust in the coaching system.
Root Cause
WorkoutNarrativeEngine computes historyKey per-exercise: if that specific exercise hasn't been done in 7–13 days, it's "Short Gap." This is exercise-level recency, not session-level recency. A user doing chest/shoulders yesterday but now doing back exercises will trigger "Short Gap" for back exercises — but the phrase bank likely says something like "You're back" which reads as a general workout return message. The user didn't take a break from training — they took a break from THAT exercise.
Proposed Solution
  • Audit all "Short Gap" and "Long Gap" phrases in phrase_bank.json — rewrite any that imply a general training absence to be exercise-specific: "It's been a while since you've done [exercise]" rather than "You're back"
  • In assembleSingle(), prepend the exercise name to the history phrase context so any phrase naturally references the exercise, not the user's overall attendance
  • Consider adding a session-gap check: if daysSinceLastSession < 3, suppress "Short Gap" history phrases entirely regardless of exercise-level gap

B-008b P1 Confirmed Open
Skipped exercises within a session are not acknowledged in coaching or session summary
What the User Saw
The user skipped exercises mid-session. The coaching system said nothing about it. The session summary and celebration screen showed aggregate stats as if the session was normal, with no acknowledgment that exercises were left incomplete.
Root Cause
WorkoutNarrativeEngine operates per-exercise in isolation — it has no session-level view. The celebration system (CelebrationSystem) receives total sets and volume but has no concept of "exercises planned vs. exercises completed." There is no skip detection anywhere in the post-session pipeline.
Proposed Solution
  • Add a skip signal to the session model: an exercise with 0 completed sets when ≥1 set was planned counts as skipped
  • Pass a skippedExercises: Int count into CelebrationKind.sessionComplete
  • When skippedExercises > 0, show a brief acknowledgment on the celebration screen: "You got through most of it. [X] exercises left for next time." — no guilt, just honesty
  • Feed skips into the session-level readiness signal (as described in the inference rules) — skipping exercises is a meaningful load signal that should affect tomorrow's recommendations
  • In SessionReviewSheet, visually differentiate skipped exercises (0 sets completed) from completed ones

WorkoutPlanEngine re-randomizes every SwiftUI render — plans are unstable
Root Cause
TrainerTabView.plans is a computed property that calls WorkoutPlanEngine.generatePlans() on every body evaluation. The engine uses randomElement() internally — no seed, no determinism. Any state change in the view (scroll position, sheet presentation, orientation) causes a completely different set of exercise recommendations to silently appear.

A-004 P1 Confirmed Open
muscleMassPercent field stores lean body mass (79.85%) but is labeled and used as muscle mass (range 10–65%)
Root Cause
UserProfile.muscleMassPercent stores lean body mass percentage (~79.85%), not skeletal muscle mass (~40–45%). The field is labeled "Muscle Mass %" in the UI with a slider range of 10–65%. The StrengthScoreEngine uses this value in PCSA allometric calculations, inflating strength scores by approximately 1.5–2× for average users.
Proposed Solution
  • Rename field to leanBodyMassPercent and update the settings UI label accordingly
  • In StrengthScoreEngine, derive skeletal muscle mass from lean body mass: muscleKg ≈ leanKg × 0.55 as a rough correction factor
  • Or remove the muscle mass input entirely and use bodyweight-relative norms directly, which are more reliable than user-estimated body composition
