Section 3 — Definition of Done for Design

H.O.N. is done from a design standpoint when a person who has never seen the app can pick it up, understand what it does within 30 seconds of looking at the home screen, complete their first workout without any friction, receive coaching feedback that feels written for them specifically, and come back the next day because the experience left them feeling capable — not monitored. Done means no screen is a number dump, every secondary view opens with a sentence that interprets its most important data point, the coaching voice is consistent from the first set to the last drill-down sheet, celebrations feel earned rather than automatic, and the app's identity — spare, direct, honest — is present in every pixel. Done does not mean beautiful. It means purposeful. Every design decision exists because it serves the person holding the phone, not because it looked good in Figma.

Section 4 — LLM Design Critique Prompt
Design Review Tool — Parametric LLM Prompt
Use this prompt to evaluate any screen of the H.O.N. app. Replace the placeholder fields with the screen name and a plain-language description or screenshot context. The output is structured feedback, not a checklist — it surfaces the most important design issue on that screen first, then works down in priority order.
CONTEXT You are reviewing a screen from H.O.N. (Habit Over Numbers), an iOS SwiftUI fitness coaching app. H.O.N. is dark mode only, uses a strict design token system (background #111111, card #1c1c1e, amber #f59e0b for signals, green #34d399 for positive states), and has a coaching voice that is spare, direct, and identity-affirming — never cheerleading, never clinical. The app has 5 tabs: Home, Workout, Progress, Insights, Trainer. SCREEN UNDER REVIEW Screen: [SCREEN NAME] Context: [SCREEN DESCRIPTION OR SCREENSHOT CONTEXT — describe layout, visible elements, key interactions, and any data states shown] EVALUATION DIMENSIONS Evaluate this screen across the following six dimensions. For each dimension, produce one paragraph of structured feedback. Lead with the most significant finding. Do not produce a checklist — write as a senior product designer giving a colleague a direct, useful critique. 1. Visual Hierarchy Where does the eye land first? Is there a single dominant element? Does the information density match the user's task at this moment? Are supporting elements subordinate without being invisible? Does the layout communicate priority or just fill space? 2. Voice & Tone Read every piece of copy on this screen aloud. Does it sound like a knowledgeable friend or a generic app? Is any phrase something that could appear on any fitness app for any user? Does the language position the user as a capable athlete or as someone being managed? Is silence used appropriately (no filler copy where no copy is needed)? 3. Information Density Is everything shown necessary at this moment in the user's flow? Is there data visible that the user cannot act on from this screen? Are there decisions the user needs to make that require information not shown? Does the screen respect the user's attention or tax it? 4. Platform Conventions Does this screen behave as an iOS user expects? Are native patterns used correctly (sheets, navigation, gestures)? Are tap targets adequate? Does any element fight the platform rather than work with it? Would a user who has never seen this app know how to interact with it based on iOS familiarity alone? 5. Emotional Design What does the user feel when they see this screen? Does the design build confidence or anxiety? Are moments of achievement acknowledged proportionally (not over-engineered, not ignored)? Does the screen feel like it belongs to a product that respects the user's time and identity? 6.
