A-01
Zero-rep completed sets pollute analytics
CRITICAL

10 sets across 6 sessions have isCompleted=true and reps=0. The model defines completedSets = sets.filter(\.isCompleted) — no reps guard. Every engine that reads completedSets (rep%, repOutcome, e1RM, volume) will include these sets.

Evidence — All 10 zero-rep incidents

Apr 20 (S1): DB Shoulder Press Set2 — 10kg × 0 reps, isCompleted=true
May 1 (S7): Chest Press Machine Set1 — 15kg × 0 reps, isCompleted=true
May 7 (S9): Machine Shoulder Press Set1/2/3 — 10kg × 0 reps each (3 sets)
May 12 (S11): Machine Shoulder Press Set1/2 — 15kg × 0 reps each
May 18 (S14): DBP Set2/3 — 17.5kg × 0 reps each
May 19 (S15): Leg Press Set3 — 37.5kg × 0 reps

Pass

Zero-rep sets are excluded from rep% calculation, e1RM, and volume, OR are flagged explicitly to the user before being included.

Fail

Any engine (WorkoutNarrativeEngine, StrengthAnalyticsEngine, ReadinessEngine) includes 0-rep sets without a guard. Failure confirmed if: Machine Shoulder Press May 7 history shows 0 e1RM contributing to trend line, or if May 18 DBP narrative does not flag that 2 of 3 sets had 0 reps.

Run against: S1 (Apr 20), S7 (May 1), S9 (May 7), S11 (May 12), S14 (May 18), S15 (May 19)

A-02
readinessBefore = null for all 18 sessions
HIGH

ReadinessEngine applies a readinessBefore modifier: avg ≤1.3 → -5, avg ≥2.7 → +4. With all values null, the modifier always computes at neutral. The ReadinessPromptCard UI exists in code — so the question appears not to have been answered in any session.

Evidence

All 18 WorkoutLogEntry.readinessBefore fields are null in the backup. Sessions span Apr 20 – May 21. The pre-workout readiness prompt should have appeared before each session. Zero fills means either: (a) user dismissed every prompt, (b) prompt has a bug and never surfaced, or (c) the field was not captured in this version.

Pass

App correctly shows ReadinessPromptCard before the workout begins. Dismissal is a valid user action and the null is expected.

Fail

Prompt never appears, OR appears but tap does not persist the value to the log entry. Verify: start a new session and confirm the prompt is shown. Check that submitting a value writes a non-null readinessBefore.
