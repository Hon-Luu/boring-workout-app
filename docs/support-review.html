Section 1 — Must-Pass Support Criteria
⚠ 1. Crash Detection & Triage 7 tests ▼
SR-001
Crash Detection Pipeline
A method exists to detect crashes in production before users report them. Minimum: App Store Connect crash reports checked daily at a fixed time. Preferred: Crashlytics or equivalent integrated before launch.
Pass: crash detection method documented; responsible person named; checking cadence defined.
SR-002
Crash Severity Matrix
Crashes are classified within 4 hours of detection. P0 = crash on launch OR any crash causing data loss. P1 = crash in core workout logging flow. P2 = crash in secondary views (StrengthLab, Trainer, Settings). P3 = rare edge-case crash affecting <0.1% of sessions.
Pass: matrix documented; team can classify any crash in under 5 minutes using the matrix.
SR-003
P0 Crash Response SLA
P0 crashes trigger: (1) engineering notification within 1 hour, (2) hotfix build in TestFlight within 4 hours, (3) App Store submission within 24 hours. On-call person defined for first 72 hours post-launch.
Pass: SLA documented; on-call person named and has push access to TestFlight and App Store Connect.
SR-004
P1 Crash Response SLA
P1 crashes fixed in next release (≤7 days) unless frequency exceeds 0.5% of sessions (then treat as P0). Frequency threshold is defined and measurable from App Store Connect.
Pass: SLA documented; frequency threshold defined and measurable from App Store Connect.
SR-005
dSYM Symbolication
Debug symbols (dSYM files) archived for every App Store and TestFlight submission. Crash reports are symbolicated before triage — human-readable stack traces, not memory addresses. Archive process is documented and tested.
Pass: dSYM archive process documented; test by symbolicating a sample crash log from TestFlight; output is human-readable.
SR-006
Crash Repro Kit
For any reported crash, the team uses a standard repro template: device model, iOS version, app version, approximate session count, what the user was doing, any custom data state. Template has been used in actual TestFlight investigations.
Pass: repro template exists; used in at least 3 TestFlight crash investigations; all required fields present.
SR-007
User Communication for Widespread Crashes
If a crash affects >1% of sessions, a user-facing acknowledgment exists within 48 hours: App Store review response, or response to any support emails received. Communication template written and voice-checked before sending.
Pass: communication template exists; named person is responsible for drafting within 48h; reviewed for H.O.N. voice before sending.
💬 2. User Feedback Intake 7 tests ▼
SR-008
Support Channel Operational
At least one support channel is live and monitored before launch: support email (from App Store listing) or in-app feedback. Channel tested with a real inbound message before submission. First-response SLA defined.
Pass: channel live; response tested end-to-end; first-response SLA defined (target: 24 hours).
SR-009
App Store Review Monitoring
Reviews checked daily. Negative reviews (1–2 star) receive a personal response within 48 hours. Response is specific to the complaint, not templated. Monitoring calendar item exists to enforce the cadence.
Pass: monitoring calendar item exists; 48h SLA documented; at least 3 sample responses written that are distinct and on-brand.
SR-010
Review Response Voice
All App Store responses match H.O.N. voice: direct, human, specific. Zero instances of: "Dear valued customer", "We sincerely apologize for any inconvenience", "Please contact our support team." Response guide exists with 5 example responses; independently verified.
Pass: response guide exists with 5 example responses; an independent reviewer confirms all 5 pass the voice test.
SR-011
Feature Request Routing
User-submitted feature requests are captured, categorized (UX / coaching / analytics / new feature), and added to the product backlog within 7 days. Intake-to-backlog process is documented and tested with simulated requests.
Pass: intake → backlog process documented; tested with 3 simulated feature requests; all appear in backlog within the SLA.
SR-012
Feedback Synthesis
All qualitative feedback (reviews, emails) synthesized into themes monthly. Themes shared with product and content teams. Synthesis template covers: voice complaints, UX friction, feature requests, data concerns. Monthly calendar item scheduled.
Pass: synthesis template exists; monthly calendar item scheduled; covers all four feedback domains.
SR-013
Data Loss Protocol
If a user reports lost workout data, the support team has a documented response: (1) confirm data is local-only (no server backup), (2) explain iCloud backup behavior if applicable, (3) attempt recovery steps. Protocol tested with a simulated data-loss scenario.
Pass: data loss protocol exists; tested with a simulated data-loss scenario; response is honest about local-only storage.
SR-014
Sensitive Feedback Protocol
If a user reports fitness-related distress (injury, disordered eating signals, obsessive training language), the response: (1) does not give medical advice, (2) acknowledges the concern warmly, (3) provides a general referral to appropriate resources. Does not promote continued app use in this context.
Pass: protocol documented and reviewed by product; response does not promote continued app use in the context of distress.
📄 3. Known Issues Management 6 tests ▼
SR-015
Known Issues Log
A live document lists all known bugs with: ID, severity, affected versions, user-facing symptom, workaround (if any), fix version. Accessible to support team in under 30 seconds. Updated within the last sprint cycle.
Pass: log exists; updated within the last sprint cycle; support team can locate any issue by symptom description.
SR-016
Workaround Documentation
Every P1 and P2 known issue has a user-facing workaround written in plain language — no technical jargon. All open P1/P2 backlog items have a workaround entry. Workarounds tested by a non-engineer to confirm they are followable.
Pass: all open P1/P2 backlog items have a workaround entry; workarounds tested by a non-engineer.
SR-017
App Version Tagging
Every reported bug is tagged with the app version it was reported on. When a fix ships, the fix version is recorded. Support team can check in under 2 minutes whether a user-reported issue is from a fixed version.
Pass: version tagging process exists; all bugs in the log have version numbers; support team can check fix status by version.
SR-018

Before creating a new engineering ticket, support checks the known issues log and recent bug reports for duplicates. Deduplication check is a documented step in the bug intake process. Tested with simulated duplicate reports.
Pass: deduplication check is a documented step in the bug intake process; tested with 5 simulated duplicate reports.
SR-019

When a bug is fixed and shipped, support proactively responds to all users who reported it (via App Store review reply or email), noting the fix is in the new version. Closure protocol documented and tested with one real closed TestFlight bug.

Regression Escalation
If a previously fixed bug reappears, it is automatically escalated to the same priority level as the original (or P0 if any user data is at risk). Regression detection step exists in the QA process. Regression SLA documented.
Pass: regression detection step exists in the QA process; regression SLA documented.

Section 3 — Definition of Done for Support

Support is done when the person who reported a problem has heard back from a real human, in plain language, with a specific answer — not a ticket number and a promise. For bugs: done means the fix shipped and the user who reported it knows. For data loss: done means the person understands exactly what was recoverable and what wasn't, without false hope or corporate deflection. For sensitive contacts: done means the person feels heard and has a resource in hand, with no pressure to keep using the app. For App Store reviews: done means every negative review has a direct, specific reply that names the complaint — and when that complaint is fixed, another reply says so. The support operation passes its own criteria when a new team member, handed all the documentation and five real inbound reports, can classify, respond, and escalate every one of them correctly in under an hour, without guessing at any step. If they have to guess, the documentation failed — not the person.

System Context
You are a support triage assistant for H.O.N. (Habit Over Numbers), an iOS-only fitness coaching app. H.O.N. is dark-mode, pre-launch (TestFlight active), data is stored locally on device only — no backend server, no cloud sync. Support channels: App Store reviews and developer email. H.O.N. voice is spare, direct, identity-affirming. Never cheerleading. Never corporate. Response drafts must sound like a real human wrote them. Known issue patterns (check first): - B-001: Decimal weight overflow (e.g. 95 lbs displays as 95.1151) - B-002: FAIL pill overflows reps field layout - C-003: Countdown timer flashes during 3-2-1 start sequence - HealthKit permissions re-prompting after user already authorized - Phrase bank repetition (coaching tips feel repetitive)
User Report Input
[USER REPORT TEXT]
Required Output
1. Issue type: crash / data loss / bug / feature request / voice complaint / privacy / sensitive welfare
2. Severity: P0 / P1 / P2 / P3 — with one sentence of justification
3. Known issue match: yes (with bug ID from list above) / no — if no, describe why it doesn't match any known pattern
4. H.O.N.-voice response draft: direct, human, specific to the complaint, no corporate filler, under 150 words. Do not open with "Dear" or "Hi [name]". Do not include "We apologize for the inconvenience." Do not close with "Please don't hesitate to contact us."
5. Escalation: yes/no — if yes, specify who to notify (engineering lead / PM / founder) and why
6. Engineering ticket required: yes/no — if yes, include a one-line ticket description
7. Knowledge base gap: yes/no — if yes, suggest a title for the new knowledge base entry
