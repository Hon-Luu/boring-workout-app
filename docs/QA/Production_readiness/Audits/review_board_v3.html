Dr. Mara Teal
UX Researcher · Interaction Design · Cognitive Load Theory
SHIP-READY

Tap Reduction Analysis

This sprint's most impactful change is treating the feel rating as a first-class action, not a post-action confirmation. The current implementation removes the finish→sheet→select→dismiss chain (4 interactions) and replaces it with a single tile tap. This is interaction design done right: the affordance (the tile) and the action (end session + record) are unified.

What Works

The feel bar is always visible at the bottom as a persistent affordance. Users don't have to hunt for the end-workout action. The emoji+label format makes options scannable in under 200ms — consistent with research on icon-label pairing for speed.

Concern: Feel Bar Always Visible During Workout

The feel bar appears from session start. Early in a workout, seeing "How did that feel?" creates mild cognitive dissonance — the user hasn't felt anything yet. Consider fading the tiles to 30% opacity until at least one set is completed, then animating to full opacity with a brief pulse. This preserves the persistent affordance without creating premature intent framing.

Concern: RPE Discoverability

Moving RPE from an explicit "Rate effort" button to automatic appearance after all sets complete is technically correct but may surprise users on first encounter. The RPE row appears with no label change — it just materializes. A one-time tooltip ("Rate effort for this exercise — optional") on first appearance would reduce abandonment caused by confusion, not fatigue.

6.8
Dr. James Kroll
Sports Scientist · Exercise Physiology · Periodization
BETA — Training Logic Gaps Remain

Scored Recommendation Engine Review

The exerciseScore() function is a meaningful step forward from random selection. The scoring weights (+20 history, -18 recent, +12 good performance, -8 struggling, +10 ready-to-progress) produce reasonable orderings in typical cases. However, three structural problems remain.

Problem 1: No Weekly Volume Cap

The engine scores exercises independently with no awareness of total weekly volume per muscle group. A user training chest 4x/week (Monday, Wednesday, Friday, Saturday) will score each session at +20 (has history) with no penalty for accumulated fatigue. Research consensus: natural trainees peak at 10–20 hard sets/week per muscle group. The engine can currently recommend more than this without any signal.

Problem 2: Deload Logic Missing

After 4–6 weeks of consistent training (≥3 sessions/week), training stress accumulates faster than adaptation. No deload cycle is detected or suggested. The readiness engine partially compensates via HRV scoring, but structural volume reduction (a deload week) is a distinct concept absent from the system.

Problem 3: Cold-Start Plan Is Physiologically Sound But Fixed

The first workout (deterministic per region, light intensity) is a good cold-start solution. However, "light" intensity is not defined — the exercise loads are drawn from user history (which doesn't exist). The plan should explicitly cap RPE recommendation at 6/10 and suggest bodyweight or minimal-load variants. Currently it surfaces target weight as 0, which shows empty weight fields on first use.

HRV Personalization Assessment

The personal median with 7-reading calibration period is scientifically appropriate. The threshold multipliers (0.88 = low, 1.02 = high) are within the range used in peer-reviewed HRV research. The 7-day minimum is on the low end — most published protocols use 14+ days for reliable baseline — but acceptable for a consumer app where engagement before calibration drops off. Score: solid for V1.

Priority Fix: Add weekly volume tracking per muscle group. This is a data aggregation problem (store.workoutLog.filter{...}), not a new model. Without it, the recommendation engine can inadvertently prescribe overtraining for consistent users within 4–6 weeks.

Dr. Sofia Navarro
Behavior Change · Habit Formation · Self-Determination Theory
SHIP-READY

H.O.N. Philosophy Alignment

"Habit Over Numbers" is a coherent and research-backed positioning. The lapse threshold increase (3→6 days) reduces false failure signals — one of the most common reasons users abandon habit apps. Research on habit disruption (Gardner et al., 2012) shows that single missed days increase dropout probability by 4x in apps that immediately penalize streaks. Six days provides a real recovery window.

Feel Rating as Identity Marker

Positioning feel rating as the exit action — not a post-workout survey — subtly changes its meaning. The user's final act in each session is a self-assessment, not a confirmation. This is identity-relevant data ("I am someone who knows how hard I train") rather than logging friction. Self-Determination Theory predicts this increases intrinsic motivation over time.

Cold-Start First Workout Narrative

The deterministic intro plan combined with a narrative engine that presumably says something meaningful about a first session is strong. First impressions in behavior change apps have outsized weight — research shows users form usage patterns within the first 3 sessions. A poor first experience that feels generic loses 60%+ of users who don't return. Getting this right is worth more than any downstream feature.

Gap: No Streak Context in Narrative

The narrative engine comments on exercise performance but has no awareness of meta-habit data: streak length, days since last session, comeback status. A user returning after 5 days of absence gets the same narrative structure as a user on a 30-day streak. The system knows this information (store.currentStreak) but doesn't use it in the post-workout narrative. This is a missed personalization signal that matters enormously for psychological reinforcement.

Missing: Identity-Level Celebration Copy

The celebration system (streaks, PRs) fires overlays but the language is transactional ("30-day streak!") rather than identity-affirming ("You're the kind of person who shows up 30 days in a row"). This is a research-backed distinction — identity-level framing significantly outperforms achievement framing for retention. Easy copy change, high leverage.

Lian Park
Senior iOS Engineer · SwiftUI · Performance Profiling
SHIP-READY with minor tech debt

Code Quality Assessment

RPE Refactor: Correct Architecture

Moving RPE from SetRow (per-set, last-set gated) to ExerciseCard (exercise-level, completion-triggered) is the right abstraction. The previous code conflated set-level state with exercise-level state. The new code correctly reads workoutExercise.sets.allSatisfy(\.isCompleted) at the appropriate level and writes to onUpdateSetRPE?(lastIdx, newRPE) — the data model stays the same (RPE on last set), the trigger is now semantically correct.

WorkoutFeelBar: Clean Extraction

WorkoutFeelBar as a private struct is well-scoped. The onFinish: ((FeelRating?) -> Void)? optional callback pattern is idiomatic SwiftUI. The haptic feedback (UIImpactFeedbackGenerator) on feel tile tap adds quality for zero cost. The .regularMaterial background correctly uses the system material blur, consistent with the rest timer banner.

Concern: VStack in ZStack bottom doesn't safe-area-pad

The WorkoutFeelBar is wrapped in a VStack inside a ZStack(alignment: .bottom). On iPhone 14+ with home indicator, the bar's bottom edge will overlap the system home indicator unless .safeAreaInset(edge: .bottom) or .padding(.bottom, geometry.safeAreaInsets.bottom) is applied. The .regularMaterial provides visual continuity, but tappable area near the bottom of the bar may be partially obscured. Test specifically on a device without the simulator.

Concern: SeedStore @Observable with large workoutLog

SeedStore appears to be an @Observable class (based on @Environment(SeedStore.self) usage). If workoutLog is a large array, every append to it will re-render all views that observe it. ActiveWorkoutView observes SeedStore, so completing each set triggers a full re-render of the ExerciseCard list. For users with 100+ exercises in a session this could cause frame drops. Consider storing activeWorkout as a separate observable object.

Note: SourceKit False Positives

PBXFileSystemSynchronizedRootGroup produces false SourceKit errors ("Cannot find type 'Exercise' in scope") throughout the project. These are a known Xcode bug with this project structure. All changes verified via xcodebuild which shows BUILD SUCCEEDED. Do not treat these as real errors.

Technical Debt Inventory

  • ActiveWorkoutView.swift is 1,700+ lines. The file contains ActiveWorkoutView, ExerciseCard, SetRow, RPERow, WorkoutFeelBar, RestTimerBanner, ExerciseHistoryMatrix, RepsFractionField, NumberField, StepButton, DropSetPanel, and more. Extract into separate files before adding any more components.
  • Bottom padding constants are magic numbers. 200 : 120 for rest-timer vs no-timer states — these should be computed from actual view heights, not estimated constants. Fragile across device sizes.
  • iCloudSync.swift is dead code for this account. The entitlements are removed, so every sync call hits the graceful nil fallback. The sync infrastructure is well-written but running no-ops. Document this state.

Alessandra Voss
Content Strategist · Narrative UX · Voice and Tone
BETA — Structural variety good, voice needs depth

Narrative Format Rotation Analysis

The 4-format rotation (standard, prescriptionLed, sparse, comparisonLed) solves the identified template fatigue problem. The format selector is elegant — Int(workout.startedAt.timeIntervalSince1970 / 3600) % 4 produces a deterministic but non-obvious cycle. Users will not perceive the pattern consciously even after 20 sessions.

Strongest Format: comparisonLed

Leading with the delta vs. last session ("Up 2.5 kg from last week") is the highest-information format. Research on progress feedback shows users respond most strongly to relative change (vs. absolute values) — it provides context, not just numbers. This format should appear more frequently for users with ≥6 sessions of history where comparisons are meaningful.

Strongest Format: sparse

Single-sentence narratives after long sessions have a different emotional resonance. Brevity signals confidence. "You hit every target. That's the work." reads differently from a paragraph — it feels like a real coach, not a data report. This format is particularly effective after PRs or clean sessions.

Concern: prescriptionLed reads as mechanical

The prescription-led format leads with "Next session: [exercise list with progression targets]." This is useful information, but leading with it rather than reflecting first makes the narrative feel like a task list, not coaching. Consider using prescription-led only as a second paragraph, after at least one sentence of reflection. "Strong session on bench. Next: try 85 kg for 5 — you've had 3 clean weeks." has more voice.

Gap: No Session-Level Narrative Arc

Every narrative comment on individual exercises independently. A session with Squat + RDL + Leg Press + Leg Curl produces 4 independent paragraphs. A human coach would read the session as a whole: "Today was a lower-body volume day — you went heavy on squat and pulled back on RDL, which is smart sequencing." The multi-exercise narrative has an opportunity for this arc framing but currently produces a concatenation, not a composition.

Gap: Feel Rating Not Integrated into Narrative

The user just rated their session as "Brutal" — but the post-session narrative makes no reference to this. "Session felt brutal — recovery nutrition in the next 90 minutes matters today" would be a trivially easy addition and would make the narrative feel reactive to the user's input, not pre-generated. This is the single highest-leverage narrative improvement available.

SHIP-READY for V1 HRV feature

HRV Baseline Implementation Review

Calibration Period: Sound

Requiring ≥7 readings before declaring a baseline is pragmatic and scientifically defensible. The median (vs. mean) is the correct choice — HRV has a right-skewed distribution and means are pulled by outlier high-HRV readings after intense sessions. Median is robust to these. Score: appropriate for consumer use.

Threshold Multipliers: Reasonable

Low threshold at 0.88× baseline (−12%) and good threshold at 1.02× baseline (+2%) are asymmetric in the right direction. HRV suppression (below baseline) is more diagnostically meaningful than elevation — elevated HRV can indicate good recovery or parasympathetic arousal from mild illness. The +2% threshold appropriately requires meaningful elevation for the "good" signal.

Concern: HRV Not Stamped to Session

HRV is read from HealthKit and stamped at session start. However, the hrvHistory dictionary stores date-keyed readings from health.latestHRV. If a user has multiple HealthKit readings per day (morning measurement + Apple Watch continuous), only one is stored. For the baseline computation, this is fine (daily granularity is correct). But for real-time readiness at session start, the relevant HRV is the most recent pre-training reading, not a daily summary. If the user trains at 6pm and their morning HRV was taken at 6am, 12-hour-old data is used. This may not matter for most users but should be documented.

Gap: HRV Trend Not Used

A single HRV reading vs. baseline is useful, but the 3-day trend direction is more predictive of training readiness. Consecutive days below baseline signal accumulated fatigue even if any single day is above threshold. The hrvHistory dictionary already stores the time series — lastThreeDays.allSatisfy({ reading < baseline * 0.95 }) would be a meaningful "chronic suppression" flag. Not required for V1, but a clear next step.

Note: Population Fallback Values

Before calibration, the engine uses hardcoded 55ms (low) and 65ms (good) thresholds. These are population-level norms pulled from general fitness literature. They will be systematically wrong for two groups: well-trained athletes (baseline HRV 70–100ms, so 65ms would always flag "good") and older or deconditioned users (baseline 30–45ms, where 55ms is unachievable). This is a known tradeoff — it resolves once calibration completes, but the first 7 sessions will use population-wrong thresholds.

Marcus Osei
Strength & Conditioning Coach · Practical Programming · Athlete Development
BETA — Coach quality needs work

Coaching Quality Review

Recommendation Engine: No Compound-Before-Isolation Rule

The exerciseScore() function ranks exercises by history, recency, and performance — but has no concept of compound vs. isolation priority. A score of 75 on Bicep Curl can outrank a score of 70 on Deadlift. In every evidence-based program, compound movements (squat, deadlift, bench, row, overhead press) are placed before isolation work within a session for peak neural drive. Without this constraint, the engine can surface Lateral Raises before Barbell Rows — which no qualified coach would allow.

Recommendation Engine: No Antagonist Pairing Awareness

Supersets are user-created. The engine has no awareness that training chest without back (or biceps without triceps) creates structural imbalances over time. Even a simple rule — "if this session includes push muscles, surface a pull option" — would improve programming quality significantly with minimal code.

daysSinceTrainedByRegion: Much Better Than Set-Based

The shift from "appeared in last 2 sessions" to calendar-day tracking is a meaningful improvement. It correctly handles users who train the same muscle group twice in a week, since the day-count accurately reflects recovery time rather than session-count adjacency. A user who trains legs Monday and legs Thursday has 3 days of recovery — the new engine sees this correctly.

Cold-Start Plan: Compound Selection Is Random-ish

The first workout takes the first compound exercise per region sorted alphabetically. Alphabetical sorting means "Barbell Back Squat" beats "Bulgarian Split Squat" for legs — which is actually fine for most users (barbell compound first is correct). But for chest, alphabetical might surface "Cable Fly" before "Barbell Bench Press" if the exercise library is sorted that way. The selection logic needs a quality filter, not alphabetical ordering. A flag like isPrimaryCompound: Bool on Exercise would solve this permanently.

Missing Coaching Concepts

P1
Rep Range Guidance Missing
App suggests weight adjustments but never suggests rep ranges appropriate to goals. Hypertrophy (8–12), strength (3–6), and endurance (15–20) require different ranges. Without goal selection, all users get the same weight-escalation logic regardless of what they're actually training for.
P2
No Warm-Up Set Protocol
The app has no concept of warm-up sets. Users logging a first "working set" at their target weight for exercises like Deadlift or Squat are not being coached to warm up. This is a safety and performance gap, particularly for heavier lifters in the target user profile.
P2
Readiness "Forced Workout" Path Missing
The readiness score can be very low (e.g., 2/10). There is no path where the app says "today might be better as a light technique day or mobility work" — it either shows the full plan or nothing. A reduced-intensity fallback is standard coaching practice for low-readiness days.

Product Readiness Assessment

Core Loop: Ready

Start workout → log sets → end with feel rating → see narrative → plan next session. This loop is now fully implemented and friction-minimal. The 3-tap minimum to complete a set (tap weight, tap reps, tap complete) is industry-standard. The feel-as-exit is a genuine differentiator from every major competitor.

Onboarding: Not Mapped

New user flow unclear from code review. When a user opens the app for the first time: What do they see? How do they log body weight (required for readiness scoring)? How do they set their goals? How do they understand what H.O.N. means? The cold-start plan now gives a deterministic first workout — but only if the user knows to start one. The transition from "empty app" to "I understand what to do next" is the highest churn point and appears unaddressed.

Settings Discoverability

Key settings (rest timer duration, weight unit kg/lb, body weight) appear to be buried in a settings screen. Power users will find them; new users won't know they exist. The weight unit toggle in the exercise card (tap the KG/LBS label) is discoverable but not obvious. At minimum, a first-session prompt for "kg or lbs?" would prevent the common complaint "it defaulted to kg and I log in lbs."

Concern: No Social / Accountability Layer

The target user ("falls and gets back up" lifter, Huberman listener) is intrinsically motivated but socially influenced. The share sheet after workout exists but feels like an afterthought. Accountability — even private, like "share to a training partner" — is a proven retention multiplier. Not a V1 requirement, but should be in the roadmap for 90-day retention improvement.

05 What's Working — Don't Touch These

Narrative Engine Architecture

The phrase-bank + assembly + format rotation design is extensible without being over-engineered. Adding a new format, new phrase category, or new analysis field doesn't require architectural changes — it's additive. The ExerciseAnalysis struct is a well-scoped value type that's easy to extend.

Verdict: Architecture is right. Grow by adding data, not by restructuring.

iCloudSync Graceful Fallback

The nil-container fallback pattern is exactly right. The sync infrastructure works when available and disappears silently when not. No crash, no error state, no user-visible failure. This is the correct architecture for a feature that may or may not be available depending on account type.

Verdict: Ship as-is. Re-enable entitlements when Team account is available.

Celebration Queue System

The celebration queue in WorkoutTabView correctly sequences PRs, streaks, and session completion. The pendingFirstWorkoutCelebration AppStorage flag that survives crash/force-quit is a thoughtful edge case. The delayed trigger (asyncAfter 0.7s) gives the feel sheet time to dismiss cleanly.

Verdict: Well-implemented. Don't refactor until the sequence needs to change.

ExerciseCard Layout

The three-column set row (SET / WEIGHT / REPS×TARGET) with kg/lbs toggle, history matrix, equipment badge, and drop-set panel is a complete and polished data-entry UI. The plate calculator sheet is a genuine quality-of-life feature. The swipe-to-delete UX is standard and correct.

Verdict: UI is mature. Don't add new columns or controls without a clear user need.

06 Prioritized Next Actions

Framing: The core loop is ready. The remaining work is in three tiers: (1) shipping blockers for any user, (2) quality gaps for target users, (3) depth features for power users. Do tier 1 before shipping anything publicly.

Tier 1 — Shipping Blockers

T1-A
Safe area padding on WorkoutFeelBar (NEW-03)
Add .safeAreaPadding(.bottom) or GeometryReader-based bottom inset to WorkoutFeelBar. Tappable targets overlapping the home indicator fail App Store review on accessibility grounds and frustrate users on device.
T1-B
Onboarding: minimum viable path (NEW-09)
New user opens app → blank workout tab with "Start Workout" button. Before first workout, prompt: (1) kg or lbs, (2) body weight. After first workout, briefly explain narrative. Three screens, one time. This alone cuts first-session churn significantly.
T1-C
Feel bar visual state before first set (NEW-01)
Reduce feel tile opacity to 0.4 and disable tapping until hasAnyCompletedSets. A one-line change prevents the cognitive dissonance of "how did that feel?" before any sets are logged.

Tier 2 — Quality for Target Users

T2-A
Integrate feel rating into post-workout narrative (NEW-05)
Pass feelRating into WorkoutNarrativeEngine.generateNarrative(). Add a feel-reactive sentence: "Brutal today — the recovery window matters." / "You rated this Strong — a sign the program is calibrated right." High perceived personalization, low implementation cost.
T2-B
Add streak/comeback context to narrative (NEW-08)
Pass store.currentStreak and days-since-last-session into the narrative engine. Surface in sparse or standard format: "Day 21 — consistency is the compound interest of training." Behavior research strongly supports this type of identity reinforcement.
T2-C
Compound-before-isolation priority in recommendation (NEW-06)
Add a isCompound: Bool property to Exercise model (or derive from body region + equipment). In generatePlans(), sort compound exercises first within each recommendation. This is the most impactful training logic fix for users who run full-body or push/pull sessions.
T2-D
Weekly volume awareness in recommendation engine (NEW-04)
In exerciseScore(), add a penalty for exercises in muscle groups that have accumulated ≥12 completed sets in the current rolling 7-day window. Prevents the engine from programming de-facto overtraining for consistent users within 4–6 weeks.

Tier 3 — Power User Depth (Post-Launch)

T3-A
HRV 3-day trend signal (NEW-12)
Add computeHRVTrend(days: 3) -> HRVTrend that returns suppressed/neutral/elevated based on 3-day direction. Feed into readiness narrative for pro users who understand HRV variability.
T3-B
Deload cycle detection (OLD-01)
After 4 weeks of ≥3 sessions/week, surface a "deload suggestion" card that recommends dropping volume by 40% for one week. Uses store.workoutLog already available.
T3-C
Goal-based rep range guidance (NEW-13)
Add a user goal field (strength/hypertrophy/endurance) in settings. Adjust target rep ranges in the recommendation engine accordingly. Moves H.O.N. from "records what you do" toward "programs what you should do."

ActiveWorkoutView.swift file split (NEW-10)
Extract ExerciseCard, SetRow, RPERow, WorkoutFeelBar, RestTimerBanner into separate files. No behavioral change — purely maintainability. Do this before the next sprint that adds to the workout view.
