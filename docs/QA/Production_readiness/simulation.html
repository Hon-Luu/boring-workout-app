Simulation Protocol

20 scenarios · Cohort × scenario priority matrix · Results capture template · Last updated: 2026-05-29
Section 1 — How to Run Simulations
Simulations are structured manual test runs that replicate real user behavior end-to-end, from first launch through long-term feature interaction. Each run assigns one cohort member (from cohort.html) to one scenario, then follows the step-by-step instructions exactly as that persona would behave — not as a developer. The tester adopts the persona's technical comfort level, logging habits, and emotional context. Runs are done on a physical device or the iOS Simulator with a fresh or seeded data state as specified. Results are captured using the template in Section 4 and filed in docs/uat/. Prioritize the 10 highlighted combos in the matrix first; run secondary combos as time permits. A scenario is only considered complete when success criteria are verified, not just when the steps are finished.
Section 2 — 20 Simulation Scenarios
SIM-01
First Open & Onboarding — Profile Setup, Goals, Equipment
Est. 10–15 min Trigger: Fresh install
What We're Testing
Onboarding completion rate, clarity of goal/equipment selection, profile save persistence, time-to-first-value under 3 minutes.
Trigger Condition
App installed fresh, no prior data. Reset simulator or delete app and reinstall from build.
Steps
  1. Launch app fresh. Note time to first interactive screen.
  2. Complete onboarding as the assigned persona: select gender, age, fitness level, primary goal.
  3. Select equipment type that matches persona's access.
  4. Complete profile setup (name, bodyweight if prompted).
  5. Note any confusing labels, ambiguous options, or missing choices.
  6. Reach the home screen. Note total elapsed time from cold launch.
  7. Background the app and reopen. Verify onboarding does not repeat.
  8. Check that profile data persisted correctly in Settings tab.
Priority Cohort Members
U-02 U-06 U-11 U-13 U-01 U-17
Pass
Onboarding complete in <3 min. All selections persist. Home screen loads with correct persona context. No repeat onboarding on relaunch.
Fail
Onboarding >5 min. Any selection lost on relaunch. Crash during flow. Missing equipment options for persona's setup.
SIM-02
First Workout — Empty Start, No History
Est. 15–20 min Trigger: First session ever
What We're Testing
Cold-start workout usability, zero-history state handling, first set logging, coaching narrative on first ever set, session completion and celebration.
Trigger Condition
Immediately after onboarding (SIM-01) or seeded with profile-only state (no workout log).
Steps
  1. Navigate to Workout tab. Note what the empty state shows.
  2. Start a new workout without using a template.
  3. Add 3 exercises that match the persona's equipment access.
  4. Log 3 sets per exercise — varied reps and weight.
  5. Note whether coaching narrative appears after first set. Read it as the persona would.
  6. Log RPE on at least one set (if persona would). Note if RPE is discoverable.
  7. Complete the workout. Note post-session screen and any celebration.
  8. Navigate to Progress tab. Verify session appears in history.
Priority Cohort Members
U-02 U-06 U-10 U-11 U-01 U-17
Pass
Session completable with zero prior knowledge. Coaching narrative appears after first set. Post-session celebration fires. Session visible in Progress tab immediately.
Fail
Any blocking UX requiring prior knowledge. Coaching narrative absent. Crash. Session not persisted. Celebration not firing on first-ever completion.
SIM-03
Second Workout Same Week — Previous Session Data Visible
Est. 15 min Trigger: 2nd session within 7 days
What We're Testing
Previous session data displayed correctly, weight suggestions appearing, coaching narrative references prior session, readiness score updated.
Trigger Condition
At least 1 session logged. Start a second session with 2–4 of the same exercises as the first.
Steps
  1. Open Workout tab. Note if home screen shows "last session" context.
  2. Start new workout. Add the same primary lift as session 1.
  3. Verify last session's sets are shown as reference (weight/reps from prior session).
  4. Log the first set. Verify weight suggestion appears and is reasonable.
  5. Log 3 sets per exercise. Accept suggested weights for 2 exercises, override for 1.
  6. Complete session. Check if coaching narrative references progress vs. last session.
  7. Navigate to StrengthLab tab. Verify new e1RM data is plotted.
Priority Cohort Members
U-01 U-03 U-07 U-02 U-15
Pass
Prior session data visible as reference. Weight suggestion present and within ±20% of prior weight. Coaching narrative contextually appropriate. e1RM updates in StrengthLab.
Fail
No prior session data shown. Weight suggestion absent or wildly off. Coaching narrative generic/ignores context. StrengthLab not updated.

Trainer tab graceful empty state, plan generation from profile-only data, plan quality and relevance with zero history.

  3. If a plan is generated, evaluate: does it match the persona's goal and equipment?
  4. Check coach notes — are they generic or personalized?

  6. Tap into plan details. Verify exercises match persona's equipment access.

Fail
Crash or blank screen. Plan suggests equipment persona doesn't have. Generic filler plan with no persona relevance. Confusing empty state with no direction.

  4. Identify one stalled exercise in the log. Does the trainer acknowledge or address it?
  5. Check if suggested exercises vary from the user's existing routine (progressive overload)
  6. Evaluate whether "today's hints" on Home tab aligns with trainer plan.

Fail

  4. Start a new session. Note if suggestions are slightly conservative vs. pre-gap.
  5. Complete session. Note if gap message clears or persists.
  6. Reopen app same day. Verify gap message does NOT repeat.
Priority Cohort Members
U-07 U-13 U-16 U-06 U-17
Pass
Gap message fires once, correct threshold (7+ days). Tone is welcoming. Message does not repeat same day. Weight suggestions conservatively adjusted.
Fail
Gap message fires at wrong threshold. Message repeats after completion. Tone is guilt-inducing or dismissive. Wrong historyKey used ("Short Gap" vs "Long Gap" swap).
SIM-09
Return After 21+ Day Gap — Comeback Flow
Est. 15 min Trigger: 21+ days since last session
What We're Testing
Long gap comeback flow, strength decay model accuracy at 3+ weeks, "Long Gap" phrase bank key triggering correctly, readiness baseline reset, recommitment mechanics.
Trigger Condition
Seed last session 21–45 days ago with good prior history (8+ sessions). This is the "Long Gap" historyKey threshold.

  3. Check StrengthLab tab — does strength decay model show estimated loss?
  4. Navigate to trainer tab — does the plan account for the return-from-gap state?
  5. Start a session. Note if weight suggestions are reduced from pre-gap levels.
  6. Log 3 sets. Verify "Long Gap" historyKey triggers in coaching narrative (not "Short Gap").
  7. Complete session. Note if a re-onboarding/recommitment moment is offered.
Priority Cohort Members
U-16 U-14 U-08 U-07 U-13
Pass
"Long Gap" phrase bank key fires (not "Short Gap"). Strength decay displayed. Weight suggestions reduced. Comeback tone is welcoming. Trainer plan is appropriately conservative.
Fail
Wrong gap key fires. No strength decay display. Weight suggestions unchanged from pre-gap. Comeback message absent or uses wrong phrase bank key.

SIM-11
Weight Progression Decision — Is the Suggestion Correct?
Est. 15 min Trigger: 3+ sessions on same exercise
What We're Testing
Weight suggestion accuracy after successful and failed previous sets, Epley/Mayhew e1RM routing at rep boundaries, suggestion conservatism vs. aggressiveness for different fitness levels.
Trigger Condition
3+ logged sessions for the same exercise with clean progression data. Test across rep ranges: 1 rep, 5 reps, 10 reps, 13 reps.
Steps
  1. Start session. Add a "main lift" (squat, bench, or deadlift).
  2. Note the weight suggestion for the first set. Is it appropriate for the persona?
  3. Log a set of 10 reps (Epley range). Log next set of 13 reps (Mayhew range).
  4. Verify e1RM calculation uses correct formula at boundary (10 vs. 11 reps).
  5. Log a failed set (missed target reps). Note if next suggestion adjusts down.
  6. Log a set exceeding target. Note if next suggestion increments up.
  7. For U-03: cross-reference app e1RM with manual Epley calculation.
Priority Cohort Members
U-03 U-01 U-15 U-08 U-05
Pass
e1RM matches manual calculation ±1%. Suggestion adjusts after failure/success. Epley used for 2–10 reps, Mayhew for 11–15 reps, exact at 1 rep. >15 reps returns 0 (excluded).
Fail
e1RM off by >2%. Wrong formula at rep boundary. Suggestion unchanged after failure. >15 reps returning non-zero e1RM. Suggestion off by >20% of expected value.
SIM-12
RPE Logging — Last Set as RPE Gate
Est. 10 min Trigger: Any session
What We're Testing
RPE discoverability, RPE chip label accuracy (6–10 range), RPE-adjusted e1RM computation, RPE persisting correctly per set, RPE influencing readiness on next session.
Trigger Condition
Any session with at least 1 completed set. Test RPE entry for both experienced (U-03) and beginner (U-02) personas.
Steps
  1. Log a set. Find the RPE entry option. Note how many taps it requires.
  2. Tap through all RPE values 6–10. Verify labels map to correct intensity words.
  3. Select RPE 8 on a set with 5 reps at 100 kg. Verify RPE-adjusted e1RM differs from non-RPE e1RM.
  4. Leave RPE blank on one set. Verify it defaults correctly (no crash, no assumed value).
  5. Complete session with high RPE (9–10) on all sets. Check readiness next session.
  6. As U-02 (beginner): is RPE clearly explained? Can she skip it without penalty?
Priority Cohort Members
U-03 U-09 U-02 U-01
Pass
RPE 6–10 range enforced. Word chips map correctly. RPE-adjusted e1RM differs from plain e1RM. Blank RPE does not crash. High RPE session influences next-day readiness.
Fail
RPE outside 6–10 accepted. Chip labels wrong. RPE has no effect on e1RM or readiness. Crash on blank RPE. Beginner with no RPE knowledge blocked from completing set.

  6. Open StrengthLab. Verify flat e1RM trend is visible for this exercise.
  7. Check progressTrend value in HomeCache — should reflect plateau state.
Priority Cohort Members
U-01 U-03 U-08 U-15 U-05
Pass

SIM-15
Interrupted Workout — Abandon Mid-Session
Est. 10 min Trigger: Active workout in progress
What We're Testing
Partial session data persistence, stale active workout state recovery, data integrity after abandon, whether partial session counts toward streak, clear abandon/save flow.
Trigger Condition
Start an active workout with 2+ exercises logged. Simulate interruption: kill app, wait 5 min, reopen. Also test explicit abandon button.
Steps
  1. Start a workout, log 2 exercises (3 sets each).
  2. Force-quit the app (kill from app switcher). Reopen after 30 seconds.
  3. Note whether active workout is restored or lost.
  4. If restored: verify logged sets are still there.
  5. Explicitly abandon the session using the UI abandon path.
  6. Check if partial data was saved or discarded.
  7. Start a NEW session the same day. Verify the abandoned session does not interfere.
  8. Check Progress tab: does abandoned partial session appear in history?
Priority Cohort Members
U-04 U-10 U-17 U-07 U-02
Pass
Active workout restored after kill. Logged sets intact. Abandon flow is clear and intentional. Partial session saved or cleanly discarded (not data corruption). New session starts cleanly.
Fail
Kill destroys logged sets. No active workout recovery. Stale session blocks new session start. Partial session causes UI bugs in Progress tab. Crash on abandon.
SIM-16
Extreme / Invalid Inputs — Validation and Robustness
Est. 20 min Trigger: Any session, adversarial inputs
What We're Testing
Input validation on all set fields, crash resilience at extreme values, analytics graceful degradation with invalid data, no NaN/Infinity in e1RM, no array overflows.
Trigger Condition
Persona U-21 (max input). Test all boundary conditions systematically.
Steps
  1. Log a set with weight = 999 kg. Note if accepted, rejected, or clamped.
  2. Log a set with 0 reps. Note behavior.
  3. Log a set with 0 kg (bodyweight). Note behavior (should be valid for U-11).
  4. Log a set with 100 reps. Note e1RM calculation (should return 0, excluded).
  5. Add 20 sets to a single exercise. Note if any crash or rendering issue.
  6. Log 50 exercises in one session. Note performance and rendering.
  7. Enter negative weight (if field allows). Note behavior.
  8. Enter RPE = 11 (out of range). Note behavior.
  9. Navigate to all analytics views after extreme data. Verify no NaN/crash.
Priority Cohort Members
U-21 U-11 U-03
Pass
No crash on any input. Invalid values rejected with clear error or silently clamped. Analytics show 0 or exclude data instead of NaN/Infinity. 0 kg accepted (bodyweight exercises).
Fail
Any crash from boundary input. NaN or Infinity in any display field. Analytics broken after extreme data. Negative weight accepted and causing wrong e1RM. Array crash at high set count.

U-04 U-20 U-01 U-06
Pass

  7. Check exercise history: are barbell and dumbbell sessions separate or merged?

Priority Cohort Members
U-20 U-04 U-10 U-01 U-15

Equipment chip row visible and tappable. Correct variant resolved. Dumbbell weight shown as per-hand. Swap doesn't corrupt prior barbell history. No crash during swap.

SIM-20
Bad Actor Scenario — Fake PRs, Gaming Streak, Logging Without Training
Est. 15 min Trigger: Deliberate gamification abuse
What We're Testing
Gamification integrity, whether fake PRs trigger celebrations at absurd values, whether minimal-effort sessions count as full streak entries, analytics resilience to manipulated data.
Trigger Condition
Personas U-21 and U-22. Test both input abuse (U-21) and engagement gaming (U-22).
Steps
  1. Log a "PR" at 300 kg squat × 1 rep. Note if celebration fires.
  2. Log 30 days of "sessions" each containing 1 set of 1 rep at 1 kg. Note streak count.
  3. Check if 1-set sessions count as full workouts in analytics.
  4. Open StrengthLab with mixed real and fake data. Note composite score sanity.
  5. Verify that fake PRs don't permanently corrupt the "best performance" cache.
  6. Delete all fake sessions. Verify analytics recalculates correctly to real baseline.
  7. Check: does a 30-day streak of 1-set sessions look meaningfully different from a 30-day streak of real sessions?
Priority Cohort Members
U-21 U-22 U-03
Pass
App does not crash on absurd PRs. Gamification mechanics have reasonable thresholds (or are documented as intentionally gameable). Analytics recovers correctly after fake data deletion.
Fail
Crash on extreme PR. Analytics permanently corrupted by fake data (not recoverable). 300 kg PR fires same celebration as a real 10-rep PR. Streak indistinguishable between real and gamed sessions.

Section 4 — Results Capture Template

Copy this template for each simulation run. File in docs/uat/ with filename SIMxx-Uxx-YYYY-MM-DD.md. Or use the uat command for quick entries.

Run Metadata
Scenario: SIM-__ · Persona: U-__ · Tester: [name] · Date: YYYY-MM-DD · Device: [iPhone model / Simulator] · Build: [commit SHA or build number]
Data Seed State
Describe the initial data state: fresh install, seeded sessions (how many, over what period), any prior test runs that may have left data.
Persona Behavioral Notes
How did you embody the persona? What technical comfort level did you simulate? Any deliberate behaviors from the persona's logging profile?
Steps Completed
List each step and whether it completed as expected. Note any deviations. Example: "Step 3 — weight suggestion not visible; had to scroll down to find it."
Pass / Fail Verdict
PASS / FAIL / PARTIAL — with one sentence explaining the verdict.
Bugs Found
List each bug: description, reproduction steps, severity (P0/P1/P2), file reference if known. "None found" is a valid entry.
UX Observations
Non-bug observations: confusing flows, unexpected behavior that isn't a bug, tone/language feedback, anything that would have confused the real persona.
Screenshots / Evidence
Attach paths to relevant screenshots (use ./scripts/hon-simulate to capture). Reference specific screen state if describing a visual bug.
Follow-Up Actions
Any items to add to BACKLOG.md, UAT_INBOX.md, or items needing engineering follow-up. Cross-reference by backlog ID if known.
