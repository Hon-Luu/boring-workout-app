H-01
Readiness coaching and HON return message directly contradict each other on gap days
SC-04 Gap Return ReadinessEngine.swift ~L292 HONMessageLibrary.swift ~L199
›
What happens
When a user returns after a 7–14 day gap, two coaching outputs fire with no integration between them. Simultaneously on the same screen:
"Don't hold back today — this is the kind of day you make progress on."
ReadinessEngine.narrative() at score ≥ 80
"Start at 70%, not hard. One solid session sets the trajectory."
HONMessageLibrary.returnAfterLapse (gap < 14 days)
These are not just different in tone — they are actionably contradictory. A user returning after an 8-day gap is told to work at full intensity on one part of the screen and at 70% on another. Additionally, the Readiness Factors list can simultaneously show a red "Been a while since your last session" factor alongside a high score and a "push hard" narrative — the factor says the gap is a problem but the coaching says it isn't.
Persona impact
P1 Returner — critical. Highest-stakes coaching moment.
Proposed Fix
Add an isReturnAfterGap(daysSinceLast:) check in ReadinessEngine.narrative(). When daysSinceLast >= 7, override the coaching note to a return-appropriate register regardless of score:

"You've had time to recover — ease back in and build momentum. Intensity next session."

Also suppress or reframe the "Been a while" Factors entry when a positive return HON message is active — the same event shouldn't be flagged as negative in one place and celebrated in another.

H-02
Two different Epley implementations produce different e1RM values for sets with reps > 10
SC-07 Cross-Tab Walk ExerciseInsightsView.swift ~L358 Models.swift ~L253
›
What happens
Two separate e1RM calculation functions exist in the codebase:
// Models.swift — SetRecord.e1RM(weight:reps:) case 2...10: return epley(w, r) // Epley formula case 11...20: return mayhew(w, r) // Mayhew formula — different result // ExerciseInsightsView.swift ~L358 private func epley(_ s: SetRecord, ...) -> Double { return s.reps == 1 ? load : load * (1 + Double(s.reps) / 30.0) // ↑ Epley for ALL rep counts — no Mayhew branch }
For reps ≤ 10: both produce identical values. For reps 11–20 (common: sets of 12, 15): they diverge. Example: 20 kg × 12 reps — Epley gives 28.0 kg, Mayhew gives ~27.0 kg. Progress views use the Mayhew path; ExerciseInsightDetailView's chart uses the local Epley path. The skeptic who spots 28.0 in one chart and 27.0 in another cannot know which is correct.
Persona impact
P2 Skeptic — trust-breaking. Numbers that don't match each other have no obvious winner.
Proposed Fix
Delete the local epley() function in ExerciseInsightsView.swift and replace all calls with SetRecord.e1RM(weight:reps:) (or effectiveE1RM(equipment:) for equipment-aware variant). There should be exactly one e1RM calculation in the codebase. The ExerciseInsightDetailView data pipeline that constructs dataPoints should use SetRecord.e1RM() rather than its local function. Effort: low — one function deletion + reference update.

H-03
e1RM term used in post-session celebration before it is ever explained
SC-02 First Workout SC-01 Onboarding FirstWorkoutCelebrationSheet.swift ~L9
›
What happens
Onboarding has three pages (goal selection, equipment selection, brief intro). None of them define e1RM, RPE, INOL, or PSI. After the user's very first session, FirstWorkoutCelebrationSheet fires and says:
"Come back tomorrow to start seeing your e1RM trend."
A P3 beginner reads this and sees a term that has no meaning to them. The celebration moment is diluted because the most forward-looking line contains an unexplained acronym. RPE compounds this: coaching output from WorkoutNarrativeEngine and EmergentInsightEngine references RPE in analytics context. The onboarding never surfaces it, so the first time a beginner sees "RPE 7" in an insight card it reads as noise.
Persona impact
P3 Beginner — first impression corrupted by jargon P1 Returner — mildly affected
Proposed Fix (choose one)
Option A — Inline expansion (recommended): Replace "e1RM trend" in FirstWorkoutCelebrationSheet with "strength estimate (e1RM) trend." Add a parenthetical explanation of RPE the first time it appears in any coaching card — "RPE 7 means 7 out of 10 effort — 3 more reps left in the tank." No onboarding changes needed.

Option B — Onboarding glossary page: Add a 4th onboarding page "Three numbers that matter" covering e1RM, RPE, and streaks. More upfront investment but users arrive informed. Onboarding goes from 3 to 4 pages.

M-01
HON notification fires same morning if workout completed before 8 AM
FC-03 Protocol 7 Regression HONMessageScheduler.swift ~L19
›
What happens
The notification trigger is built without a day component:
var comps = DateComponents() comps.hour = 8 comps.minute = 0 // comps.day is never set let trigger = UNCalendarNotificationTrigger(dateMatching: comps, repeats: false)
With no day component, UNCalendarNotificationTrigger fires at the next occurrence of 08:00. For an early-morning user who finishes a session at 7:45 AM, the HON coaching notification fires 15 minutes later on the same morning — not the intended "next day motivation" behavior.
Proposed Fix
Set comps.day to tomorrow's calendar day before creating the trigger, or use a time-interval trigger: UNTimeIntervalNotificationTrigger(timeInterval: secondsUntil8AMTomorrow, repeats: false). Effort: very low — one date calculation added.

M-02
4–6 day gap produces no acknowledgement — the WelcomeBackCard threshold is too high
SC-04 Gap Return HONHabitEngine.swift ~L166 HomeView.swift WelcomeBackCard ~L286
›
What happens
Both the HONHabitEngine gap message threshold and the WelcomeBackCard threshold require 7+ days of inactivity to fire. A user who trains Monday, takes Tuesday–Saturday off (5 days), and returns Sunday sees nothing that acknowledges the gap. This is the most common real-world gap pattern. The app is silent when the user is most likely to feel mild uncertainty about returning.
Persona impact
P1 Returner — the 4–6 day gap window is where most real gaps live
Proposed Fix
Lower the WelcomeBackCard threshold to 4 days and add a separate "short gap" message pool to HONMessageLibrary for gaps of 4–6 days. These messages should be lighter in tone than the 7+ day messages — no mention of "break" or "days away," just a forward-looking welcome:

"Ready to pick up where you left off?"

The HONHabitEngine lapse threshold can stay at 7 days since that path produces more substantive coaching.

M-09
Zero-rep alert blocks without helping a user who genuinely failed a set
SC-06 FC-01 ActiveWorkoutView.swift ~L783
›
What happens
Alert text: "No Reps Entered — Enter at least 1 rep before completing a set." This is correct for the most common case (forgot to enter reps). M-10
~88% of HON coaching messages are generic — only two categories reference real user data
Protocol 8 Coaching Tone Audit HONMessageLibrary.swift
›
What happens
Auditing all message pools — only two are data-specific:
• patternFlag: references dayName and pct ✓
• rampDetection: references count and avg ✓
• All others (sessionMilestone, weeklyCount, returnAfterLapse, driftDetection, deloadDetection, streakMilestone, specialMoment): generic — could apply to any user.

specialMoment is the most notable gap: messages like "Something's different about today's session" are supposed to fire when the app detects something notable, but none of them state what that something is. Proposed Fix (in priority order)

2. driftDetection: Add the specific category: "[Push] volume is down two weeks running. One session resets it."

3. returnAfterLapse: Add the last exercise logged: "You're back. Last time you were here: [exercise]."

M-12
Feel selector sheet drag-to-dismiss is disabled but Skip button is not prominent enough
FC-04 WorkoutTabView.swift ~L669
›
What happens
FeelSelectorSheet uses .interactiveDismissDisabled(true) — the sheet does not respond to a swipe-down. Users who try the standard iOS dismiss gesture get no response and may think the app is frozen. The "Skip" button exists and functions correctly, but it's positioned as a secondary action (small, below the emoji row) when it is actually the most common path for users who find the feel prompt friction-heavy.
