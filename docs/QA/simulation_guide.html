H.O.N. UX Simulation Guide Rev 2 · 2026-05-26

How to set up, run, and interpret real-experience simulations — functional accuracy and felt quality

What This Is — And Why It Covers More Than You Think

There are three distinct questions you can ask about an app. An audit asks: does the app do what it's designed to do? A simulation asks: what does it feel like to be a user of this? And a UI evaluation asks: does it look and feel like an app you'd trust with your health data? These are not the same question, and they cannot be answered by the same method.

The audit finds broken. A feature works or it doesn't. Numbers match or they don't. This is testable against code.

Simulation finds wrong. A feature can work perfectly and still feel cold, confusing, or untrustworthy. This is testable only through experience.

UI evaluation finds ugly. Visual hierarchy, animation quality, coaching tone, information density — these determine whether someone opens the app on day 8. None of them appear in a test suite.

The curse of knowledge: You built this app. You chose the colors. You wrote the coaching messages. You know where every button is. Simulation is not about pretending to forget — it's about creating conditions where your knowledge advantage is neutralized by real friction. A hesitation is a finding regardless of the rationalization. Write it down first. Explain it later.

The Four Simulation Personas

You cannot simulate everyone. Four personas cover the behavioral range that matters for this app. Each has a distinct emotional posture, data vocabulary, relationship with consistency, and sensitivity to visual and tonal signals.

P1 — The Returner
Trains 2–3×/week · Disappears for 1–3 weeks · Mild guilt on return

Consistency is aspirational, not reliable. They return with hope and low-grade guilt. They want the app to welcome them back, not remind them how long they've been away. They notice emotional tone before content.

VocabularyKnows reps, sets, weight. Doesn't know e1RM, RPE, INOL.
InputsWeight and reps only. Skips anything that adds steps.
UI sensitivityNotices the emotional register of the first screen. Does it feel welcoming or clinical?
Will they skip?Feel rating, readinessBefore, RPE — anything that feels like homework.
Key simulationLog 4 sessions, simulate 6 days away, return. Observe the first 15 seconds.
P2 — The Skeptic
Data-oriented · Will verify numbers · Won't trust opacity

Works in a field that uses data. Attracted to the analytics promise but won't buy a number they can't verify. One unexplained discrepancy breaks the relationship permanently. Evaluates visual design as a signal of rigor.

VocabularyKnows all fitness terms. Understands progressive overload, periodization, 1RM.
InputsPrecise. Will re-enter if mistyped. Will use RPE if clearly explained.
UI sensitivityLabels that are ambiguous. Numbers without units. Charts without axes. Dense UI = competent; cluttered UI = untrustworthy.
Will they skip?No. They want data density and will explore every tab.
Key simulationAfter 5 sessions: manually calculate a metric and compare to what the app shows. Navigate all tabs looking for the same number.
P3 — The Beginner
First fitness app · No vocabulary · Easily overwhelmed · Easily delighted

Starting. Doesn't know what e1RM means. Interprets every piece of coaching as authoritative because they have no baseline to push back. Jargon makes them feel stupid, not informed. Beautiful UI makes them feel safe.

VocabularyReps, sets, weight. "Bigger number = stronger" is their mental model.
InputsWhatever the placeholder says. Will enter whatever number looks reasonable.
UI sensitivityInformation overload. Screens that require reading before acting. Anything that looks like a quiz.
Will they skip?Onboarding steps when "Skip" is visible. Any screen that feels like a form.
Key simulationComplete full onboarding without reading instructions. Log the first session using only what the UI tells you. Count every word you don't understand.
P4 — The Optimizer
4–5×/week · Wants specificity · Drills into analytics · High bar for coaching quality

Already fit and disciplined. Not using the app to build a habit — wants it to tell them something they don't already know. Generic coaching actively annoys them. Expects features to earn the screen real estate they occupy.

VocabularyAll fitness terms. Researches training science. Will notice if advice contradicts literature.
InputsEverything. RPE, drop sets, toFailure. They'll use every enrichment field if explained.
UI sensitivityGeneric phrasing, shallow drill-downs, repeated insight patterns. Rewards density when it earns attention.
Will they skip?No. But they'll dismiss features that feel shallow within one use.
Key simulationAfter 10 sessions: is the experience qualitatively different from session 3? Does the app know more, or just display more?

Data State Setup

Every simulation requires a specific app state. You cannot simulate "session 1 experience" with 18 sessions loaded. HealthKit authorization is an explicit state variable — it changes readiness scoring, CSV health columns, and home tile values in ways that fundamentally alter the experience.

Before anything: Export a backup and store it safely. Every simulation that modifies data must be restorable. Never simulate against your only copy of real session data.

Canonical STATE-5 JSON Specification

Use this spec when crafting STATE-5 via JSON editing rather than live logging. It ensures coaching signals are meaningful and scenarios are repeatable.

  • Exercises: Use 3 exercises that repeat across all 5 sessions — e.g., Dumbbell Bench Press, Lat Pulldown, Leg Press. Using 5 different exercises produces almost no coaching signal.
  • Progression: Session 1→5, increase weight by 2.5 kg on each exercise per session. This produces visible e1RM trending that SC-07 can verify.
  • Sets per exercise: 3 sets each, reps 8–10. Do not include toFailure sets — those complicate analytics without adding signal for most scenarios.
  • Dates: 5 consecutive calendar days ending 3 days before today. This gives a gap that makes the readiness timing predictable without triggering WelcomeBackCard.
  • Feel rating: Include on sessions 3 and 5 (Normal, Strong respectively). Leave blank on 1, 2, 4 to simulate realistic skip behavior.
  • e1RM sanity check after import: For Lat Pulldown at 25 kg × 10 reps, you should see e1RM = 33.3 kg. If the Insights view shows something different, the import or the analytics pipeline has a problem.

Eight Simulation Protocols

Eight distinct methods for generating real-experience signal. They are not interchangeable — each exposes a different class of problem. Protocols 1, 2, and 8 are mandatory. Run at least five total.

1
First-Week Sprint Revised from Rev 1
State: STATE-0 · 5 structured sessions over real days · Personas: P1 or P3 · Effort: Medium
Mandatory

Reset the app completely and log 5 real sessions over 5–7 days. This is not open-ended natural use — each session has a specific observation focus. The goal is to experience how the app evolves from day 1 to day 5 under a persona's lens, with structured checkpoints to prevent rationalization.

SessionObservation focusUI dimension to evaluateCoaching to read verbatim
Day 1Onboarding flow. Can you complete it without reading instructions? What does the empty home screen tell you?First-impression aesthetics: hierarchy, color, density. Does it look like an app you'd trust?The home card after 0 sessions. What does it say? Is it specific or generic?
Day 2Second session UX. Does the weight pre-fill from session 1? Is the history matrix meaningful yet?Exercise card layout. Is the target weight hint visible without hunting for it?Post-set coaching after completing set 1. Quote it exactly.
Day 3The Progress tab for the first time — what can you see vs. what's still locked behind "not enough data"?Empty state quality across every card in Progress.The coaching note on Home after 3 sessions. Is it more specific than day 1?
Day 4Simulate tiredness: enter weights slightly lower than session 3. Select feel=Tired at session end.The feel selector sheet. Is it visually distinct from the rest of the session flow?Home coaching after logging Tired. Does it acknowledge the feel?

What to log during each session

  • First moment of confusion — screen, element, what you expected vs. what you found
  • Any field you skipped — which field and why (didn't understand / too much friction / didn't seem worth it)
  • Any coaching text that felt generic — quote it verbatim. "Generic" = could apply to any user in any session.
  • One visual element that earned your attention — what it was and why it worked
  • The last thing you saw before closing — verbatim. Did it make you want to come back?
2
Screen Recording + Think-Aloud
State: STATE-0 or STATE-GAP · 45–60 min · All personas · Effort: Medium
Mandatory

Record your screen and voice while logging one complete session from start to finish. Before every tap, narrate what you expect will happen. After every tap, narrate whether it matched. Review the recording — every gap between expectation and reality is a finding. Every silence longer than 3 seconds is a finding.

Setup

1
Enable iOS screen recording with microphone. Control Center → Screen Recording (long press) → enable microphone. Confirm audio is recording before starting.
2
The rule: narrate every expectation before you tap. "I expect this button to open the workout logger." Then tap. Then: "It did / it didn't because..." Do not skip steps. Silence is also signal.
3
Do not stop to fix anything. If you notice a bug or friction, say so aloud and continue. You are simulating, not debugging.
4
After the session: review the recording. Mark timestamps where: (a) expectation didn't match, (b) you hesitated, (c) you said something negative, (d) you said something positive, (e) you were silent for 3+ seconds. Each timestamp is a candidate finding.

What the recording captures that real-time observation misses

  • Scroll depth — did you reach the bottom of long cards, or stop scrolling early? If you stopped early, the section above the cutoff point is eating screen real estate it doesn't deserve.
  • Re-taps — tapping the same element twice. Tap-target feedback is insufficient, or the result was unexpected.
  • Voice tone on coaching text — flat reads indicate generic coaching; engaged reads indicate specific coaching. You can't fake engagement.
  • Which parts you skipped reading entirely — any text block you visually skip has failed to earn attention. Note the first word of every block you skimmed.

4
Persona Walkthrough
State: Varies · 90 min per persona · All personas · Effort: Medium
Structural

Each persona walks a scripted path through the app — not to test specific features, but to experience the app as a coherent product from their perspective. The script defines starting state and goal only. The path is the persona's.

PersonaStarting stateTheir goalThe question being answered
P1 ReturnerSTATE-GAP"I want to get back on track. What should I do today?"Does the app make returning feel easy, or does it highlight the failure? Is the tone shame-free?
P2 SkepticSTATE-10"I want to know if the strength numbers are accurate. Show me your work."Can the analytics withstand scrutiny? Can you find the same number in more than one place?
P3 BeginnerSTATE-0"I just want to log my workout and see if I'm getting stronger."Does the language make sense to someone with no fitness-app vocabulary? Does the visual design guide or overwhelm?
P4 OptimizerSTATE-LIVE"I want to find something in this data that surprises me — something I didn't already know."At 18 sessions, does the app offer genuine analytical depth, or well-formatted summaries of things you already knew?

Rules

  • Write the persona's 3 sentences before you start. Keep them visible. Refer to them whenever you're about to make a decision.
  • Stop when the persona would stop. P1 wouldn't spend 20 minutes in StrengthLab. P3 wouldn't keep scrolling once they feel lost. When the persona would quit, you quit — and log why.
  • Rate both task completion AND experience quality separately. A feature that works 100% but feels 40% is a finding. Don't conflate them.
5
Competitive Friction Test
State: External app + STATE-LIVE · 2 sessions + 1 comparison · Personas: P2, P4 · Effort: Low
Calibration

Use Strong or Hevy (both free) to log two real sessions. Then return to H.O.N. for the same session type. Every friction point you've normalized will surface as contrast. Every H.O.N. feature the competitor lacks will feel earned. Pay specific attention to: logging speed, dumbbell weight convention communication, post-set feedback quality, session summary depth, and whether the competitor's coaching text feels more or less generic than H.O.N.'s.

6
Cold User Test
State: STATE-0 on separate device · 15–20 min observed · Personas: Real person · Effort: Highest to arrange, highest fidelity
Highest fidelity — run after first cycle

Give the app to one person who fits your target user profile and has never seen it. Give them no instructions. Watch without helping. This is not a friendly demo — it is a calibration instrument.

The handoff script (read exactly this): "This is a fitness tracking app. Imagine you're starting a workout today. Use it however makes sense to you. I'm going to watch quietly — please don't ask me questions. Just talk through what you're thinking as you go."

Do not explain anything. Do not react to their choices. Take notes on: pauses longer than 3 seconds, double-taps, any spoken confusion or positive surprise, scroll depth, and abandonment points. In the 5-minute debrief: ask what was easy, what was confusing, what surprised them. Do not ask "did you like it?" — that produces politeness, not signal.

8
Aesthetic & Tone Impression Test New in Rev 2
State: STATE-LIVE · 30 min · Persona: P1 and P3 · Effort: Low
Mandatory

A structured visual and tonal evaluation. Run this protocol separately from any functional testing — aesthetic judgment is corrupted when you're also thinking about data accuracy. The goal: does this look like an app you'd recommend to a friend who cares about their health?

Part A — First Impression (5 min, no interaction)

1
Open the app to the Home screen. Do not scroll. Do not tap anything. Wait 30 seconds and write down the first 3 things your eye goes to. If those 3 things are not the 3 most important pieces of information on that screen, the visual hierarchy has failed.
2
Navigate to each tab in order (Home → Workout → Progress → Insights → Settings). Spend exactly 20 seconds on each without tapping. Rate each tab: does it communicate its purpose without requiring any interaction? (1 = clearly yes, 0 = requires a tap to understand what tab this is)
3
Screenshot the most information-dense screen you can find. Show it to someone who has never seen the app. Ask them: "What is the most important thing on this screen?" If their answer doesn't match what you intended, information hierarchy has failed.

Part B — Color Semantic Consistency (5 min)

Open Progress and Insights. For every color you see used for data: write down what you believe that color means. Then verify: is that meaning consistent across every screen where that color appears? Known expectations to check:

  • Green (positive/sage): Should mean "good" or "above target." Check: ExerciseHistoryMatrix cells, INOL zone labels, tier badges, completion indicators.
  • Red/rose: Should mean "caution" or "below target." Check: missed-rep set cells, negative INOL zones, deload indicators.
  • Amber: Should mean "active" or "your brand." Check: Set number labels, active set indicator, accent buttons. Does amber ever mean something negative? It shouldn't.
  • Secondary/muted: Should mean "supporting information." Check: date labels, unit labels, subtitle text. Are any primary data points accidentally muted?

Part C — Coaching Tone Audit (15 min)

Collect 10 coaching messages from across the app. Include: post-set narratives (3), home coaching notes (2), EmergentInsights (3), session summary messages (2). For each message, answer the three questions below and score it:

QuestionPassFailScore
Specific? Does this message contain data from this user's actual sessions — a weight, a date, an exercise name, a percentage change?References at least one specific data point.Could apply to any user who just completed a set.+1 / 0
Tone-appropriate? Does the emotional register match the physical outcome? A missed-target set should feel different from a PR.Language matches what the data says happened.Generic encouragement regardless of outcome (e.g., "keep it up!" on a 3-set failure).+1 / 0
Actionable? Does the message give you something to do differently next time, or just describe what already happened?Forward-looking: what to do next.Pure description: "you completed 3 sets."+1 / 0

Target: average score ≥ 2.0 across 10 messages. Below 1.5 = coaching layer needs a rewrite pass. Individual messages scoring 0 = candidates for immediate replacement.

Part D — Animation Quality Inventory

Trigger each interaction below. For each: does the animation add information (it communicates something), add delight (it feels satisfying), or is it noise (it happens but contributes nothing)?

  • Completing a set — the checkmark fill and row flash. Does it feel like an accomplishment?
  • Rest timer counting down — the progress bar animation. Does it feel urgent as time runs out?
  • PR banner appearing — the top-slide-in. Does it feel celebratory without feeling intrusive?
  • Sheet presentations (feel selector, weight suggestions, exercise swap) — do they slide up with intentional weight, or feel generic?
  • Tab switching — is the transition instant (good) or animated (potentially slow)?
  • Progress chart appearing — does the chart animate into view, or does data just appear?

Scenario Library — 14 Experiments

Each scenario has a precise starting state, a defined sequence of actions, and observation questions covering both functional accuracy and felt experience. Every scenario now includes a UI/tone observation column.

SC-01
First Open — What Does the App Teach You in the First 90 Seconds?
P3 Beginner P1 Returner · State: STATE-0

Setup

STATE-0. Open the app for the first time. Do not tap faster than you understand. Treat every screen as a decision point that requires active comprehension before proceeding.

Run + Observe (Functional)

  • Is readinessBefore asked during onboarding? Is RPE explained before it appears?
  • Does the app explain the dumbbell per-hand convention?
  • What does the Home tab show with no data? Does the empty state give a specific action?
  • Can you find the workout logger without help? Time yourself.

Observe (UI + Tone)

  • Does the onboarding feel like it wants to know you, or like a form? Tone matters on first impression.
  • Is the typography on the welcome screen legible at arm's length? (This is a real gym condition.)
  • Does the color palette communicate "health" or "finance"? First-impression aesthetics prime expectations.
  • Is the first coaching message warm or clinical? Quote it verbatim and rate: warm / neutral / clinical.
SC-02
The First Workout — Does the App Guide or Require Prior Knowledge?
P3 Beginner · State: STATE-0 (post-onboarding)

Setup

STATE-0, onboarding complete. Open a new workout session. Your exercise: Dumbbell Bench Press, 3 sets. You have 20 kg dumbbells (per hand). You have never used a fitness app before.

Run + Observe (Functional)

  • Did you enter 20 or 40? What visual cue drove that choice?
  • Log 3 sets: 8, 7, 6 reps. Was feel rating mandatory or skippable?
  • What was the post-set coaching? Did it reference the actual weight and reps?
  • What was the last screen before you closed? Quote it verbatim.

Observe (UI + Tone)

  • Does the set-row layout communicate its purpose without instruction? If you had never seen this before, would you know that the two input fields are weight and reps?
  • How many words on the active workout screen are jargon? Count them as P3.
  • Does completing a set feel rewarding? Rate the set-completion animation: underwhelming / appropriate / excessive.
  • Does the session end feel like an accomplishment or an administrative task?
SC-03
Confidence Threshold — Does Anything Change at Session 3 and Session 10?
P4 Optimizer P2 Skeptic · State: STATE-3 → STATE-10

Setup

Build STATE-3 and STATE-10 with the same exercises. Screenshot every analytics view at both states so you can compare directly side by side.

Run + Observe (Functional)

  • Is the confidence transition acknowledged? Any notification, message, or visual change at session 3 or 10?
  • Are more features available at session 10? Can you feel the difference?
  • Does the tilde (~) prefix disappear at the right session?
  • Does coaching text change in depth or specificity?

Observe (UI + Tone)

  • Does the visual "density" of the Progress tab increase meaningfully between session 3 and 10? Not just more numbers — does it feel richer?
  • At session 10, does any screen feel visually overcrowded because features unlocked but layout didn't adapt?
  • Does the BeginnerProgressCard graduation (session 10 trophy) feel earned or perfunctory?
SC-04
The Gap Return — Is the Language Shame-Free?
P1 Returner · State: STATE-GAP

Setup

STATE-GAP: 5 sessions completed, exactly 6 calendar days have passed. Do not open the app during the gap. Open on day 7 as if returning from real life.

Run + Observe (Functional)

  • What is the exact wording of the first element on Home? Paste it verbatim.
  • Does any text mention the absence, the gap, or missed sessions?
  • Is readiness presented as opportunity or deficit?
  • Does the WelcomeBackCard fire? What does it say exactly?

Observe (UI + Tone)

  • Does the first screen feel welcoming? Rate the emotional tone of the opening: welcoming / neutral / punishing.
  • Is the visual weight of the first card appropriate? A heavy, dark card on return can feel like a confrontation even if the text is neutral.
  • Does the color of any return message accidentally read as negative? Red or amber on a return message can undermine neutral text.
  • If you were genuinely feeling guilty about the gap, would this opening make it worse or better? This is the single most important emotional question in the product.
SC-05
Five Consecutive Days — Does Cumulative Fatigue Appear?
P4 Optimizer · State: STATE-5

Setup

Log 5 real or simulated sessions on 5 consecutive calendar days. Use the same exercises across sessions. Note readiness score before each session.

Run + Observe (Functional)

  • Does readiness decline across consecutive days? Plot the 5 scores.
  • On day 5, does coaching acknowledge that 5 consecutive days is significant?
  • Does the Trainer plan on day 5 adjust intensity?
  • Is the coaching on day 5 inconsistent with readiness? Coaching to push hard while readiness is low = contradiction.

Observe (UI + Tone)

  • Does the readiness score visually communicate urgency at consecutive day 5? A number without color, size, or contextual change is just a number.
  • Does the coaching on day 5 feel different in register? Advice to a fatigued athlete should sound different from day 1 advice. Does it?
  • Is there any visual "fatigue state" communicated on the Home screen? Or is the UI the same regardless of physiological state?
SC-06
The Zero-Rep Session — What Does the App Do With Impossible Data? Updated Rev 2
P2 Skeptic · State: STATE-ANOMALY

Setup

Important: The UI now rejects 0-rep sets with an alert. You cannot produce this condition through normal use. You must create STATE-ANOMALY by editing the backup JSON directly — set reps:0, isCompleted:true on one set entry, then import via Settings → Import Backup. Verify import succeeded before running this scenario.

Run + Observe (Functional)

  • After import: does the 0-rep set appear in the ExerciseHistoryMatrix?
  • Does the analytics engine exclude it from e1RM calculations? (It should, per the reps >= minReps filter.)
  • What does the next session pre-fill as target weight for this exercise? Does it use the 0-rep set's weight or the last valid set?
  • Does any coaching message reference the 0-rep session?

  • When you attempted 0-rep via the UI, was the rejection alert clear? Did it explain what to do instead, or just block?
  • If the 0-rep set appears in the history matrix, is it visually distinct from valid sets? Red/warning styling, or indistinguishable from a real set?

SC-08
The Drop Set Experience — Does the Coaching Know What Happened?
P4 Optimizer · State: STATE-5

Setup

Start a workout. Pick any dumbbell exercise. Log set 1 with a drop set: main weight = 20 kg × 8 reps, drop weight = 14 kg × 6 reps. Mark both as completed.

Run + Observe (Functional)

  • Does the narrative reference the drop specifically (not just the main set)?
  • Is the coaching path for "droppedCompleted" distinct from a normal completed set?
  • Does the coaching for the second drop set differ from the first?
  • Is the e1RM computed from the main set weight, not the drop weight?

Observe (UI + Tone)

  • Is the drop set entry panel visually discoverable? P4 would look for it; P1 would never find it unless it's surfaced proactively.
  • Does the drop set row feel visually subordinate to the main set row? The hierarchy should communicate that the drop is addendum, not primary.
  • When the drop is completed, does the completion animation feel different from main set completion? It should feel like a bonus, not an obligation.
SC-09
The Double Session — How Does the App Handle Two Workouts in One Day?
P4 Optimizer P2 Skeptic · State: STATE-5

Setup

Log a complete morning session. Start a second session in the same calendar day using different exercises.

Run + Observe (Functional)

  • Are both sessions separately visible in Progress history?
  • Does the streak count 1 day regardless of session count?
  • Does weekly volume include both sessions?
  • Does the readiness score for session 2 reflect intra-day fatigue?

Observe (UI + Tone)

  • In the Progress history list, if two sessions share the same date, is the visual grouping clear? Do they appear as two entries or one? Is the date shown once or twice?
  • Does the Home screen after both sessions feel "done for the day" or still open-ended?
SC-10
Session 1 vs Session 18 — Is the Experience Qualitatively Richer?
P4 Optimizer · State: STATE-0 then STATE-LIVE

Setup

First in STATE-0: screenshot every analytics view (Home, Progress, Insights). Then restore STATE-LIVE: screenshot the same views. Compare side by side.

Run + Observe (Functional)

  • Count charts, numbers, and insights available at session 1 vs session 18.
  • Read the coaching note at session 1 and most recent coaching note. Measure the difference in specificity.
  • How many EmergentInsights are populated vs. placeholders at session 18?
  • What is still a placeholder at session 18 that you would expect to be populated by now?

Observe (UI + Tone)

  • Does the session 18 Home screen feel visually richer, or just more populated? Density and richness are not the same thing.
  • At session 18, does any section of the Progress tab feel overcrowded — too many charts competing for attention?
  • Has the coaching language become more personal — more "you" and less "users generally"?
  • Would you describe the session 18 experience as impressive or overwhelming? There is a specific point where analytical depth becomes cognitive load. Find that point.
SC-11
The Drill-Down Reward — Does Going Deeper Tell You More?
P2 Skeptic P4 Optimizer · State: STATE-LIVE

Setup

STATE-LIVE. Start from the top level of any analytics section. Drill down as deep as the app allows. Repeat for: MomentumChipsCard, OverloadScoreboard, Plateau Risk, ExerciseInsights row → full detail view.

Run + Observe (Functional)

  • At each drill-down: what is the first piece of information not visible at the parent level?
  • Which drill-downs reward curiosity with genuinely new data?
  • Which drill-downs show the same card in a larger sheet?
  • At the deepest level (ExerciseInsightDetailView): how many of the 6 charts contain substantive data vs placeholder text?

Observe (UI + Tone)

  • Is it visually obvious that each card is tappable? If there's no chevron, no affordance, no visual cue — users won't discover the drill-down. P4 will hunt for it; P1 and P3 will never find it.
  • Does the transition between levels feel like moving deeper, or like opening a random sheet? Navigation should have direction.
  • At the detail level, is the information hierarchy clear? The most important insight should be at the top without scrolling.
SC-12
The Empty State Audit — What Does the App Say When It Has Nothing to Say?
P3 Beginner P1 Returner · State: STATE-0

Setup

STATE-0. Navigate every tab, every section, every card. Find every empty state in the app. Photograph or screenshot each one.

Run + Observe (Functional)

  • What % of empty states give you a specific action to take?
  • What % say "not enough data" without explanation?
  • Is there any feature that shows a chart or number with no data, rather than an honest empty state?
  • After 0 sessions, does any feature appear fully populated? That's a red flag (likely seed data showing through).

Observe (UI + Tone)

  • Do the empty states look intentional or broken? An empty state with good illustration and clear direction feels like a feature. A blank card feels like an error.
  • Is the emotional tone of empty states inviting or deflating? "Not enough data" is deflating. "Start your first session to unlock this" is inviting.
  • Is there visual consistency across empty states, or do they look like each one was designed separately?
SC-13
The Tired Session — Does the App Know What You're Feeling? Corrected Rev 2
P1 Returner · State: STATE-5

Setup

STATE-5. Log a session where you select feel=Tired at the end. Log weights that are slightly below your typical performance (5–10% lower reps or weight vs prior session).

Run + Observe (Functional)

  • Does any coaching text acknowledge that you felt tired?
  • Does the coaching contradict the feel rating — e.g., "strong session" language when you marked Tired?
  • Does the EmergentInsight section surface anything related to the Tired feel?
  • In a subsequent session, does the coaching reference yesterday's Tired rating? (Note: the readinessBefore pathway is active — verify whether the previous feel is carried forward into the next session's readiness context.)

Observe (UI + Tone)

  • Does the feel selector sheet feel like a meaningful check-in, or an afterthought appended to the session end? Its position and visual weight communicate its importance.
  • Is the coaching after a Tired session gentler in tone than after a Strong session? Tone should adapt to emotional state, not just data state.
  • Does any visual element on Home after a Tired session communicate "recovery mode"? Or does the UI look identical regardless of how you felt?
SC-14
The Closing Moment — Does the App Plant a Hook for the Next Session?
P1 Returner P4 Optimizer · State: Any

Setup

Run any complete session. Observe everything from "Finish" button through the last screen before you leave the app. This moment determines whether users return.

Run + Observe (Functional)

  • Note the exact sequence: feel prompt → recap → coaching → what else?
  • Does the closing sequence reference the next session? (next target weight, next exercise, next goal)
  • Would you remember anything from the closing if you came back 3 days later?
  • Does anything tease a feature you haven't explored yet?

Observe (UI + Tone)

  • Does the session end feel like a conclusion or a cutoff? A good closing should feel like the end of a chapter, not an abrupt stop.
  • Is the recap visually satisfying? Numbers, colors, and hierarchy should make the session feel real and measurable.
  • Is the last coaching message forward-looking? "You hit 3 sets of 10 at 15 kg" is backward-looking. "At this rate, you'll reach 20 kg in 3 sessions" is forward-looking. Which is it?
  • Does the app give you a reason to come back tomorrow that's specific to your data — or a generic "keep it up" that could have been sent to anyone?

Biases and Traps — What Will Corrupt Your Simulation

Simulation run by the product's creator has a structural validity problem. These are the failure modes. Name them before you start.

The Rationalization Trap
You hesitate at a screen. Before you can write it down, your brain generates an excuse: "users will learn this," "that's fine once you know where to look." You believe the excuse and don't log the hesitation.
Any hesitation is a finding regardless of the rationalization. Write it down first, explain it later. The rationalization is the bias. The hesitation is the data.
The Feature Familiarity Effect
You navigate without thinking because you built the navigation. You know the drop set button is in the set row's trailing actions. A real user doesn't. Your 0-second navigation time is not the user's navigation time.
Before each major navigation action: pause and state aloud where you expect the feature to be and how you'd find it as a first-time user. If you're right without thinking, you have the familiarity effect. Force yourself: find [feature X] in under 30 seconds as if you'd never seen the app.
The Aesthetic Blindspot New in Rev 2
You chose the colors. You chose the fonts. You've looked at this interface for hundreds of hours. You've stopped seeing it. The amber that felt precise and premium on day 1 may have crept into places it doesn't belong. The text size that looked comfortable on your device may be too small on a different one. You cannot see these things without forcing yourself to look.
For Protocol 8, use a device you don't normally test on. Run the first-impression test with a physical distance from the screen (30–40 cm, not the typical 15 cm dev distance). Take a screenshot and view it at 50% zoom — small-scale visual hierarchy problems reveal themselves at thumbnail size that are invisible at full size.
The Feature Pride Effect New in Rev 2
You spent weeks on the PSI chart smoothing, the EmergentInsightEngine, the fatigue-adjusted e1RM. Features you worked hardest on get more favorable evaluation than they deserve. You read the coaching message for INOL and think "that's useful" — because you know what INOL means. P3 reads it and thinks nothing at all, because the term means nothing to them yet.
For features you're proud of, deliberately run them through P3's lens last. After you've already evaluated as P2 and P4, come back as P3 and ask: "If I had no idea what this number is, would I care?" The answer often exposes unnecessary complexity in the surface layer of otherwise deep features.
Emotional Neutrality
You test when you're calm, focused, and attentive. Real users open the app after a long day, when they're tired, when they have 20 minutes before the gym closes, when they're coming back after two weeks away with low-grade guilt. Their emotional state affects their perception of friction.
Run at least two simulation sessions in genuinely suboptimal conditions: physically tired, time-constrained, or emotionally distracted. These are the conditions under which the UX either holds or fails.
The "It Works" Definition of Success
You complete a task and call it a pass. "I was able to log the session" is not a UX pass. The question is whether you wanted to do it again, whether it felt like it was working with you or making you work for it, whether the coaching after made you feel understood. Functional is table stakes.
After every scenario, rate two dimensions separately: task completion (0–100%) and experience quality (0–100%). A feature can be 100% on task and 40% on experience. Both scores matter. Only reporting task completion is a form of self-deception.
Positive Recency Bias
Whatever the last screen you saw was good, you'll rate the whole session higher. A satisfying post-session summary inflates your evaluation of the entire experience. The friction that happened 10 minutes earlier gets discounted.
Capture observations in real-time, not from memory. The end-of-session reflection should summarize notes you already took — not reconstruct the experience. If you didn't take notes during, the session produced no findings.

Finding Classification

Simulation findings are a different type from audit findings. Classify them accordingly when logging.

TypeDefinitionExampleSeverity signal
F-FrictionA task that was possible but harder than it needed to be. More taps, more thought, or more time than the task's importance warranted."Finding the drop set input required knowing it existed."P3 can't find it = High. P4 had to look = Low.
F-MismatchThe outcome of an action was different from what the persona expected before taking the action."I expected the coaching to reference my tiredness. It talked about progressive overload."Trust breach. Any mismatch between emotional context and coaching register is high severity.
F-VoidThe persona wanted something the app simply didn't provide. Not a broken feature — an absent one."After the session, I wanted to know what to do next time. Nothing told me."Depends on how central the missing thing is to the product promise.
F-Aesthetic NewA visual design problem: hierarchy unclear, color misused, density too high or too low, animation adds noise, layout doesn't guide the eye."The amber accent color appears on both the active-set indicator and an error state — I can't tell what amber means."If it corrupts information meaning (e.g., color semantics), high severity. If it's just less beautiful, low severity.
F-Tone NewA coaching language failure: message is generic, contradicts the physical outcome, uses jargon without explanation, or sets the wrong emotional register for the context."The session end message said 'Great work!' on a session where I missed every target."Generic = medium severity. Contradictory = high. Tone-deaf on an emotionally loaded moment (return after gap) = critical.
F-DelightThe app provided something the persona didn't expect that improved the experience. Document these — they're the features to amplify and make more visible."The coaching mentioned the exact weight from last session. That's specific. I noticed."Not a severity — a signal to amplify. Note what made it delightful so you can replicate the pattern.
F-AbandonThe persona would have stopped using the app because of this experience. The highest severity finding in simulation."As P3, I entered the wrong weight format and got analytics that looked wrong. I'd distrust everything after that."Always critical. Any F-Abandon is a P0 regardless of how it was produced.

Simulation generates hypotheses; the audit verifies them with code and data.
