Section 1 — Must-Pass Analytics Criteria
📡 1. Event Taxonomy 8 tests · AN-001–AN-008 ▼
AN-001
Core Workout Events
Three events must fire correctly: session_started, set_logged (with exercise_id, weight, reps, completed), and session_ended (with duration, exercise_count, set_count). These are the foundation of all session analytics. The exercise_id must be a UUID that matches the exercise database and remains stable across relaunches.
Pass: All three events fire correctly in TestFlight. exercise_id is UUID-stable and matches the static exercise database. session_ended fires at most once per session regardless of how the session is terminated.
AN-002
Onboarding Funnel Events
Three events must fire in sequence on fresh install: onboarding_started, profile_saved (with goal, equipment, fitness_level), onboarding_completed. Drop-off between steps must be identifiable from the event sequence. If a user quits mid-onboarding and returns, only one onboarding_started should exist per install.
Pass: All three events fire in sequence for a fresh install. Drop-off point identifiable from event sequence. No duplicate onboarding_started events for a single install.
AN-003
Feature Discovery Events
Three event types cover feature discovery: tab_viewed (with tab_name, session_index), feature_first_used (with feature_name), drill_down_opened (with source_screen, destination). Every tab and every drill-down sheet must fire a view event. feature_first_used must fire exactly once per feature per install — not per session.
Pass: Every tab and every drill-down sheet fires a view event. feature_first_used fires exactly once per feature per install. No duplicate discovery events on re-use.
AN-004
Coaching Engagement Events
Every coaching note interaction requires instrumentation: coaching_note_shown (with note_type, exercise_id), coaching_note_dismissed, and coaching_note_expanded (if the note supports expansion). The distinction between dismiss and expand is the core coaching engagement signal — it cannot be inferred after the fact.
Pass: Every coaching note shown fires coaching_note_shown. Dismiss vs. expand is distinguishable from distinct event names. coaching_note_expanded fires only when user deliberately expands, not on show.
AN-005
Trainer Tab Events
The Trainer tab funnel requires four events: plan_generated (with readiness_score, exercise_count), plan_started (user began the generated workout), plan_completed, plan_dismissed. A user who generates a plan and never starts it is a critical signal. A user who generates, starts, and abandons mid-workout is a different signal. Both must be distinguishable.
Pass: The funnel from plan_generated → plan_started → plan_completed is fully traceable. Dismissed plans are distinguishable from started-but-abandoned. No plan generates two plan_generated events.
AN-006
Progress Tab Engagement Events
Three events instrument Progress tab depth: progress_tab_viewed (with data_age_days — how old is the newest session), section_expanded (with section_name matching the 5 question categories), drill_down_opened. Each of the 5 question categories needs its own distinct section_name value so engagement per category is measurable independently.
Pass: Each of the 5 question categories has its own section_name value. Engagement per category is independently measurable. data_age_days correctly reflects days since the most recent logged session, not since install.
AN-007
Cardio Session Events
Cardio instrumentation requires three events: cardio_session_started (with circuit_type: AMRAP or EMOM), round_completed, cardio_session_ended (with rounds_completed, duration). AMRAP and EMOM sessions must be distinguishable so completion rates can be calculated by circuit type. A cardio session started must never be confused with a strength session started.
Pass: AMRAP and EMOM sessions are distinguishable from circuit_type alone. Completion rate calculable from cardio_session_started vs. cardio_session_ended counts. Cardio events never conflated with strength session_started events.
AN-008
Error and Recovery Events
Validation errors must be fully instrumented: input_validation_error (with field_name, error_type), error_recovered (user corrected and proceeded), error_abandoned (user left without correcting). Every validation error type — weight out of range, reps out of range, missing required field — must fire its own error event with the correct field name.
Pass: Recovery vs. abandonment distinguishable for every validation error type. field_name is populated with the actual field (e.g., weight_kg, reps) not a generic value. Each error type has a distinct error_type value.
↓ 2. Funnel Definitions 8 tests · AN-009–AN-016 ▼
AN-009
Onboarding Funnel
Five-step funnel: install → onboarding_started → profile_saved → onboarding_completed → first_workout_started. This is the most critical funnel in the product. Drop-off at each step must be independently measurable. Target completion rate is greater than 85% through onboarding_completed.
Pass: Each step identifiable from events. Drop-off at each step measurable. Target: >85% of users who start onboarding reach onboarding_completed. Time between each step calculable.
AN-010
First Workout Funnel
Three-step funnel: session_started → first_set_logged → session_ended. Average time-to-first-set measures onboarding-to-action velocity. Completion rate distinguishes users who start a workout and finish from those who start and abandon. Both signals inform early retention.
Pass: Completion rate measurable from session_started vs. session_ended counts. Average time-to-first-set calculable from event timestamps. Abandoned sessions (started but no session_ended) identifiable.
AN-011
Trainer Adoption Funnel
Five-step funnel: app_opened → trainer_tab_viewed → plan_generated → plan_started → plan_completed. Conversion rate at each step must be calculable within 48 hours of launch. Users who generate a plan but never start it are a key re-engagement target. Users who start but don't complete are a plan-quality signal.
Pass: Each step measurable. Conversion rate at each step calculable within 48 hours of TestFlight launch. Plan-generated-but-never-started users identifiable as a segment.
AN-012
Feature Depth Funnel
Four-step depth funnel for the Progress tab: tab_viewed → section_expanded → drill_down_opened → drill_down_closed. For each of the 5 Progress question categories, the depth-of-engagement funnel must be independently traceable. Category with the lowest drill-down rate is a design signal.
Pass: For each of the 5 Progress categories, the depth funnel is traceable independently. Category with lowest drill-down rate identifiable. Time spent in drill-down calculable from open/close timestamps.
AN-013
Retention Cohort Funnel
D1/D7/D30/D90 retention must be measurable from session events alone. A user is retained on Day N if they log a session on or within ±1 day of the Nth day after install. A user who logs a session on install day and then 29 days later is counted as retained at D30. Cohort must be groupable by install week.
Pass: D1, D7, D30, D90 retention calculable from events. A user logging sessions on day 0 and day 29 counts as D30-retained. Cohort groupable by install week for cohort analysis.
AN-014
Cardio Adoption Funnel
Three-step cardio funnel: cardio_tab_viewed → cardio_session_started → cardio_session_ended. Cardio adoption must be measurable separately from strength adoption. Dual-modality users (use both cardio and strength features) must be identifiable from event data — they are the highest-value retention segment.
Pass: Cardio adoption separate from strength adoption. Dual-modality users identifiable from having both cardio_session_started and strength session_started events. Cardio completion rate calculable.
AN-015
Comeback Funnel
Three-step comeback funnel: gap of ≥7 days detected → comeback message shown → session_started within 24 hours. The correlation between comeback message and same-day session is the core signal for messaging effectiveness. A user who opens the app after a 7-day gap and does not log a session is a pre-churn warning.
Pass: Correlation between comeback message and same-day session is measurable. Message-to-session rate calculable per comeback message variant. Opens-without-sessions after gap identifiable.
AN-016
Churn Signal Funnel
Three-signal churn model: last_session_date → no activity for 14 days → app_opened_no_session (opened but didn't log). Churn is defined as 30+ days without a logged session. The pre-churn warning window is days 12–29. Users in this window who open but don't log are the highest-priority re-engagement target.
Pass: Pre-churn warning signal identifiable from events. Churn defined as 30+ days no session and consistently measured. app_opened_no_session distinguishable from session_started. Opens-without-sessions countable per user.
■ 3. Dashboard Definitions 7 tests · AN-017–AN-023 ▼
AN-017
North Star Dashboard
One primary metric: Weekly Active Loggers (users who log ≥1 session in a rolling 7-day window). This metric must be visible in App Store Connect or an equivalent analytics tool within 48 hours of any session event. It must update at least daily. All other metrics are secondary to this one number.
Pass: Weekly Active Loggers metric defined, measurable, and updated at least daily. Visible in App Store Connect within 48 hours of any session. Historical trend visible (not just current value).
AN-018
Retention Dashboard
D1/D7/D30/D90 retention by install cohort, breakable down by onboarding goal (strength/consistency/body composition). This dashboard answers whether different user goals produce different retention curves — a core product hypothesis. Cohort filtering by goal must work without requiring a database join.
Pass: Retention curves visible for each install cohort. Cohort filtering by onboarding goal works. D1/D7/D30/D90 breakpoints all present. Week-over-week trend visible for each retention milestone.
AN-019
Feature Adoption Dashboard
Percentage of MAU who used each major feature at least once in the last 30 days. Features tracked: Trainer tab, Progress drill-downs, RPE logging, drop sets, cardio sessions. Adoption percentage must be calculable for each feature independently. Week-over-week trend visible to detect adoption growth or stall.
Pass: Adoption % calculable for each of the 5 named features. Trend visible week-over-week. A feature that didn't exist in a prior period does not create a false decline. All 5 features have named, stable feature_name values in events.
AN-020
Coaching Engagement Dashboard
Two event counts: coaching_note_shown vs. coaching_note_expanded. Ratio = coaching engagement rate. This must be calculable per coaching note type: post-set, readiness, trainer, summary. The lowest-engagement note type is a product signal — either the content is weak or the placement is wrong.
Pass: Engagement rate calculable per coaching note type. Four note types (post-set, readiness, trainer, summary) each have distinct note_type values. Trend visible over time. Lowest-engagement type identifiable without manual query.
AN-021
Funnel Health Dashboard
Each of the 8 defined funnels (AN-009–AN-016) must have a live conversion rate visible in a single view. An alert must be configured for any funnel whose conversion rate drops more than 15% week-over-week. This dashboard is the first thing the analytics lead checks each Monday morning.
Pass: All 8 funnels have current conversion rates visible. Week-over-week delta visible for each funnel. Alert configured for >15% week-over-week drop in any funnel. Dashboard updates within 24 hours of any event.
AN-022
Error Rate Dashboard
input_validation_error count, error_recovered rate, and error_abandoned rate by field name and error type. The top 3 error types by abandonment rate must be identifiable within 48 hours of launch. High abandonment on a specific field is a UX failure requiring immediate attention.
Pass: Top 3 error types by abandonment rate identifiable within 48 hours of launch. Error rates breakable by field_name and error_type. Recovery rate vs. abandonment rate visible as a ratio per error type.
AN-023
Session Quality Dashboard
Three session quality metrics: median session duration, median set count per session, median exercises per session. Breakdowns available by cohort (onboarding goal, fitness level, equipment). These metrics answer whether users are having substantive training sessions or skimming the surface of the app.
Pass: All three metrics calculable from session_started/session_ended events. Breakdown by onboarding goal, fitness level, and equipment available. Median (not mean) used to avoid distortion by outliers. Week-over-week trend visible.
⋮ 4. A/B Test Design 6 tests · AN-024–AN-029 ▼
AN-024
Test Hypothesis
Every A/B test must have a written hypothesis in the exact format: "If we [change], then [metric] will [direction] because [reason]." The metric named must be measurable from existing events — no new instrumentation required to evaluate the hypothesis. This prevents starting tests that cannot be concluded.
Pass: Hypothesis exists in the specified format before test launches. Metric named is measurable from existing event schema. No new instrumentation required to evaluate the test result. Hypothesis is filed in a doc accessible to the full team.
AN-025
Sample Size Calculation
Minimum sample size must be calculated before the test starts, based on expected effect size and 80% statistical power. The test must not be evaluated before reaching minimum sample size — early peeking at results is not permitted. Sample size calculation must be documented alongside the hypothesis.
Pass: Sample size documented before test launches. Calculation based on expected effect size and 80% power. Test not evaluated before minimum sample reached. Calculation methodology documented (e.g., which calculator or formula used).
AN-026
Single Variable Principle
Each test changes exactly one variable. The test spec must list all elements that remain unchanged to confirm isolation. No test simultaneously changes copy and layout, or metric and trigger. If a proposed test would require changing two variables, it must be split into two sequential tests.
Pass: Test spec lists all unchanged elements alongside the one changed variable. No test changes both copy and layout simultaneously. Test reviewer confirms single-variable isolation before launch approval.
AN-027
Guardrail Metrics
Every test must have at least one guardrail metric — a metric that would trigger early stopping if it drops below a defined threshold even if the primary metric improves. Example: session completion rate must not drop more than 10% in any variant. The threshold must be specified before the test starts, not after results are visible.
Pass: Guardrail metric defined and threshold specified before test launches. Guardrail monitored daily during test run. Early-stopping protocol documented: who is notified, who makes the call, how fast the test is stopped.
AN-028
Test Duration
Each test runs for at least 2 full weeks (to capture weekday vs. weekend behavior patterns) and at most 8 weeks. Start and end dates must be set before the test launches. No test is extended past 8 weeks without a documented reason — extended tests are a signal that the hypothesis was wrong or the metric is too noisy.
Pass: Start and end dates set before launch. Minimum duration 2 weeks. Maximum duration 8 weeks unless documented extension reason exists. Extension requests require team review, not unilateral decision.
AN-029
Rollout Decision Criteria
The decision rule must be written before the test launches: "If metric X improves by Y% with p<0.05, ship variant; if not, ship control." No post-hoc rationalization of results is permitted. If the result is inconclusive, ship control and document why the test failed to reach significance — the failure is data.
Pass: Decision rule written before launch. Rule specifies metric, improvement threshold, and significance level. Inconclusive results default to shipping control. Post-hoc metric switching explicitly prohibited in team norms.
● 5. Data Integrity 8 tests · AN-030–AN-037 ▼
AN-030
Duplicate Event Prevention
No event fires more than once per user action. This is most likely to fail for set_logged (double-tap on a submit button) and session_ended (background/foreground transitions). Duplicate events corrupt funnel conversion rates and session quality metrics irreversibly.
Pass: Log a single set; confirm exactly one set_logged event in the analytics stream. Terminate a session; confirm exactly one session_ended event. Test across double-tap, background/foreground, and force-quit scenarios.
AN-031
UUID Stability
Exercise IDs and session IDs must be stable across app launches and data migrations. An exercise logged on Day 1 must have the same UUID when queried on Day 30 after multiple app relaunches. UUID instability makes exercise-level analytics impossible and breaks the coaching narrative engine's historical lookup.
Pass: Exercise logged on Day 1 has the same UUID on Day 30 after multiple relaunches. UUID stable across app updates. UUID stable if user deletes and reinstalls (exercise UUIDs are compile-time constants, not generated at runtime).
AN-032
Timestamp Accuracy
All events must have device-local timestamps in UTC. No event timestamp should be in the future or more than 1 second in the past at time of firing. A critical edge case: changing the device timezone mid-session must not retroactively shift existing session timestamps or corrupt duration calculations.
Pass: Timestamps correct across timezone changes tested by changing device timezone mid-session. No future timestamps. UTC format consistent across all events. Duration fields calculated from UTC timestamps, not local time.
AN-033
Null Safety
No event fires with a null or empty required field. This includes exercise_id on set_logged, tab_name on tab_viewed, and note_type on coaching_note_shown. Null fields in analytics events cannot be fixed retroactively — they are permanent gaps in the data.
Pass: All required event properties have non-null, non-empty values in 100% of fired events. Static analysis or runtime assertions confirm required fields are non-optional in the event schema. Zero null events in 1000 consecutive test events.
AN-034
Backfill Accuracy
For a fully local app, this applies to analytics export scenarios: if a session is logged and the analytics event is queued locally before export, the event timestamp must reflect when the session happened, not when the export occurred. This matters for retention calculations — an event exported 3 days after the session should not appear as a D3 session.
Pass: Event timestamp reflects time of occurrence, not time of export. If analytics are exported in batch, timestamps are not normalized to export time. Offline sessions (if tested) preserve original timestamps when later synced.
AN-035
Schema Versioning
The event schema must have a version field. Breaking changes to event structure — renaming a field, removing a field, changing a field's type — must increment the schema version. Analytics consumers must be able to filter by schema version so historical data remains analyzable after a schema change.
Pass: Every event includes a schema_version field. Breaking changes increment the version. Historical data queryable with either old or new schema version. No breaking change silently invalidates historical event data.
AN-036
Sampling Strategy
If event volume requires sampling, the sampling rate must be documented and consistent. Most importantly: the same user must always fall in the same sample bucket. Non-deterministic sampling by user_id corrupts cohort analysis — a user who appears in 70% of samples looks like a lighter user than they actually are.
Pass: If sampling is used, same user always in the same sample bucket (deterministic sampling by user_id or equivalent stable identifier). Sampling rate documented and visible in the analytics dashboard. Sampled metrics clearly labeled as sampled.
AN-037
Data Completeness Check
For every 100 session_started events, at least 80 session_ended events are expected — accounting for crashes, force-quits, and battery deaths. A completion rate below 80% in TestFlight signals that sessions are being lost, not just abandoned, which corrupts duration and quality metrics.
Pass: Session completion rate ≥80% in TestFlight. Alert configured if rate drops below 70%. Incomplete sessions (started but no ended) are distinguishable from abandoned sessions in the analytics stream.
■ 6. Privacy & Instrumentation Ethics 7 tests · AN-038–AN-044 ▼
AN-038
No PII in Events
No event may contain a user's name, email, phone number, or any directly identifying information. This includes indirect identifiers: a combination of fitness level + equipment + goal that uniquely identifies a person in a small cohort. The test requires a non-engineer to review the full event schema and confirm zero PII fields.
Pass: Event schema reviewed by a non-engineer who confirms zero PII fields. Review is documented with the reviewer's name and date. Schema review is repeated after any new event or field is added.
AN-039
Aggregate-Only Off-Device
Any data that leaves the device must be aggregate — not individual session records. H.O.N. is a local-first app. The App Store privacy label must reflect this. If any analytics pipeline sends individual events off-device (e.g., App Store Connect metrics, TestFlight crash reports), explicit user consent must be documented.
Pass: No individual session data transmitted without explicit user consent. Any off-device data pipeline is aggregate-only or clearly disclosed. Privacy label accurately reflects all off-device data flows.
AN-040
HealthKit Data Never in Analytics
HealthKit-derived data — HRV, heart rate, sleep, step count, active energy — must never appear in any analytics event. This is both a privacy requirement and an App Store Review compliance requirement. The event schema must contain no HealthKit field names, confirmed by schema review.
Pass: Event schema contains no HealthKit field names (HRV, heart_rate, sleep_hours, step_count, active_energy, etc.). Confirmed by schema review. HealthKit data stays on-device and is never exported, logged, or transmitted in any analytics pipeline.
AN-041
Opt-Out Respected
If a user opts out of analytics, zero events must fire from that point forward — not reduced events, not anonymized events. Zero. The opt-out toggle must be accessible in Settings. After toggling off, the next app session must produce no events in any analytics stream. This must be testable by the QA team without access to backend infrastructure.
Pass: Toggle analytics off; verify zero events in stream for subsequent sessions. Opt-out persists across app restarts. Opt-out respected immediately — not queued events sent before the toggle takes effect.
AN-042
Retention Policy Documented
Analytics data must have a defined retention period — recommended 2 years for a product at this stage. The policy must be written, reviewed by the team, and communicated to users in a privacy policy or equivalent disclosure. Data beyond the retention period must be automatically deleted, not manually managed.
Pass: Retention policy written and communicated to team before launch. Retention period defined (e.g., 2 years). Automatic deletion mechanism exists for data past retention period. Policy accessible to users in privacy policy.
AN-043
App Store Privacy Label Accurate
The Privacy Nutrition Label in App Store Connect must accurately reflect all data collected. Every event property type must map to a disclosed data category. If fitness_level is collected, the label must disclose "Health & Fitness" data. If device crash data is collected, the label must disclose "Crash Data." The label must be updated whenever new events are added.
Pass: Every event property type maps to a disclosed data category in the privacy label. Label reviewed and updated before each App Store submission. Discrepancy between label and actual collection is a P0 issue.
AN-044
Minimal Collection Principle
No event may collect more fields than are required to answer the business question it was designed to answer. Each event must have a documented "question this answers." No field exists without a question it answers. Any field that cannot be linked to a specific business question must be removed before launch.
Pass: Each event has a documented "question this answers." No field exists without a linked question. Schema reviewed for orphan fields before launch. Any new field added post-launch requires a documented business question before it ships.

Section 3 — Definition of Done for Analytics

Analytics instrumentation is done when we can answer our five most important product questions — is the onboarding funnel healthy, are users coming back, which features are being used, is the coaching resonating, and where are users failing — without writing any new code. Done means all 44 criteria in this document are green, the funnel health dashboard is live and alerting, the privacy label is accurate, and a teammate who joined the team yesterday can pull a retention curve from App Store Connect without needing to ask how. Done does not mean we have instrumented everything imaginable — it means we have instrumented exactly what we need to make product decisions with confidence, nothing more. Every field in our event schema has a question it answers. Every event that fires has a dashboard that reads it. Every dashboard has an owner who checks it on Mondays. Done also means we have drawn a hard line: no HealthKit data in events, no PII in any field, no individual records leaving the device, and opt-out produces zero events. Instrumentation that violates user trust is not instrumentation — it is a liability. We ship when the data is honest, complete, and useful. Not before.
