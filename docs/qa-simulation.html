QSIM-01 Smoke run: cold launch gate
Smoke
Precondition
Fresh simulator with no prior app data. App not installed or data container wiped via Simulator > Device > Erase All Content.
Action Steps
  1. Install and launch the app on a clean simulator.
  2. Observe the initial render time from launch to first visible HomeView content.
  3. Tap each of the 5 tabs (Home, Workout, Progress, Insights, Trainer) in sequence.
  4. Return to HomeView and confirm no crash or blank screen.
Pass / Fail Criteria
Pass
  • HomeView renders within 3 seconds of launch.
  • No crash or unhandled exception at any point.
  • All 5 tabs respond to tap and render without blank state.
  • SeedStore.isLoaded transitions to true before first render.
Fail indicators
  • App crashes on launch or within first 10 seconds.
  • HomeView remains blank for more than 3 seconds.
  • Any tab tap produces a blank or spinner-only screen.
  • Console shows unhandled fatal errors on first launch.
QSIM-02 Unit: e1RM formula boundary check
Unit
Precondition
Unit test target available. SetRecord.e1RM(weight:reps:) callable in isolation or via a logged workout session with specific rep counts.
Action Steps
  1. Log one set at 1 rep (e.g. 100 kg × 1).
  2. Log one set at 10 reps (e.g. 100 kg × 10).
  3. Log one set at 11 reps (e.g. 100 kg × 11).
  4. Log one set at 15 reps (e.g. 100 kg × 15).
  5. Log one set at 16 reps (e.g. 100 kg × 16).
  6. Inspect the e1RM value surfaced per set in StrengthLabView or via unit test assertion.
Pass / Fail Criteria
Pass
  • 1 rep: e1RM equals the logged weight exactly (no formula applied).
  • 2–10 reps: Epley formula used — w × (1 + r/30).
  • 11–15 reps: Mayhew formula used.
  • 16+ reps: e1RM returns 0 (excluded from analytics).
Fail indicators
  • Epley applied to 11-rep set (off-by-one boundary error).
  • 16-rep set contributes a nonzero e1RM to analytics.
  • 1-rep set inflated by any formula multiplier.
  • Any formula mismatch at the exact boundary values 10 and 11.
QSIM-03 Functional: drop set narrative path
Functional
Precondition
Active workout session open with at least one exercise selected. One normal set already logged for that exercise.
Action Steps
  1. On the active workout screen, enable the drop set toggle for the current set.
  2. Enter a lower weight and target reps for the drop set.
  3. Complete the drop set (mark isDropCompleted = true).
  4. Observe the coaching narrative text that appears after the set is completed.
  5. Repeat the drop set completion 3–4 more times to verify phrase rotation.
Pass / Fail Criteria
Pass
  • Narrative uses a phrase from the "Drop Set — Completed" bank slot (IDs 76–85).
  • The weight phrase slot is replaced by the drop-set modifier, not a standard weight phrase.
  • Subsequent completions rotate through IDs 76–85 without repeating consecutively.
  • No crash or empty narrative string returned.
Fail indicators
  • Narrative shows a generic weight phrase instead of a drop set phrase.
  • Empty or placeholder text rendered after drop set completion.
  • Same drop set phrase repeated on two consecutive completions.
  • App crashes when isDropCompleted is set to true.
QSIM-04 Integration: refreshAnalytics() token race
Integration
Precondition
Active workout session with at least one exercise. Instrument the app or use console logging to observe which analytics token is committed. Optionally add a print statement in the background block's commit path.
Action Steps
  1. Complete 5 sets in rapid succession (within ~2 seconds) to trigger refreshAnalytics() five times.
  2. Monitor the Xcode console for token commit messages.
  3. After all completions settle, inspect homeCache values in HomeView.
  4. Verify the UI reflects only the final state, not any intermediate state.
Pass / Fail Criteria
Pass
  • Only the last-minted token's result is committed to homeCache.
  • HomeView updates once with final state; no mid-flight flicker.
  • Console shows at most one commit log entry (or multiple discarded entries with one final commit).
  • No race condition visible in UI (e.g., readiness score jumping between values).
Fail indicators
  • HomeView flickers through multiple intermediate readiness states.
  • Console shows a stale token result applied after the latest token's result.
  • Readiness score or today hints reflect a mid-sequence set count, not the final tally.
  • Any data corruption or crash during rapid set completion.
QSIM-05 Regression: ConsistencyDotGrid ordering
Regression NAV-004
Precondition
At least 30 days of workout log history. Workouts must span multiple calendar weeks so the grid is non-trivially populated. Use the scripted seed data or manually log workouts across 30+ days.
Action Steps
  1. Open the Progress tab.
  2. Scroll to the ConsistencyDotGrid component.
  3. Identify the leftmost/topmost dot and its associated date.
  4. Identify the rightmost/bottommost dot and its associated date.
  5. Confirm the temporal direction matches oldest-to-newest, left-to-right (or top-to-bottom).
Pass / Fail Criteria
Pass
  • Oldest workout date appears at the grid origin (top-left).
  • Newest workout date appears at the grid end (bottom-right or rightmost column).
  • Chronological order is consistent across all visible rows/columns.
  • NAV-004 regression: no recurrence of reverse-ordering bug.
Fail indicators
  • Newest date appears first (NAV-004 regression).
  • Dates are in random/non-chronological order.
  • Grid renders with all dots the same shade (no data distinction).
  • Crash when displaying 30+ day history in the grid.
QSIM-06 Functional: equipment chip swap
Functional
Precondition
Active workout containing barbell bench press (or any compound barbell exercise that has a dumbbell variant in the exercise database). A prior log entry for the dumbbell variant exists so bestVariant has history to match against.
Action Steps
  1. In ActiveWorkoutView, locate the EquipmentChipRow for the barbell exercise.
  2. Confirm "Barbell" chip is shown as selected.
  3. Tap the "Dumbbell" chip.
  4. Confirm the exercise swaps to the best matching dumbbell variant via bestVariant(equipment:matching:).
  5. Inspect the weight field — it should reflect per-hand weight, and any displayed total should be ×2.
  6. Swap back to barbell and confirm reverse swap works.
Pass / Fail Criteria
Pass
  • Exercise name updates to the dumbbell variant immediately on tap.
  • Equipment.effectiveWeight(_:) doubles the per-hand weight for bilateral display.
  • quickSwapEquipment returns available chip options without O(n_log) cost on every render.
  • Reverse swap back to barbell restores the original exercise or best barbell match.
Fail indicators
  • Exercise name does not change after chip tap.
  • Weight displayed is not doubled for dumbbell (per-hand) selection.
  • App freezes or janks during chip row render (O(n_log) called per render).
  • Crash when no matching dumbbell variant exists in the exercise database.

System

Fresh install, no prior data. Simulator or device with no existing app data container.

Fail indicators

  • Workout log entry missing from Progress tab after session completion.
  • HomeView readiness still shows empty/zero state after first workout.
  • userProfile not persisted — prompts onboarding again on next launch.
QSIM-08 Smoke: 500-entry log performance
Smoke
Precondition
SeedStore seeded with 500 WorkoutLogEntry records spanning ~2 years. Use the scripted seed utility or inject mock data. Confirm record count before launching.
Action Steps
  1. Launch the app with the 500-entry seed.
  2. Measure time from launch to SeedStore.isLoaded = true (via console or Instruments).
  3. Navigate to all 5 tabs in order: Home, Workout, Progress, Insights, Trainer.
  4. On Insights/StrengthLabView, scroll through all exercises.
  5. Trigger a refreshAnalytics() by logging a new set; measure completion time.
Pass / Fail Criteria
Pass
  • SeedStore.isLoaded within 5 seconds of launch.
  • All 5 tabs load without dropped frames (60 fps maintained).
  • refreshAnalytics() completes and commits within 200 ms.
  • No memory warnings in Instruments during navigation.
Fail indicators
  • Load time exceeds 5 seconds on a modern simulator.
  • Visible jank (dropped frames) during tab navigation or list scroll.
  • refreshAnalytics() takes more than 200 ms for the 500-entry dataset.
  • Memory warning or OOM crash during performance run.
QSIM-09 UAT: U-02 new user HomeView comprehension
UAT
Precondition
Cohort U-02 tester (new user, first week of use). App seeded with 3–5 days of beginner workout data so HomeView has populated readiness, today's hints, and a progress trend. Do not brief the tester on HomeView layout.
Action Steps
  1. Hand the device to the U-02 tester with HomeView visible.
  2. Say: "Tell me what you see on this screen. What does this app think you should do today?"
  3. Observe and time how long until the tester: (a) identifies the readiness score, (b) identifies today's exercise hints, (c) navigates to start a workout.
  4. Record any confusion points or vocabulary that trips the tester up.
  5. Ask the tester to rate clarity on a 1–5 scale after the session.
Pass / Fail Criteria
Pass
  • Tester correctly identifies the readiness score and its meaning within 15 seconds.
  • Tester reads today's exercise hints without confusion about units or terminology.
  • Tester navigates to start a workout within 30 seconds without coaching prompts.
  • Tester rates clarity 4 or 5 out of 5.
Fail indicators
  • Tester cannot locate or interpret the readiness score without guidance.
  • Tester confused by fitness terminology in coaching note or today's hints.
  • Tester requires more than 30 seconds or a coaching prompt to navigate to workout.
  • Clarity rating of 3 or below — escalate to design review.

QSIM-11 Integration: HealthKit permission denied
Integration
Precondition
HealthKit permissions denied for the app. Set via Settings > Privacy & Security > Health on device, or deny during the system permission prompt on simulator launch. App has existing workout log data so ReadinessEngine has input to process.
Action Steps
  1. Deny all HealthKit read/write permissions for the app.
  2. Launch the app (or relaunch after changing permissions).
  3. Navigate to HomeView and observe the readiness section.
  4. Wait for SeedStore.isLoaded to be true.
  5. Check that ReadinessEngine returns a valid ReadinessState using only app-internal log data.
Pass / Fail Criteria
Pass
  • ReadinessEngine.compute(log:) returns a valid ReadinessState with a score 0–100.
  • A coaching note is displayed — not an error message or empty string.
  • No infinite spinner; HomeView fully renders within normal load time.
  • No crash related to HKHealthStore access denial.
Fail indicators
  • App crashes on HealthKit permission denial (unguarded HKStore call).
  • HomeView shows an infinite spinner when HealthKit is unavailable.
  • Readiness score returns nil or is absent from the UI.
  • Error message surfaced to the user referring to HealthKit directly.

Functional

QSIM-13 System: app killed mid-workout recovery
System
Precondition
Active workout session in progress. Two exercises fully logged with all sets completed. Third exercise in progress with 1 set logged out of 3.
Action Steps
  1. Start a workout and log 2 complete exercises (3 sets each).
  2. Begin the 3rd exercise and log exactly 1 set.
  3. Force-kill the app via the app switcher (swipe up to dismiss).
  4. Relaunch the app.
  5. Observe the state of the Workout tab and the workout log.
Pass / Fail Criteria
Pass
  • Acceptable outcome A: active session restored with all 7 logged sets intact.
  • Acceptable outcome B: clean empty state — no partial workout shown, no corruption.
  • In either outcome, no data corruption visible in Progress tab history.
  • No crash on relaunch; SeedStore.isLoaded completes normally.
Fail indicators
  • App crashes on relaunch after mid-workout kill.
  • Partial workout data appears in Progress tab (incomplete log entry written).
  • Previously completed workouts corrupted or missing from history.
  • UI shows an inconsistent state (some sets from killed session, not others).
QSIM-14 UAT: U-03 comeback message tone
UAT
Precondition
Cohort U-03 tester (returning user, 30-day absence). Simulate a 30-day gap in the workout log by backdating all log entries or using the seed tool. Confirm homeCache.readiness.coachingNote is populated from the "Long Gap" history path before handing device to tester.
Action Steps
  1. Open HomeView for the tester with the 30-day gap simulation active.
  2. Ask the tester to read the coaching note aloud.
  3. Ask the tester: "How does this message make you feel about coming back to training?"
  4. Have the tester rate the tone on a 5-point scale: 1 = Shaming/Guilt-inducing, 3 = Neutral, 5 = Supportive/Welcoming.
  5. Record the verbatim coaching note text and the tester's rating.
Pass / Fail Criteria
Pass
  • Tester rates tone as 3 (Neutral), 4, or 5 (Supportive) on the 5-point scale.
  • Tester does not use words like "guilty", "embarrassed", or "judged" in their verbal response.
  • The coaching note uses the "Long Gap" phrase path from phrase_bank.json.
  • Note is grammatically correct and free of placeholder text.
Fail indicators
  • Tester rates tone 1 or 2 (shaming/guilt-inducing) — escalate immediately to content review.
  • Coaching note is empty, generic (not gap-specific), or contains raw template keys.
  • Tester interprets message as a reprimand for not training.
  • "Long Gap" key missing from phrase bank (see also QSIM-15).
QSIM-15 Regression: phrase bank key sync
Regression
Precondition
Access to phrase_bank.json in the app bundle. This is a static file audit — no runtime app state required. Can be run as a CI pre-check or manually before any release.
Action Steps
  1. Open phrase_bank.json in the app bundle.
  2. Search for key "Short Gap" under the history section.
  3. Search for key "Long Gap" under the history section.
  4. Search for key "Stalled" under the history section.
  5. Search for key "Drop Set — Completed" under the weight section — confirm 10 phrases (IDs 76–85).
  6. Search for key "Build From Drop Set|Single" under whats_next.
  7. Search for key "Use Drop Set Strategy|Single" under whats_next.
  8. Search for key "To Failure" under the connective section — confirm 10 phrases (IDs 105–114).
Pass / Fail Criteria
Pass
  • All 7 keys present with exact string match (case-sensitive, punctuation-sensitive).
  • "Drop Set — Completed" contains exactly 10 phrase entries (IDs 76–85).
  • "To Failure" contains exactly 10 phrase entries (IDs 105–114).
  • No stale or renamed keys that the engine no longer references.
Fail indicators
  • Any of the 7 keys absent — engine falls through to empty string (silent failure).
  • Key present but with wrong casing or punctuation (e.g. "Drop Set - Completed" vs "Drop Set — Completed").
  • Fewer than 10 entries under drop set or to-failure keys.
  • Engine references a key not in the JSON (runtime nil lookup).
QSIM-16 Functional: strength decay floor
Functional
Precondition
Workout log contains a log entry for a specific exercise (e.g., barbell squat at 140 kg × 5) dated exactly 120 days in the past. No more recent log entries for that exercise. Peak e1RM for that set should be calculable via Epley formula.
Action Steps
  1. Launch the app with the 120-day-old exercise log entry.
  2. Navigate to Insights / StrengthLabView.
  3. Locate the specific exercise (e.g., barbell squat).
  4. Read the displayed current strength value and compare to the known peak e1RM.
  5. Calculate expected floor: peakE1RM × 0.50 (50% floor applies after 14 + sufficient decay days).
  6. Verify displayed value equals or is no lower than the 50% floor.
Pass / Fail Criteria
Pass
  • Displayed strength at 120 days equals exactly peakE1RM × 0.50 (floor applied).
  • Value is not below 50% of peak e1RM under any circumstances.
  • Decay model: no decay days 0–14, then −0.7%/day, floor 50%.
  • StrengthLabView renders the exercise without crash or missing value.
Fail indicators
  • Displayed strength falls below 50% of peak e1RM (floor not applied).
  • Displayed strength shows 0 or nil for a 120-day-old log entry.
  • Decay rate applied to the first 14 days (no-decay window broken).
  • Crash or missing exercise in StrengthLabView for stale log entries.
QSIM-17 System: power user stress (90-day log, 20 exercises)
System
Precondition
SeedStore loaded with 90 days of varied workout data covering 20+ distinct exercises. Data should include progressive overload trends, some stalls, and at least 3 exercises with 30+ log entries each. Ensure exerciseHistoryCache is populated with the 20-entry cap per exercise.
Action Steps
  1. Launch the app with the 90-day, 20-exercise dataset.
  2. Navigate to Progress tab — scroll through full history list.
  3. Navigate to Insights / StrengthLabView — scroll through all exercises; inspect decay values.
  4. Navigate to Trainer tab — observe plan generation from WorkoutPlanEngine.
  5. Return to Progress tab and open a specific log entry with 6+ exercises.
Pass / Fail Criteria
Pass
  • All 3 views (Progress, Insights, Trainer) load without crash.
  • No stale cache data: strength values, readiness, and trends reflect the full 90-day log.
  • exerciseHistoryCache correctly caps at 20 entries per exercise (newest-first).
  • Trainer plan reflects current readiness and last performance without stale data.
Fail indicators
  • Any of the 3 views crash with 90-day / 20-exercise dataset.
  • Stale cache data visible (e.g., strength value not reflecting recent log).
  • History cache shows more than 20 entries for any exercise (cap not enforced).
  • Trainer plan generates recommendations based on outdated last-performance data.
QSIM-18 UAT: U-01 power user gear-icon settings discovery
UAT NAV-001
Precondition
Cohort U-01 tester (power user, familiar with previous app version that had a dedicated Settings tab). Post-NAV-001 build: Settings tab removed; gear icon moved to HomeView toolbar. Tester has not been briefed on the navigation change.
Action Steps
  1. Hand the device to the U-01 tester on HomeView.
  2. Say: "Without me telling you where it is, find Settings."
  3. Start a 15-second timer.
  4. Observe where the tester looks first — tab bar, swipe gestures, HomeView toolbar.
  5. Stop timer when tester taps the gear icon or gives up.
  6. Record time-to-find and first-look location.
Pass / Fail Criteria
Pass
  • Tester locates the gear icon in HomeView toolbar within 15 seconds.
  • Tester does not express confusion or frustration about Settings moving.
  • NAV-001 UAT: confirms tab reduction to 5 tabs is acceptable to power users.
  • Settings view opens correctly from the gear icon tap.
Fail indicators
  • Tester cannot find Settings within 15 seconds — escalate to NAV-001 redesign discussion.
  • Tester expresses strong dissatisfaction with Settings relocation.
  • Gear icon not visible or tappable in HomeView toolbar (layout regression).
  • Settings view crashes or renders incorrectly when opened from toolbar.
