## Bug Found and Fixed

### BUG-1: CSV Export Never Shows Share Sheet

**Root cause:** `SettingsView` had two `.sheet` modifiers chained on the same view:
```swift
.sheet(item: $exportItem) { ... }     // JSON export — WINS, always active
.sheet(isPresented: $showExportSheet) { ... }  // CSV export — IGNORED by SwiftUI
```
SwiftUI applies only the first `.sheet` modifier on a given view. The second is silently discarded. The CSV share sheet never appeared because `showExportSheet = true` was setting state on a dead modifier.

**Fix:** Removed `showExportSheet` and `exportURL` state variables. Changed `exportCSV()` to set `exportItem = ExportFile(url: url)` — the same `Identifiable`-based pattern the JSON export already uses. One sheet modifier now handles both exports.

## Real Data Issues Found

### ISSUE-1: Machine Shoulder Press — ToFailure Sets Have reps=0 (Medium)

Sessions S09 (3 sets), S11 (2 sets) log Machine Shoulder Press with `toFailure=true` and `reps=0`. The analytics engine filter is:
```swift
var sets = we.completedSets.filter { $0.reps >= minReps && $0.reps <= maxReps && eq.effectiveWeight($0.weight) > 0 }
```
Sets with `reps=0` fail `>= minReps` (minReps ≥ 1) and are excluded. **Machine Shoulder Press will have near-zero analytics data** despite being one of the most-logged exercises in recent sessions (S07–S11).

**Why this matters:** The user appears to be logging Machine Shoulder Press "to failure" without entering actual rep counts. The app completes the set (button tap triggers completion) but doesn't capture a rep number. This is a UX problem — the "to failure" flow should prompt for a rep count before closing the set.

### ISSUE-2: Drop Sets — Main Reps=0, dropReps Logged (Medium)

S14 Dumbbell Bench Press has 2 completed sets where `reps=0` but `dropReps=10`. The `dropAdjustedE1RM` function:
```swift
let mainE1RM = equipment.effectiveWeight(weight) > 0 && reps > 0 ? estimated1RM : 0  // → 0
let dropRaw  = SetRecord.e1RM(weight: dropWeight ?? 0, reps: dropReps ?? 0)           // → 0 if dropWeight=nil/0
```
If `dropWeight` is 0 or nil, the drop e1RM is also 0. These sets contribute no e1RM data. The user may have logged the drop reps but not the drop weight, producing invisible data.

### ISSUE-3: S06 Early Exit — 6 of 16 Sets Completed (Low)

Apr 29 (feel=Tired): 16 sets were planned across 5 exercises, only 6 completed. The session was clearly cut short. The app correctly logs what was done, but the analytics engine sees a very thin session. This is expected behavior — no bug.

### ISSUE-4: readinessBefore Never Logged (Low)

Not a bug, but 0 out of 18 sessions have `readinessBefore` set. The pre-workout readiness slider is never used. Any readiness-score features that depend on `readinessBefore` input produce nothing. Consider whether the prompt is too easy to skip.
