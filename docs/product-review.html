Section 1 — 48 Must-Pass Product Criteria
✎ 1. Feature Definition Completeness 8 criteria ▼
PR-001
Problem Statement
Every feature entering development must have a written one-sentence problem statement in the format: "Users who [context] cannot [do X] because [root cause]." This is the non-negotiable first artifact before any design or engineering work begins.
Pass: Problem statement exists, references a specific user segment (not "users in general"), and is falsifiable — meaning it could be proven wrong by data.
PR-002
Success Metric
Each feature has exactly one primary success metric defined and agreed upon before the build starts. Secondary metrics are permitted but the primary drives go/no-go decisions. The metric must be answerable without a manual data pull or a human judgment call.
Pass: The metric is quantitative, time-bound, and measurable without manual data collection — a query or dashboard should answer it automatically.
PR-003
Scope Boundary
Every feature spec includes a written "not in scope" list with at least 3 explicit exclusions. This list is written by the PM, reviewed by engineering and design, and treated as a binding constraint for the sprint — not a preference.
Pass: The "not in scope" list exists and specifically prevents the three most likely scope creep additions for this feature type.
PR-004
User Story Coverage
Every user-facing interaction in the design has a corresponding user story written in the format: "As [persona], I want [action] so that [outcome]." Stories must map to specific personas from the cohort, not a generic user archetype.
Pass: No interaction exists in the design spec without a corresponding user story. Each story names a real cohort persona category.
PR-005
Edge Case Documentation
At least 5 edge cases are documented in the spec before development begins. Edge cases are not bugs to handle later — they are design decisions that must be made now. Undocumented edge cases discovered during development represent a spec failure.
Pass: Edge cases explicitly cover at minimum: empty state, maximum data, error state, first-time use, and returning user with existing data.
PR-006
Dependency Mapping
All cross-feature and cross-system dependencies are listed in the spec before the sprint is locked. Dependencies include data model requirements, engine changes, phrase bank additions, and any other feature that must ship first or in parallel.
Pass: No dependency is discovered during development that was not in the pre-build dependency list. Zero "surprise" blockers mid-sprint.
PR-007
Rollback Plan
If a feature must be disabled after launch — due to a bug, poor adoption, or a bad data outcome — the rollback path is documented before the feature ships. Rollback should not require a new App Store submission in most cases.
Pass: Rollback takes under 4 hours and does not require a new App Store submission. Feature flag infrastructure tested before submission.
PR-008
Acceptance Criteria Agreement
Engineering, Design, and Product have all reviewed and confirmed the same acceptance criteria document before build starts. This review must happen within 7 days of sprint start — not at kickoff months prior.
Pass: All three disciplines have reviewed and confirmed the same document within 7 days of the sprint start date. Sign-off log exists.
💡 2. User Value Clarity 7 criteria ▼
PR-009
The "So What" Test
When the feature is described in one sentence to a non-technical person — someone outside the product team — they should immediately understand why it matters without follow-up questions. If the listener needs context or explanation, the value proposition is not clear enough.
Pass: 3 of 3 non-technical test listeners can explain the feature's value back in their own words without prompting.
PR-010
Cohort Mapping
At least 3 cohort members (U-01 through U-22) are identified as primary beneficiaries before build starts. "Primary" means the feature directly addresses a documented problem for that cohort member — not a peripheral benefit.
Pass: The feature solves a documented problem for at least 3 named cohort members. Each mapping references a specific pain point in the cohort profile, not a general assumption.
PR-011
Frequency of Use
Estimated weekly interactions per active user is documented before launch. A feature used once per month and a feature used daily require fundamentally different approaches to design, placement, and coaching integration.
Pass: The frequency estimate exists and is grounded in analogous behavior already observed in the workout log — not a guess or an aspiration.
PR-012
Value Without Coaching
The feature must deliver its core value to a new user who has never read a tooltip, onboarding explanation, or help document. If the feature only works after training, it is not ready to ship. H.O.N. users are in motion — they do not stop to learn.
Pass: Observational test shows a new user can use the feature correctly without any instruction. No verbal cues from the tester.
PR-013
Differential Value
The feature provides something the user cannot get from a competing fitness app or a spreadsheet. H.O.N. is not competing on feature count — it is competing on intelligent, context-aware coaching that only works because of the data model underneath it.
Pass: One clear differentiator is documented that is unique to H.O.N.'s data model, coaching system, or identity-affirming voice — not just better UX on a commodity feature.
PR-014
Negative Impact Assessment
Features are built for a target cohort, but they live in an app used by everyone. A feature that is valuable for U-03 and clutters the experience for U-11 is not ready. Every feature must be assessed for its impact on cohort members who were not its primary target.
Pass: Cohort members outside the primary target can ignore the feature entirely without it adding noise, confusion, or friction to their experience.
PR-015
Minimum Viable Version
The smallest version that delivers the core user value ships first. Everything beyond that is labeled v2 or later. This is not a compromise — it is discipline. V1 must stand on its own; it cannot rely on v2 to make sense.
Pass: The v1 spec is describable in one paragraph. All additional complexity — edge cases, power user features, personalization layers — is explicitly labeled v2+ in the spec.
🎯 3. OKR Quality 7 criteria ▼
PR-016
Objective Clarity
Each Objective is qualitative, aspirational, and time-bound. It describes a direction the team wants to move in, not a metric the team wants to hit. Numbers belong in KRs. An O with a number in it is a KR in disguise.
Pass: The Objective contains no number. It describes a direction and a timeframe. A new team member reading it understands where the product is trying to go.
PR-017
Key Result Measurability
Every Key Result has a baseline value (what it is today), a target value (what it must be by the review date), and a measurement method (how exactly the number will be pulled). A KR without all three is a wish, not a commitment.
Pass: Each KR can be evaluated true or false without subjective interpretation. Anyone on the team should reach the same verdict from the same data.
PR-018
KR Independence
No two Key Results for the same Objective measure the same underlying user behavior. If shipping one feature automatically moves two KRs, those KRs are not independent — and the OKR set is measuring coverage, not impact.
Pass: Achieving any single KR does not automatically achieve any other KR in the same Objective. Each KR requires a distinct behavioral change.
PR-019
Leading vs. Lagging Balance
Every Objective has at least one leading indicator KR (something measurable before the outcome is known) and at least one lagging indicator KR (the outcome itself). Leading KRs let the team course-correct. Lagging KRs confirm whether they did.
Pass: Both types are present. The product team can name which KRs are leading and which are lagging — and why the distinction matters for this Objective.
PR-020
OKR Coverage
The full OKR set covers the business risks relevant to the current product stage. For a pre-launch founder-led product, this means at minimum: activation (do users complete onboarding?), retention (do users come back?), and product quality (is the core loop working?). Gaps in coverage are blind spots.
Pass: No major business risk for the current stage goes uncovered by the OKR set. The team can map each risk to at least one KR.
PR-021
Stretch Calibration
Key Results are set at a 70% confidence level — not guaranteed if the team executes normally, but not a lottery either. KRs the team hits 100% of the time were set too low. KRs the team never hits were set to motivate, not to measure. Both are counterproductive.
Pass: The team rates confidence between 60–80% at the time of setting. Any KR rated above 85% is reset; any rated below 50% is restructured.
PR-022
Retrospective Cadence
OKRs have a defined check-in schedule. Quarter-end reviews are not sufficient — the value of OKRs is in mid-period course correction, and that only happens if check-ins are calendared before the quarter begins.
Pass: Check-ins are scheduled at 4-week intervals with a named responsible person who owns each KR. Check-in notes are documented, not just verbal.
🗺 4. Roadmap Coherence 7 criteria ▼
PR-023
Narrative Arc
The roadmap is not a feature list — it is a story about where the product is going and why. A new team member should be able to read it cold and describe the strategic direction in 5 minutes. If the roadmap requires verbal explanation, it is not doing its job.
Pass: Someone unfamiliar with the project can read the roadmap and accurately describe the product direction in their own words within 5 minutes.
PR-024
Dependency Sequencing
No feature on the roadmap is scheduled before its dependencies are complete. The dependency graph has no cycles. A feature that cannot ship because another feature isn't done is a sequencing failure, not a scope problem.
Pass: The dependency graph has no cycles. Every dependency is scheduled earlier on the timeline than its dependents. The graph was verified, not assumed.
PR-025
Capacity Realism
Roadmap items fit within the team's actual delivery velocity, with a 20% buffer for the unexpected. This is not pessimism — it is the baseline condition for sustainable delivery. A roadmap that requires heroics every quarter is a fictional roadmap.
Pass: Planned scope is at or below 80% of historical quarter capacity. Velocity was measured, not estimated from ideal conditions.
PR-026
Now / Next / Later Structure
The roadmap has three horizons with decreasing specificity. "Now" is what is in active development with full specs. "Next" is what is coming with problem statements written. "Later" is directional — a bet, not a commitment. Treating all three the same creates false certainty.
Pass: "Now" items have complete specs. "Next" items have problem statements. "Later" items have only a direction statement. The distinction is visible in the document.
PR-027
Backlog Link
Every item on the roadmap links to a specific backlog entry with the full detail needed to execute it. The roadmap is a summary view; the backlog is the source of truth. Roadmap items without a backlog link are vaporware.
Pass: 100% of roadmap items have a corresponding backlog entry with a specific ID. No orphaned roadmap items.
PR-028
Kill Criteria
Every feature on the roadmap has a pre-defined kill criterion written before it ships: if metric X does not reach Y within Z days, the feature is removed or pivoted. Kill criteria make removing features the default response to poor performance, not a hard conversation.
Pass: Kill criteria are documented for every feature before it ships. The threshold is specific and time-bound — not "if users don't like it."
PR-029
Stakeholder Alignment
All team leads have reviewed the roadmap within the last 14 days. Roadmaps that have not been reviewed recently are not aligned — they are aspirational artifacts. At the pace H.O.N. moves, 14 days is an eternity; things change.
Pass: A sign-off log shows each team lead reviewed the roadmap within 14 days of any major revision or sprint boundary.
⚠ 5. Risk Assessment 6 criteria ▼
PR-030
Technical Risk
The highest-risk engineering challenges — novel integrations, uncertain performance bounds, untested Apple frameworks — are identified before the sprint is committed and time-boxed as spikes. A spike that discovers the approach won't work is a success. A mid-sprint discovery is a failure of planning.
Pass: No technical unknown consumes more than 20% of a sprint without a prior spike result that justified the approach.
PR-031
User Behavior Risk
Every assumption about how users will actually use a feature is written down explicitly and made falsifiable before launch. Assumptions left implicit become unverifiable after the fact. "Users will log RPE naturally" is an assumption. "Users who see RPE after a completed set will tap it within 3 sessions" is a testable one.
Pass: At least 3 behavioral assumptions are documented. Each has a test that could prove it wrong. Results tracked in post-launch review (PR-045).
PR-032
Competitive Risk
Any competitor shipping similar functionality within 90 days is identified and the team has decided whether to accelerate, differentiate, or proceed unchanged. Competitive moves discovered after launch are planning failures, not bad luck.
Pass: Competitive landscape reviewed in the last 30 days. No known competitor move that would materially affect launch timing or positioning went undetected.
PR-033
Regulatory / Privacy Risk
Any feature touching HealthKit data, Apple Health integration, or personal health metrics has a documented privacy review before it ships. HealthKit is a privileged API. How H.O.N. uses that data — and what it does not do with it — is a core part of the product's identity and trust contract with users.
Pass: HealthKit data use is documented. No health data leaves the device without explicit user consent disclosed in the UI. Privacy review completed and signed off.
PR-034
Scope Creep Risk
Each sprint has a scope freeze date — a date after which no new items are added to the sprint under any circumstance. Additions after the freeze go into the next sprint. The only exception is a P0 bug that blocks shipping.
Pass: Scope freeze date is documented and enforced. All post-freeze additions are logged in the backlog with the sprint they were deferred to.
PR-035
Launch Timing Risk
App Store review is not a variable the team controls — it is a dependency that must be on the critical path. TestFlight distribution, marketing asset completion, and the App Store review buffer must all be accounted for in the launch plan before the submission date is set.
Pass: The launch plan explicitly accounts for a 7-day App Store review buffer. Submission date is set with that buffer included, not as the target launch date.
🚀 6. Launch Readiness Gates 7 criteria ▼
PR-036
Feature Flag
All new features can be toggled off in production without submitting a new build to the App Store. Feature flags are not optional infrastructure — they are the difference between a controlled rollback and a weekend incident.
Pass: Feature flag infrastructure is in place and tested. The flag toggle is verified to work correctly in the TestFlight build before submission.
PR-037
Analytics Instrumented
All meaningful interactions in the new feature are tracked before it launches — not after the first incident reveals a gap. The event list is reviewed by product and engineering before the TestFlight build. Post-launch analytics instrumentation is a D7 data failure waiting to happen.
Pass: The full event list is reviewed and confirmed present in the TestFlight build. No events added post-launch.
PR-038
Support Documentation
The support team has a runbook for every known failure mode before launch day. A runbook that is written after the first incident is a retrospective, not a protocol. Known failures must be anticipated and scripted before real users encounter them.
Pass: Runbook exists and has been reviewed by at least one person outside of engineering. The reviewer confirmed they could execute the runbook without asking for help.
PR-039
Marketing Alignment
The App Store listing — screenshots, description, what's new text — accurately reflects the feature before submission. A listing that describes a feature differently than the feature works creates user expectations H.O.N. cannot meet. This is a trust problem, not a marketing problem.
Pass: Store listing reviewed and approved by both product and brand before submission. Screenshots match actual UI. No aspirational language about capabilities the feature doesn't have.
PR-040
TestFlight Validation
The feature has been used in real conditions by at least 10 TestFlight users. "Used" means complete sessions, not installation. Zero P0 bugs reported from those sessions. If the 10-user bar reveals a P0, fix it and run 10 more sessions.
Pass: 10 or more TestFlight sessions completed by real users. Zero P0 bugs reported from those sessions. Results documented.
PR-041
Rollout Plan
If the feature warrants a phased rollout — new data model, significant UX change, anything that could affect all users immediately — the rollout plan is documented: starting percentage, timeline, and the success threshold required to expand to the next tier.
Pass: Rollout plan exists with defined criteria for moving from 10% to 100%. If no phased rollout is needed, the decision not to phase is documented with rationale.
PR-042
Post-Launch Monitoring
The monitoring dashboard and alert thresholds are configured before the submission is made — not after the first incident reveals they were missing. A dashboard built in response to a crash is a diagnosis. A dashboard built before launch is a defense.
Pass: Monitoring dashboard is live and alert thresholds are configured before the App Store submission is sent. At least one person is on-call for the 24 hours after launch.
📊 7. Post-Launch Review 6 criteria ▼
PR-043
D7 Data Review
Seven days after launch, the full team reviews KR progress together. The review is not a status update — it is a decision meeting. At least one decision must come out of it: iterate, hold, or pivot. A review without a decision is a ceremony.
Pass: D7 review meeting occurred. Data was presented (not summarized verbally). At least one decision was made and documented based on that data.
PR-044
User Feedback Synthesis
Qualitative signals — App Store reviews, support tickets, direct user research — are synthesized into named themes within 14 days of launch. Themes must map to actionable backlog items. A folder of unread reviews is not a feedback process.
Pass: A written feedback summary exists within 14 days. Themes are named and each maps to a specific backlog item or is explicitly recorded as "not actionable now."
PR-045
Hypothesis Validation
Every behavioral assumption documented in PR-031 is evaluated against actual user behavior within 30 days of launch. Each assumption gets a verdict: confirmed, disproved, or inconclusive — with specific data cited. "Inconclusive" is an acceptable outcome; silence is not.
Pass: Each assumption from the risk assessment has a verdict with data cited. No assumption is left unevaluated. Inconclusive results include the specific data gap that prevented a verdict.
PR-046
Retention Impact
At D30, measure the retention delta between users who engaged with the new feature and those who did not. This is the clearest signal of whether the feature is adding to the product's core loop or sitting on the side of it. The delta — positive or negative — directly informs the next roadmap decision.
Pass: The delta is measured and documented. The result — not just the number, but the interpretation — informs a specific next roadmap decision.
PR-047
Technical Debt Log
Any shortcuts taken during development — skipped edge cases, hardcoded values, deferred refactors, known performance trade-offs — are logged within 7 days of launch with a specific remediation timeline. Debt documented is debt manageable; debt undocumented becomes a surprise.
Pass: Tech debt log is updated within 7 days of launch. Each item has a specific remediation timeline — not "someday" — assigned to a named owner.
PR-048
Retrospective
A team retrospective is completed within 21 days of launch. Action items from the retro are assigned to named owners with specific due dates — not shared responsibilities or open-ended timelines. A retro without assigned action items is a venting session, not a process improvement.
Pass: Retro notes exist and are accessible. Every action item has a named owner and a due date. At least one item addresses a repeatable process, not just a one-time fix.

Section 3 — Definition of Done for Product

A feature is done from a product perspective when every stakeholder — engineering, design, and product — is looking at the same written acceptance criteria and can say without qualification that the feature meets them, when the problem statement it was built to solve has been verified to be the right problem by at least three named cohort members who experience the feature directly, when the success metric has a baseline and a measurement method that does not require anyone to manually pull data, when analytics are firing correctly in the TestFlight build and the D7 review is already calendared, when the kill criteria are written down and the team understands what it would take for this feature to be removed, and when a new user can encounter the feature for the first time without any onboarding and understand both what it does and why it matters to them. Done is not "code complete." Done is not "shipped." Done means the feature fulfills its contract — with the problem statement, with the cohort members it was built for, with the metrics that will evaluate it, and with the H.O.N. standard that every word, every number, and every coaching phrase is honest, specific, and worth the attention of the person on the other end of it.

[FEATURE NAME] [PROBLEM STATEMENT] [SUCCESS METRIC] [TARGET COHORT]
You are a product reviewer for H.O.N. (Habit Over Numbers), a founder-led iOS fitness coaching app at pre-launch stage. The app uses a single data store (SeedStore), a phrase bank coaching system, and a coaching voice that is spare, direct, and identity-affirming — never cheerleading. The user cohort has 22 members (U-01 through U-22) across a wide range of training backgrounds, ages, and goals. Review the following feature proposal: Feature Name: [FEATURE NAME] Problem Statement: [PROBLEM STATEMENT] Primary Success Metric: [SUCCESS METRIC] Target Cohort: [TARGET COHORT] Evaluate and return exactly these five outputs: 1. PROBLEM CLARITY (1–5): Rate the problem statement. Does it name a specific user segment? Is it falsifiable? Does it describe a root cause, not just a symptom? Explain the score. 2. USER VALUE SCORE (1–5): Rate how clearly the feature delivers differentiated value. Could a competing app or a spreadsheet deliver the same outcome? Is the value self-evident without onboarding? Explain the score. 3. RISK FLAGS: List any risks this feature introduces — technical risk (SeedStore side effects, analytics engine changes, phrase bank additions), user behavior risk (assumptions that could be wrong), competitive risk, privacy/HealthKit risk. Be specific. No generic risk categories. 4. MISSING SPEC ELEMENTS: List what is absent from the spec that must exist before build starts. Reference the PR-001 through PR-008 criteria. Name each missing element. 5. RECOMMENDED PRE-BUILD ACTIONS: List the 3 most important actions the team must take before this feature enters a sprint. Be direct. No hedging.
