Analytics Simulation Protocol

15 analytics scenarios · Measurement gap matrix · Data capture template · LLM pre-triage prompt · Last updated: 2026-05-30
Section 1 — How to Run Analytics Simulations
Analytics simulations are structured test runs that verify our instrumentation answers real product questions — not just that events fire, but that the resulting data is actually usable. Each simulation assigns a specific analyst or team member to one scenario, defines the exact data source and query approach, and produces a documented finding. Simulations are run in order of priority: start with ASIM-01 through ASIM-07, then ASIM-08 through ASIM-15 as time permits. Priority pairings in the matrix in Section 3 (marked H) indicate which team role is best suited for each scenario. A simulation is only complete when the Pass or Fail verdict is written with supporting evidence — not when the steps are finished. Results are captured using the template in Section 4.
Section 2 — 15 Analytics Simulation Scenarios
ASIM-01
New User Funnel Trace
Funnel: Onboarding Priority: Critical
What We're Measuring
Can we trace a specific new user from install through their first completed workout using only event data, without any developer tools or device access?
Data Source
Analytics event stream (App Store Connect / TestFlight diagnostics or local event log). Events: onboarding_started, profile_saved, onboarding_completed, session_started, set_logged, session_ended.
Steps
  1. Simulate a fresh install. Complete onboarding with a standard profile (goal: strength, equipment: barbell, fitness level: intermediate).
  2. Log one complete workout: 3 exercises, 3 sets each, all sets marked complete.
  3. Export or inspect the event stream. List all events in order of timestamp.
  4. Attempt to reconstruct the full user journey from events alone: install → onboarding → first session → first set → session end.
  5. Confirm each funnel step is represented by at least one event with a correct timestamp and non-null required fields.
Pass
Full journey traceable from events alone. All 6 event types present. Timestamps in correct sequence. No required field is null. An analyst with no app access could reconstruct the journey from the event log alone.
Fail
Any step missing from the event stream. Any required field null or empty. Events out of chronological order. Journey reconstruction requires developer-level app access or database inspection beyond the event log.
ASIM-02
Feature Adoption Audit
Funnel: Feature Adoption Priority: High
What We're Measuring
30 days after a new feature ships, can we answer: what % of users tried it, what % used it more than once, and what % use it in every session?
Data Source
Events: feature_first_used, tab_viewed, and feature-specific events (e.g., coaching_note_expanded, plan_generated). Query: group by user, count distinct sessions with feature use.
Steps
  1. Choose one feature that has shipped in the last 30 days (e.g., RPE logging, drop sets, Trainer tab).
  2. Identify the key event for that feature (e.g., rpe_logged, drop_set_completed, plan_generated).
  3. From the event log, count: users with exactly 1 use (one-time tryout), users with 2–4 uses (occasional), users who use it every session (habitual).
  4. Calculate adoption tiers: tryout %, occasional %, habitual % of total users who have logged at least one session.
  5. Identify the top adoption barrier: which users tried and never returned to the feature?
Pass
All three adoption tiers measurable (tryout, occasional, habitual). Top adoption barrier identifiable. Segment of try-and-abandon users extractable for follow-up. Query runs without requiring non-event data sources.
Fail
Cannot distinguish one-time tryout from habitual use. Feature event not present in stream or uses inconsistent naming. Query requires device-level data access beyond the event log.
ASIM-03
Churn Signal Detection
Funnel: Churn Signal Priority: Critical
What We're Measuring
Can we identify users who have not logged a session in 12+ days but opened the app in the last 7 days — the pre-churn signal window?
Data Source
Events: session_started (to find last session date) and app_opened_no_session (app open without logging). Filter: last_session_date > 12 days ago AND last_app_open < 7 days ago.
Steps
  1. From the event log, identify the most recent session_started per user. Calculate days since last session.
  2. Filter for users where days since last session is between 12 and 29 (pre-churn window).
  3. From that filtered set, further filter for users with an app_opened_no_session event in the last 7 days.
  4. Calculate this segment as a percentage of MAU. This is the pre-churn warning cohort.
  5. Confirm: can you distinguish an app open that resulted in a session from one that did not, using only events?
Pass
Pre-churn segment identifiable and sized as % of MAU. App opens with and without sessions are distinguishable from events. Days-since-last-session calculable without database access. Segment extractable for re-engagement.
Fail
Cannot distinguish app opens with sessions from opens without. app_opened_no_session event missing or not firing. Pre-churn segment not quantifiable from event data alone.
ASIM-04
Coaching Engagement Measurement
Dashboard: Coaching Engagement Priority: High
What We're Measuring
What % of post-set coaching notes result in the user reading more (expanding) vs. immediately dismissing? Which coaching note type has the lowest engagement?
Data Source
Events: coaching_note_shown and coaching_note_expanded, both with note_type field. Engagement rate = expanded / shown per note type.
Steps
  1. Pull all coaching_note_shown events. Group by note_type (post-set, readiness, trainer, summary). Count per type.
  2. Pull all coaching_note_expanded events. Group by note_type. Count per type.
  3. Calculate expansion rate per type: expanded count / shown count. Express as a percentage.
  4. Rank note types by expansion rate. Identify the lowest-engagement type.
  5. For the lowest-engagement type, confirm: are dismissals happening within 1 second (not read) or after 3+ seconds (read but not expanded)?
Pass
Expansion rate per note type calculable. Lowest-engagement note type identifiable without manual inspection. Dismiss vs. expand distinguishable from event names alone. Time-in-view calculable if timestamps are millisecond-precision.
Fail
Expand vs. dismiss not distinguishable from events. note_type field missing or inconsistent values. Coaching note shown events not firing or not matched to user actions correctly.
ASIM-05
A/B Test Evaluation Simulation
Category: A/B Testing Priority: High
What We're Measuring
Given two event streams (variant A and B), can we determine which performed better on the primary metric using proper statistical methods?
Data Source
Simulated event data for 200 users split 50/50. Variant field attached to each user's events. Primary metric: session completion rate (session_ended / session_started). Apply chi-squared test or equivalent.
Steps
  1. Generate or obtain simulated event data: 100 users in variant A, 100 in variant B. Both groups have 2 weeks of sessions.
  2. Calculate session completion rate for each group: session_ended / session_started events per user, then average across group.
  3. Apply a two-proportion z-test or chi-squared test to determine if the difference is statistically significant (p < 0.05).
  4. If p < 0.05, declare a winner. If p ≥ 0.05, declare result inconclusive.
  5. Check: was the sample size sufficient to detect the observed effect at 80% power? If not, document the underpowered result.
Pass
Clear winner identifiable at p < 0.05 OR result is declared inconclusive at current sample size with a documented explanation. Statistical test can be run from event data alone. Sample size check performed.
Fail
Cannot perform statistical test from available data. Variant assignment not recoverable from events. Metric not calculable at per-user level. Test evaluated before reaching minimum sample size.
ASIM-06
Retention Cohort Analysis
Funnel: Retention Cohort Priority: Critical
What We're Measuring
For users who installed in a specific week, what are their D1, D7, and D30 retention rates? Can we generate a retention curve from events alone?
Data Source
Events: onboarding_started (install proxy) and session_started (retention proxy). Group by install week. Count users with sessions on or within ±1 day of D1, D7, D30.
Steps
  1. Identify the install cohort: users whose onboarding_started event falls within a target week (e.g., the week of TestFlight launch).
  2. For each user in the cohort, calculate their install date from onboarding_started timestamp.
  3. For D1 retention: count users with a session_started event on install_date + 1 (±1 day tolerance). Divide by cohort size.
  4. Repeat for D7 and D30. Plot the resulting three-point retention curve.
  5. Filter the cohort by onboarding goal (strength/consistency/body composition). Generate a separate retention curve per goal. Compare curves.
Pass
Retention curve generated from event data alone. D1, D7, D30 all distinct and measurable. Cohort filterable by onboarding goal. A user with sessions on day 0 and day 29 correctly counted as D30-retained.
Fail
Cannot distinguish D7 retention from D30 retention. Install date not recoverable from events. Goal-based cohort filtering not possible. Retention metric requires device-level data beyond the event stream.
ASIM-07
Funnel Drop-off Diagnosis
Funnel: Onboarding Priority: Critical
What We're Measuring
In the onboarding funnel, which step has the highest drop-off rate? Can we pinpoint exactly where we are losing users during first-run setup?
Data Source
Five onboarding events in sequence: onboarding_started, profile_saved, onboarding_completed, session_started (first session). Count users reaching each step. Drop-off = users at step N minus users at step N+1.
Steps
  1. Pull all onboarding_started events. This is the funnel entry count.
  2. Pull users who also have profile_saved. Calculate step 1 → step 2 conversion rate.
  3. Pull users who also have onboarding_completed. Calculate step 2 → step 3 conversion rate.
  4. Pull users who also have a session_started within 24 hours of onboarding_completed. Calculate step 3 → step 4 conversion rate.
  5. Identify the step with the largest absolute drop in users. This is the primary UX intervention target.
Pass
Biggest drop-off step identified by name. % of users who stopped at that step is calculable. Funnel steps sequenceable for individual users (not just aggregate counts). Time between steps calculable from timestamps.
Fail
Cannot sequence the funnel steps for a single user. Events aggregated at the count level with no per-user attribution. Biggest drop-off step not distinguishable from noise in small samples.
ASIM-08
Trainer Adoption Funnel
Funnel: Trainer Adoption Priority: High
What We're Measuring
Of users who viewed the Trainer tab, what % generated a plan? Of those, what % started it? Of those, what % completed it? Is the Trainer funnel healthy?
Data Source
Events: trainer_tab_viewed, plan_generated, plan_started, plan_completed. Three conversion rates calculable from sequential event counts per user.
Steps
  1. Count users with at least one trainer_tab_viewed event. This is the funnel top.
  2. Of those, count users who also have plan_generated. Calculate view → generate conversion rate.
  3. Of those, count users who also have plan_started. Calculate generate → start conversion rate.
  4. Of those, count users who also have plan_completed. Calculate start → complete conversion rate.
  5. Identify users who generated a plan but never started it. Size this segment — they are the highest-priority Trainer re-engagement target.
Pass
Three-step funnel visible with conversion rates. Each conversion rate calculable without manual lookup. Plan-generated-but-never-started segment sized. Users who started but did not complete are distinguishable from completers.
Fail
Any step in the funnel missing from the event stream. Cannot distinguish plan_generated from plan_started. Completers not distinguishable from starters in event data.
ASIM-09
Cardio vs. Strength User Segmentation
Funnel: Cardio Adoption Priority: Medium
What We're Measuring
Can we identify dual-modality users (strength + cardio) vs. single-modality users from event data? What % of active users use both modalities?
Data Source
Events: strength session_started and cardio_session_started. A dual-modality user has both event types in their history. Single-modality users have only one type.
Steps
  1. From the event log, identify all users with at least one strength session_started event (strength users).
  2. From the event log, identify all users with at least one cardio_session_started event (cardio users).
  3. Find the intersection: users who appear in both lists. This is the dual-modality segment.
  4. Calculate: dual-modality % of total active users. Strength-only %. Cardio-only %.
  5. Compare retention curves for dual-modality vs. single-modality users. Hypothesis: dual-modality users have higher D30 retention.
Pass
Dual-modality segment identifiable and sized from events alone. Cardio and strength sessions distinguishable without inspecting session content. Retention comparison between modality segments calculable.
Fail
Cardio and strength sessions not distinguishable from event data (same event name with no modality field). Dual-modality users cannot be identified without device-level data access.
ASIM-10
Error Rate Hotspot
Dashboard: Error Rate Priority: High
What We're Measuring
Which input field has the highest abandonment rate after a validation error? Where are users giving up instead of correcting their input?
Data Source
Events: input_validation_error (with field_name, error_type), error_recovered, error_abandoned. Abandonment rate per field = abandoned / (abandoned + recovered).
Steps
  1. Pull all input_validation_error events. Group by field_name. Count total errors per field.
  2. For each field, pull error_recovered and error_abandoned events that follow an error on that field.
  3. Calculate abandonment rate per field: error_abandoned / (error_abandoned + error_recovered).
  4. Rank fields by abandonment rate. Identify the top 3 highest-abandonment fields.
  5. For the highest-abandonment field, identify the most common error_type. This is the primary UX fix target.
Pass
Top 3 error-prone fields identifiable with abandonment rates within 48 hours of launch. field_name links the error event to the specific input. Recovery vs. abandonment distinguishable per field. Most common error type per field identifiable.
Fail
Cannot link the error event to the specific field (field_name missing or generic). Recovery vs. abandonment not distinguishable as separate event types. Top-abandonment fields not rankable from event data alone.
ASIM-11
Session Quality Trend
Dashboard: Session Quality Priority: Medium
What We're Measuring
Is median session duration improving, flat, or declining over the first 90 days of TestFlight? Is the app producing substantive workouts over time?
Data Source
Events: session_started and session_ended (both with UTC timestamps). Duration = session_ended.timestamp minus session_started.timestamp per session. Group by calendar week. Calculate median per week.
Steps
  1. Pull all matched session_started / session_ended event pairs. Calculate duration in minutes for each session.
  2. Discard outliers: sessions under 3 minutes (likely test sessions) and over 120 minutes (likely forgot to end).
  3. Group by calendar week. Calculate median duration per week for 13 weeks (90 days).
  4. Plot the weekly median trend. Assess direction: improving, flat, or declining.
  5. Break down by user segment (goal: strength vs. consistency). Do different goals produce different duration trends?
Pass
Trend direction (improving/flat/declining) identifiable over 13 weeks. Week-over-week median calculable. Session pairs matched correctly by session ID. Outlier filtering possible from duration field alone.
Fail
Session duration not calculable from events (timestamps missing or unpaired). Session started and ended events cannot be matched to the same session. Trend direction not assessable due to insufficient data or incorrect pairing.
ASIM-12
Privacy Audit
Category: Privacy Priority: Critical
What We're Measuring
Simulate a regulatory request: confirm that no PII appears in any event in the analytics stream. This is a compliance exercise, not a product question.
Data Source
Raw event stream. Pull 100 random events across all event types. Manually inspect every field value — not just field names, but actual values — for any string that could identify a person.
Steps
  1. Pull 100 random events from the analytics stream, sampled across all event types and time periods.
  2. For each event, list every field and its value. Do not skip fields with values that appear to be IDs or hashes.
  3. Check each value against a PII checklist: name, email, phone, device name (can include real names), HealthKit-derived values, precise geolocation.
  4. Flag any field value that could identify or re-identify a specific person, including indirect combinations (e.g., age + zip + fitness level in a small population).
  5. Document findings: zero PII found (pass) or list each PII instance with field name, event type, and recommended fix (fail).
Pass
Zero PII found in 100 events. Review conducted by a non-engineer (person without knowledge of the schema design). Review documented with reviewer name, date, and list of events inspected.
Fail
Any field contains a name, email, HealthKit value, device name that includes a real name, or any combination of fields that re-identifies a user in a cohort of fewer than 22 people.
ASIM-13
Dashboard Latency
Dashboard: All Priority: Medium
What We're Measuring
How long after a session ends does it appear in the analytics dashboard? Is the data fresh enough to make daily product decisions?
Data Source
Live analytics dashboard (App Store Connect or equivalent). Log one session at a known timestamp. Record dashboard state at 1h, 6h, 24h after session. Note when the session first appears.
Steps
  1. Log a complete session. Note the exact time the session_ended event fires (to the minute).
  2. Open the analytics dashboard. Confirm the session is not yet visible. Record current dashboard state.
  3. At exactly 1 hour after session end, refresh the dashboard. Is the session visible? Record result.
  4. At 6 hours after session end, refresh again. Is the session visible? Record result.
  5. At 24 hours after session end, refresh again. The session must be visible by this point. If not, document the latency as a failure and escalate.
Pass
Session reflected in the analytics dashboard within 24 hours of session_ended event. Target: visible within 6 hours. North Star metric (Weekly Active Loggers) updates within 24 hours.
Fail
Session not reflected in any dashboard within 24 hours. Dashboard last-updated timestamp more than 24 hours old. Session visible in raw event stream but not in computed metrics (pipeline failure).
ASIM-14
Comeback Correlation
Funnel: Comeback Priority: High
What We're Measuring
Do users who receive a comeback message have a higher same-day session rate than those who open the app after a gap but do not receive a message? Does the message actually work?
Data Source
Events: comeback_message_shown (users who received the message after a 7+ day gap) and session_started within 24 hours. Control group: users with a 7+ day gap who opened the app but did not see a message.
Steps
  1. Segment users: those with a comeback_message_shown event (treatment) and those with a 7+ day gap + app open but no message (control).
  2. For each segment, count users who also have a session_started event within 24 hours of the gap-day app open.
  3. Calculate same-day session rate for treatment vs. control. Is treatment rate higher?
  4. Calculate the difference in same-day session rates. If treatment is higher by more than 10 percentage points, the message is effective.
  5. Document: message text variant (if A/B tested), gap length distribution for treatment vs. control, and whether the correlation is consistent across different gap lengths (7–14 days vs. 15–29 days).
Pass
Correlation measurable from events. Treatment vs. control groups identifiable. Same-day session rate calculable per group. Message effectiveness quantifiable as a percentage-point difference. Gap length available from timestamps.
Fail
Cannot link comeback message event to subsequent session event for the same user. No control group (all gap users receive the message, preventing comparison). Gap length not calculable from event timestamps.
ASIM-15
Schema Version Stress Test
Category: Data Integrity Priority: High
What We're Measuring
After a breaking schema change (renaming a field), is historical data still analyzable? Does schema versioning protect us from silent data loss when the event structure evolves?
Data Source
Event stream with mixed schema versions. Old events use exercise_id. New events use exerciseId (hypothetical rename). Verify both are queryable without data loss.
Steps
  1. Confirm a baseline: run a query using exercise_id on historical events. Note the count of events returned.
  2. Simulate a schema change: in the event schema definition, rename exercise_id to exerciseId. Increment the schema_version field from 1 to 2.
  3. Log 5 new events with the updated schema (version 2). Confirm new events use exerciseId.
  4. Run the original query using exercise_id. Confirm historical (version 1) events are still returned. Confirm version 2 events are not counted (correct — they use the new field name).
  5. Run a new query using exerciseId. Confirm only version 2 events are returned. Document: is it possible to join both schema versions in a single query by checking schema_version?
Pass
Historical data queryable with the old field name after schema change. New data queryable with new field name. schema_version field allows filtering by version. A combined query spanning both versions is possible by checking schema_version.
Fail
Historical data becomes inaccessible after schema change. schema_version field absent or not populated. No mechanism to distinguish old-schema from new-schema events. Breaking change silently invalidates historical queries.

Section 4 — Results Capture Template

Copy this template for each simulation run. File in docs/uat/ with filename ASIMxx-YYYY-MM-DD.md. Or use the uat command for quick entries.

Run Metadata
Scenario: ASIM-__ · Analyst: [name] · Date: YYYY-MM-DD · Data source: [event stream / App Store Connect / local log] · Build or TestFlight version: [build number]
Scenario
State the scenario in one sentence. Example: "Can we trace a new user from install to first completed workout using only event data?"
Data Source
Describe where the event data came from (e.g., local debug event log, App Store Connect aggregate, TestFlight crash + analytics report). State the time range and cohort of events used.
Query Used
Describe the query or filter logic used to obtain the data. If SQL-like syntax was used, include the key clauses. If manual inspection was used, describe the inspection method. Be specific enough that someone else could reproduce the query.
Finding
State the result in plain language. Include numbers: rates, counts, percentages. Example: "Session completion rate: 84%. Top drop-off step: set_logged to session_ended (11% abandonment)." Do not editorialize — state what the data shows.
Data Gap Discovered
Any measurement gap found during this run: a missing event, a null field, an untracked user segment, or a funnel step that cannot be reconstructed from events. "None discovered" is a valid entry. Data gaps discovered here should be filed as instrumentation bugs in BACKLOG.md.
Action Required
What must happen as a result of this finding? Examples: "Fix null field in coaching_note_shown event — assign to engineering," "Add schema_version to all events by next TestFlight build," "Update privacy label to reflect fitness_level collection." Be specific about owner and urgency.
Pass / Fail
PASS / FAIL / PARTIAL — with one sentence of justification. Example: "FAIL — cannot distinguish cardio_session_started from strength session_started without inspecting session content, violating ASIM-09 pass criteria." Cross-reference the relevant criteria from analytics-review.html (AN-0xx).
