Support Simulation Protocol

15 support scenarios · Triage priority matrix · Response capture template · LLM pre-triage prompt · Last updated: 2026-05-30
Section 1 — How to Run Support Simulations
Support simulations are structured drills that replicate real inbound contacts — crash reports, data loss emails, App Store reviews, HealthKit complaints, sensitive welfare contacts — before any of them happen at scale. Each scenario gives the support operator a trigger (a realistic user message or situation) and a set of steps to execute using only the documented support materials. The operator follows the steps exactly, timing each stage against the defined SLAs. Results are captured using the template in Section 4 and filed in docs/uat/. Any step that requires guessing is a documentation gap — mark it and fix the documentation, not the person. Run all 15 scenarios before App Store submission. Prioritize the matrix H-marked scenarios for each role first; secondary scenarios as time allows. A scenario is complete only when the response has been drafted, the SLA measured, and the escalation decision confirmed — not just when the steps are read.
Section 2 — 15 Support Simulation Scenarios
SSIM-01
P0 Crash on Launch
15 min Trigger: Crash report Focus: P0 crash SLA
What We're Testing
P0 classification speed, repro kit usage, engineering notification timing, and first user response within the defined SLA window.
Trigger
User report: "App crashes every time I open it. Happened 3 times."
Steps
  1. Classify severity using the crash matrix — is this P0? (Launch crash = P0 by definition.)
  2. Check known issues log for a matching crash pattern before assuming it's new.
  3. Execute the repro kit: gather device model, iOS version, app version, approximate session count from the report. Note what's missing.
  4. If reproduced: notify engineering immediately using the escalation tree; draft user response acknowledging the crash and stating a fix is being investigated.
  5. If not reproduced: draft response requesting the missing repro info — specific, not a generic "please send more details."
  6. Measure time from report received to: classification complete, engineering notified, user response sent.
Pass
Classified P0 within 30 minutes. Engineering notified within 1 hour. User response sent within 4 hours. Repro kit used; all fields populated or marked missing.
Fail
Any step exceeds the SLA window. Crash classified below P0. Repro kit skipped. User response is generic or delayed past 4 hours.
SSIM-02
Data Loss Report
15 min Trigger: User email Focus: Data loss protocol
What We're Testing
Immediate P0 classification, honest communication about local-only storage, recovery step execution, and engineering escalation with full context.
Trigger
User report: "I updated the app and all my workouts are gone. 6 months of data."
Steps
  1. Classify immediately as P0 — any data loss is P0 by the severity matrix.
  2. Request: prior version number, current version, iOS version, device model. Ask for these specifically, not generically.
  3. Check if this symptom matches a known migration bug in the current release notes or known issues log.
  4. Execute the data loss protocol: explain that data is stored locally on device only (no server backup), ask about iCloud backup presence, provide recovery steps if any exist.
  5. Draft a response that is honest about recoverability — no false hope, no deflection. Warm and direct.
  6. Escalate to engineering with full context: version numbers, repro kit data, and the response you sent.
Pass
Response sent within 4 hours. Honest about local-only storage — no implication of a server backup. Recovery steps included where available. Engineering escalated with repro kit completed.
Fail
Response promises recovery that isn't possible. Delays past 4 hours. Suggests a server restore that doesn't exist. Engineering not escalated with full context.
SSIM-03
App Store 1-Star Review — Known Bug
10 min Trigger: App Store review Focus: Review response quality
What We're Testing
Known-issue matching speed, specificity of response, voice compliance, and 48-hour posting discipline.
Trigger
Review: "Decimal weights are always wrong. 95 lbs becomes 95.1151 somehow."
Steps
  1. Recognize this as B-001 (weight decimal overflow — known bug). Check the known issues log to confirm fix status.
  2. Check fix status: is B-001 fixed in the currently live build? If yes, which version? If no, what's the fix timeline?
  3. Draft a response that names the specific bug (decimal overflow in weights), explains it's known and being fixed with version if available, or confirms it's fixed and asks the user to update.
  4. Review the draft against the voice guide: no "Dear valued user", no "we apologize for the inconvenience", no template language. Response should sound like one human to another.
  5. Post the response within 48 hours of the review appearing.
Pass
Response is personal, names the decimal overflow specifically, on-brand, posted within 48 hours. If fixed: version number included. If not fixed: honest about timeline.
Fail
Generic template response. No mention of the specific bug. Posted after 48 hours. Uses any banned corporate filler phrase.
SSIM-04
Coaching Voice Complaint
10 min Trigger: User email Focus: Feedback routing + response quality
What We're Testing
Correct classification as feedback (not a bug), routing to content team, quality of response, and backlog/synthesis logging.
Trigger
User email: "The coaching tips feel robotic. It keeps saying the same thing every session."
Steps
  1. Classify: this is feedback, not a bug. Route to content team. Do not open an engineering ticket.
  2. Check the phrase rotation logic — is the phrase bank rotation window working? Is this a specific exercise with limited phrase variants? (Check known issues log for B-related phrase bank issues.)
  3. Draft a response that acknowledges the feedback without being defensive, gives a brief honest explanation of how the rotation works (without being technical), and asks for a specific example exercise to investigate.
  4. Tag this in the monthly feedback synthesis log under "voice complaints."
  5. Create a phrase bank review task for the content team — include the user's specific complaint text.
Pass
Response is warm, specific, invites more detail, not defensive. Feedback logged in synthesis. Content team task created with user complaint text attached.
Fail
Response is defensive ("our coaching is highly personalized"). No follow-up action. Bug ticket opened instead of feedback routing. Synthesis log not updated.
SSIM-05
HealthKit Permission Issue
10 min Trigger: User email Focus: HealthKit diagnostic
What We're Testing
Diagnostic script retrieval speed, clarity of written instructions for a non-technical user, and escalation threshold for unresolved cases.
Trigger
User email: "The app keeps asking for health permissions even though I already said yes."
Steps
  1. Pull the HealthKit diagnostic script from the support knowledge base.
  2. Walk through the script: (a) Check Health app → Sources → H.O.N. → verify all permissions are granted, not just some. (b) If permissions appear granted but the issue persists: remove app from Health Sources and re-grant. (c) If still failing: check iOS version compatibility with current build.
  3. Draft a clear, numbered response the user can follow without needing to email again. Plain language. No technical jargon.
  4. If the issue is still unresolved after the user follows all steps: escalate to engineering with the diagnostic results.
Pass
Response includes clear numbered steps a non-technical user can follow. Steps resolve the issue for the majority of similar reports. Escalation threshold defined and documented.
Fail
Steps are technical or unclear. User requires a second email to get actionable instructions. Escalation threshold not defined.
SSIM-06
Feature Request — Apple Watch
5 min Trigger: User email Focus: Feature request routing
What We're Testing
Accurate backlog check, honest response without false promises, product team routing, and feature request logging.
Trigger
User email: "Any chance you'll add Apple Watch support for rep counting?"
Steps
  1. Check backlog — velocity profile feature exists in the models (Apple Watch data is partially implemented in the data layer). Do not communicate this internal detail; just understand the context.
  2. Draft a response that acknowledges the request, does not promise a timeline, gives an honest status ("it's something we've thought about"), and communicates that the request is logged.
  3. Log the request in the feature request tracker with a note of the user's context (rep counting specifically).
  4. Notify the product team that this request has been received.
Pass
Response is honest (no false promises), warm, confirms the request is logged. Product team notified. Feature request in tracker with user context.
Fail
Response promises Apple Watch support in a specific version. Request dismissed without logging. Product team not notified.
SSIM-07
Three Duplicate Reports — Same Bug
10 min Trigger: 3 emails, 1 week Focus: Duplicate detection efficiency
What We're Testing
Deduplication process execution, single-ticket discipline, all-user response coverage, and priority escalation for multi-user bugs.
Trigger
Three separate user emails in one week describing the same symptom: FAIL pill overflowing the reps field (B-002).
Steps
  1. Read all 3 reports. Identify the common symptom (FAIL pill layout overflow).
  2. Match to B-002 in the known issues log. Confirm it's the same bug, not three variations.
  3. Create one engineering ticket (if not already open) rather than three. Note "3 user reports" in the ticket.
  4. Respond to all 3 users with consistent information: the bug is known, fix timeline if available, workaround if any exists.
  5. Flag to engineering that this is affecting multiple users — this may increase fix priority.
Pass
All 3 correctly linked to B-002. One ticket created (not three). All 3 users receive responses within 48 hours. Engineering notified of multi-user impact.
Fail
3 separate tickets created. One or more users do not receive a response. Engineering not told this affects multiple users.
SSIM-08
Post-Fix Review Update
8 min Trigger: v1.1 ships with B-001 fix Focus: Issue closure
What We're Testing
Review history retrieval, proactive follow-up protocol execution, specific fix-version communication, and reviewer re-engagement.
Trigger
B-001 (decimal weight overflow) fixed and shipped in v1.1. User left a 2-star review citing this bug in v1.0.
Steps
  1. Identify all App Store reviews that mention the decimal weight bug, posted before the v1.1 release date.
  2. Draft a follow-up reply on each review: names the fix, names the version, invites the user to confirm it's resolved and update their rating if it is.
  3. Post all replies within 7 days of v1.1 shipping.
  4. Track whether any reviewers update their rating (note: Apple does not always notify; check manually).
Pass
All affected reviews replied to within 7 days of v1.1 shipping. Specific version number (1.1) mentioned. Reviewers explicitly invited to update their rating.
Fail
No follow-up on fixed bugs. Old negative reviews remain with no update response. Version number omitted from reply.
SSIM-09
Launch Day Monitoring Run
20 min (dry run) Trigger: Simulated launch day Focus: Runbook execution
What We're Testing
Runbook completeness, monitoring cadence execution, P1 simulation handling under time pressure, and runbook gap identification before real launch.
Trigger
Simulate launch day: app is live, monitoring begins. Execute the full runbook start to finish.
Steps
  1. Execute the launch day runbook step-by-step. Do not skip steps because the situation is simulated.
  2. Check App Store Connect crash reports (even if empty — confirm the access and navigation work).
  3. Check support email inbox. Confirm response SLA clock would be running.
  4. Check TestFlight feedback panel.
  5. Simulate one P1 report arriving: classify it, draft a response, decide escalation — using only the documented runbook.
  6. Note every runbook step that was ambiguous, took longer than expected, or required a judgment call not covered by the documentation.
  7. Update the runbook based on all findings before real launch.
Pass
All runbook steps completed within the planned time window. P1 simulation handled correctly. All ambiguous steps identified and resolved in the runbook before real launch day.
Fail
A runbook step is missed or mishandled. P1 simulation results in wrong classification or missed escalation. Runbook not updated after findings.
SSIM-10
Sensitive User Contact
10 min Trigger: User email Focus: Sensitive feedback protocol
What We're Testing
Welfare concern recognition, non-medical-advice discipline, warm non-judgmental response, appropriate resource provision, and no-app-promotion compliance.
Trigger
User email: "I've been tracking everything obsessively and I can't stop working out even when I'm in pain. I don't know what to do."
Steps
  1. Recognize this as a sensitive welfare concern — not a bug, not a feature request. Pull the sensitive feedback protocol.
  2. Confirm you will not give medical advice in the response. Not one sentence.
  3. Draft a response that: acknowledges the experience warmly and without judgment, does not use the word "obsessive" back at them (mirrors their concern, does not label it), gently mentions that talking to someone — a friend, a doctor — can help, and provides a general mental health resource (e.g. NAMI helpline, or similar).
  4. Do not promote continued use of the app anywhere in the response. This is not the moment.
  5. Flag to the founder for awareness only — not for any data use or follow-up marketing.
Pass
Response is warm, non-judgmental, contains no medical advice, provides a concrete resource. App not promoted in the response. Founder flagged for awareness.
Fail
Response gives medical advice. Response is clinical or dismissive. App use encouraged in this context. The word "obsessive" used back at the user. No resource provided.
SSIM-11
GDPR Data Deletion Request
10 min Trigger: User email Focus: Privacy compliance
What We're Testing
24-hour acknowledgment SLA, data disclosure accuracy (local-only), deletion instruction completeness, 7-day confirmation deadline, and request logging.
Trigger
User email: "Please delete all my data immediately and tell me exactly what you had."
Steps
  1. Acknowledge receipt within 24 hours — even if the full response takes longer.
  2. Send the data disclosure document: what's stored (workout log, exercise history, user profile, cached analytics — all local on device), where it lives, and what is not stored (no server, no analytics backend).
  3. Instruct the user how to delete their data: delete the app from their iPhone deletes all local data. Confirm whether they also want any email correspondence deleted from support records.
  4. Confirm deletion within 7 days (app deletion is immediate if they do it; confirm receipt of their action).
  5. Log the request with a timestamp and the date confirmation was sent.
Pass
Acknowledgment within 24 hours. Data disclosure document accurate (local-only; no server). Deletion instructions correct. Confirmation sent within 7 days. Request logged with timestamps.
Fail
Acknowledgment delayed past 24 hours. Disclosure document incomplete or implies server storage. Confirmation not sent within 7 days. Request not logged.
SSIM-12
Knowledge Base Gap Discovery
8 min Trigger: User asks uncovered question Focus: Knowledge base maintenance
What We're Testing
First-principles answering accuracy, knowledge base update discipline within 24 hours, and feature-demand signal detection.
Trigger
User asks a question not in the knowledge base: "Does H.O.N. work with Garmin?"
Steps
  1. Answer from first principles: H.O.N. is Apple ecosystem only — HealthKit, no Garmin integration. Be honest and specific about what integrations do exist (HealthKit, Apple Watch velocity data).
  2. Draft a response that is direct, does not apologize for the limitation, and explains what the app does instead.
  3. After sending, write a knowledge base entry for this question: "Does H.O.N. work with Garmin or non-Apple fitness trackers?" with the accurate answer.
  4. Check if this signals demand for a third-party integration — if the question is received from 3 or more users, log it as a feature request with the "new feature" category.
Pass
Question answered correctly (Apple-ecosystem only). Knowledge base updated within 24 hours. Feature request logged if threshold reached.
Fail
Question answered incorrectly or vaguely. Knowledge base not updated within 24 hours. No feature demand signal tracked.
SSIM-13
Negative Review — Positioning Complaint
8 min Trigger: 1-star App Store review Focus: Review response quality — positioning
What We're Testing
Non-defensive response discipline, one-sentence differentiator communication, re-engagement invitation, and voice compliance under pressure.
Trigger
1-star review: "This app doesn't do anything my old spreadsheet couldn't do. Waste of money."
Steps
  1. Recognize this as a positioning complaint — not a bug, not a missing feature. The person chose a different mental model for fitness tracking.
  2. Draft a response that: does not argue with the comparison, acknowledges their preference is valid, communicates H.O.N.'s specific differentiator (coaching voice, habit engineering, readiness system) in exactly one sentence — not a feature list, and invites them to try one more session with the coaching enabled.
  3. Do not promise features or discuss refunds unless they asked. Do not apologize for the product existing.
  4. Verify the response passes the voice test: direct, human, no corporate filler.
Pass
Response is direct and non-defensive. Differentiator communicated in one sentence. Re-engagement invited. No feature promises. Passes the voice test.
Fail
Response argues with the spreadsheet comparison. Response is defensive. Feature list dumped instead of one differentiator sentence. Voice test fails.
SSIM-14
Phased Rollout Bug Report
10 min Trigger: Report during 50% phased rollout Focus: Rollout management
What We're Testing
Version confirmation from report, correct P2 classification, rollout-continue vs. rollout-pause decision discipline, and response within SLA.
Trigger
New version in 50% phased rollout. User reports: countdown timer flashes during the 3-2-1 start sequence (C-003 — known bug).
Steps
  1. Confirm the user is on the new version — ask for or check the version number from the report. Do not assume.
  2. Check the known issues log — C-003 (timer flash) is documented. Confirm this is the same bug.
  3. Classify severity: P2 (visual bug, non-blocking — does not prevent workout logging). Decision: continue rollout. Document the reasoning.
  4. Draft user response: acknowledge the flash, confirm it's known and being fixed, provide the workaround if one exists.
  5. Define the threshold: if a P1 report comes in from the same new version, escalate to pause-rollout decision with engineering lead.
Pass
Version confirmed within 2 hours. Classification correct (P2). Rollout continues (not paused for a visual-only bug). User response sent within 48 hours. P1 escalation threshold defined.
Fail
Version not confirmed. Rollout paused for a P2 issue (over-response). Response delayed past 48 hours. Escalation threshold for pause not defined.
SSIM-15
New Support Team Member Onboarding Test
20 min Trigger: New team member + 5 mixed reports Focus: Documentation usability
What We're Testing
Whether the documentation alone is sufficient for a new operator to classify, respond, and escalate correctly — without guidance from an experienced team member.
Trigger
New support team member given 5 mixed inbound reports and all support documentation. No prior coaching.
Steps
  1. Hand the new member 5 reports (one each: crash, data loss, feature request, voice complaint, HealthKit issue) and access to all support documentation. No additional context.
  2. New member must: classify each report by type and severity, check the known issues log, draft a response for each, and decide escalation — using only documentation.
  3. Observer times each task from report-received to response-drafted. Note any step where the person had to pause, guess, or ask for help.
  4. Reviewer evaluates: classification accuracy (5 of 5), response quality against the voice guide, and escalation decision correctness.
  5. Any step that required guessing = a documentation gap. Mark it explicitly and fix the documentation before marking this scenario as passed.
Pass
All 5 reports classified correctly. All 5 responses pass the voice test. All escalation decisions correct. Total time under 60 minutes. Zero guesses required.
Fail
Any misclassification, off-brand response, or incorrect escalation — this is a documentation failure, not a people failure. Any step requiring a guess = automatic fail until the documentation is fixed.

crash / data loss / bug / feature request / voice complaint / privacy / sensitive welfare
Severity Classified

Time to Classification
Elapsed time from "report received" to severity classified. Compare against target: P0 ≤30 min, P1/P2 ≤2 hours.
Time to First Response
Elapsed time from "report received" to response drafted and ready to send. Compare against SLA: P0 ≤4h, P1/P2 ≤48h.

Paste the full response draft here, or reference the file path if saved externally. Include the voice check result: pass / fail + notes.

Action Item

Section 5 — LLM Pre-Triage Prompt
Support Triage — LLM Prompt

Paste this prompt into any capable LLM (Claude, GPT-4) with the user report appended. Treat the output as a first draft — verify against the known issues log and run the response through the voice check before sending anything.

System Context
You are a support triage assistant for H.O.N. (Habit Over Numbers), an iOS-only fitness coaching app. H.O.N. is dark-mode, currently in pre-launch TestFlight with App Store submission pending. All user data is stored locally on device — no backend server, no cloud sync. Support channels are App Store reviews and developer email. The H.O.N. voice is spare, direct, identity-affirming. Never cheerleading. Never corporate. Response drafts must sound like one human to another — not a support ticket system. Known issues (match against these before creating new ones): - B-001: Decimal weight overflow (e.g. 95 lbs displays as 95.1151) - B-002: FAIL pill overflows reps field layout - C-003: Countdown timer flashes during 3-2-1 start sequence - HealthKit re-prompting despite prior authorization - Phrase bank repetition (coaching tips feel repetitive, same phrases each session) Voice rules — NEVER use in response drafts: - "Dear valued customer" or any "Dear [name]" opener - "We sincerely apologize for any inconvenience" - "Please do not hesitate to contact us" - "Our team is working hard to..." - Any passive-voice deflection ("it has been noted that...")
User Report Input
[USER REPORT TEXT]
Required Output
1. Issue type: crash / data loss / bug / feature request / voice complaint / privacy / sensitive welfare — one word only
2. Severity: P0 / P1 / P2 / P3 — with one sentence of justification using the matrix (P0 = launch crash or data loss; P1 = core workout flow; P2 = secondary views; P3 = edge case <0.1%)
3. Known issue match: yes (cite the bug ID) / no — if no, explain in one sentence why it doesn't match any known pattern
4. Response draft in H.O.N. voice: direct, human, specific to this complaint, under 150 words. No opener like "Dear" or "Hi [name]". No apology boilerplate. No close like "Please don't hesitate." If sensitive welfare: acknowledge warmly, no medical advice, include one concrete resource.
5. Escalation: yes/no — if yes, specify who (engineering lead / PM / founder) and the one-line reason why
6. Engineering ticket required: yes/no — if yes, one-line ticket description suitable for a backlog
7. Knowledge base gap: yes/no — if yes, suggest a title for the new knowledge base entry that would have answered this question
