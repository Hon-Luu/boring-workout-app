H.O.N. Audit Approach

A framework for evaluating the app — before any finding is written

What This Document Is

This is the audit plan. Not the findings. The plan tells us what questions to ask, in what order, under what conditions, and how to grade the answer. The audit runs those questions against real sessions and produces verdicts. Nothing in this document assumes a finding — it only defines the standards.

The central question of this audit: Does the app know what it knows, say what it means, and use what it has? Three failure modes exist: the app has data and ignores it, the app lacks data and pretends otherwise, or the app has data and interprets it incorrectly. The audit exists to find all three.

What the log is used for: The workout log establishes what data the app actually captures — the fill rate of every field. This tells us which audit questions are live (testable with real data) and which are structural (testable by attempting to produce data). It does not pre-determine findings.

Derived from 18 sessions, 149 sets. Every field that exists in the data model is listed with its actual capture rate. This is the ground truth for what can be tested.

Set-Level Fields

weight / reps
87%Core logging fields. 13% of sets have weight=0 — floor rules apply. 18% of sets have reps=0. Not all zero-reps are errors (warmup sets, skipped sets) but all completed zero-rep sets are semantically contradictory.
targetWeight
50%Only half of sets have a target weight. Pre-fill comes from lastPerformanceCache. If prior session was a zero-rep session, pre-fill inherits a bad target. This is the primary vector for coaching quality degradation.
targetReps
90%Most sets have a target rep count. 10% missing — these sets cannot produce a meaningful rep outcome classification.
isCompleted
84%84% of sets are marked completed. 16% not completed — these are sets that were logged but not executed. How does the analytics engine treat non-completed sets?
rpe
0/149Designed for per-set perceived exertion (6–10). Zero entries. Downstream: ReadinessEngine RPE modifier, ProgramCalibration insight, efficiency metrics — all using null/default. Is RPE entry offered in the UI? Is it prominent enough?
dropWeight / dropReps
29/149Drop sets are being actively used — not a dead feature. 19% of sets have drop data logged. This is significant: these sets have a different analysis path in WorkoutNarrativeEngine (weightState = droppedCompleted / droppedStillUnder).
isDropCompleted
22/14922 drop sets have been completed. Slight discrepancy vs dropWeight count (29) — some drop sets were planned but not completed. Narrative engine has specific paths for completed vs. incomplete drop sets.
toFailure
14/149Used on 9% of sets — another active feature. WorkoutNarrativeEngine has a specific "To Failure" connective modifier. These sets represent the user's hardest efforts and should trigger specific coaching responses.
velocityProfile
0/149Apple Watch rep detection. Zero entries. Either no Apple Watch or the feature isn't being used. If the UI surfaces this, it should explain the requirement. If it doesn't surface it, it's a background feature with no user-facing presence to audit.
What this map tells us: The app has a three-tier data reality. Tier 1 — always captured (time, basic sets, feel). Tier 2 — sometimes captured (drop sets, toFailure, targetWeight). Tier 3 — never captured (readinessBefore, RPE, velocity, notes). The audit must treat each tier differently. Tier 3 gaps are the most consequential because they power the app's most personalized features.

Audit Dimensions

Eight dimensions. Each represents a distinct layer of product quality. They are ordered from foundation to surface — data capture before interpretation, interpretation before output, output before experience.

1
Input Capture
Is the app collecting what it's designed to collect?
Every session · Pre-session focus

This dimension asks whether the designed data collection actually works. Capture failures upstream contaminate every downstream feature. A feature that's designed to use RPE but never gets RPE data is not a coaching feature — it's a logging feature with coaching theater.

Data Correctness
Does the app interpret captured data without introducing errors?
Edge cases · Formulas · Boundaries

Data correctness is about the system's internal logic. Given what was captured, did the app compute the right thing? Failures here are invisible to the user — they trust the number displayed, even when it's wrong.

3
Coaching Output
Does the coaching reflect this session's data, specifically and honestly?
Post-set narrative · Coach notes · Session summary

Coaching is the product's primary promise. The questions here go beyond "is the text correct" to "does it add value specific to this user, this session, this set." Generic coaching that could have been written without any data is a failure of the coaching promise, even if it's technically accurate.

IDQuestionWhat to look forTrigger
D3-Q1Is the coaching specific to this session's data — or is it generic?After each set, read the narrative. Could this exact text have been generated without knowing the weight, reps, exercise, or history? If yes, it's generic. If it references at minimum one of: actual weight, actual reps, trend, history gap — it is specific. Generic coaching is a brand failure regardless of accuracy.every set
D3-Q2Does the coaching contradict any other signal visible in the same session?If feel=Tired, does any coaching output suggest "push hard today"? If readiness is low, does the coach note encourage max effort? If the session had a significant failure (zero reps, incomplete sets), does the coaching celebrate success? One contradiction in a session is a trust-breaking event.every session
D3-Q3Does the coaching voice match the brand criteria?Brand spec: Direct, not cheerleader. Dry, not corporate. Specific, not generic. Shame-free. Check for: exclamation points used as enthusiasm signals, empty affirmations ("Great job!", "Amazing effort!"), corporate hedging ("It's important to remember that..."), gap-shaming ("You've been away for X days"). Any of these is a voice failure.every coaching output
D3-Q4When the session is incomplete or anomalous, does the coaching reflect reality?For a session where sets were not completed, zero reps were logged, or targets were significantly missed: does the coaching output match the qualitative reality of the session? An incomplete session should not receive a "session complete" narrative. A zero-rep set should not receive praise.incomplete or anomalous session
D3-Q5Does the coaching for a drop set use the drop-set specific narrative path?Drop sets are used on 19% of sets. WorkoutNarrativeEngine has specific weightState values (droppedCompleted, droppedStillUnder) and phrase bank keys ("Drop Set — Completed"). Verify that a completed drop set triggers the drop-set narrative branch, not the standard held/increased branch.drop set completed
D3-Q6Does the coaching for a toFailure set use the to-failure modifier?toFailure is used on 9% of sets. WorkoutNarrativeEngine has a "To Failure" connective modifier (phrase IDs 105–114). Verify that a set marked toFailure produces coaching that references the failure attempt, not the standard "hit reps" path.set marked toFailure
D3-Q7Does the coaching change meaningfully as session count grows?Compare the coaching text for the same exercise at: session 1 (historyKey="First Time"), session 2-3 (historyKey="Normal", first trend data), session 8+ (trend data available, possible plateau signal). Is there a qualitatively different depth of insight at higher session counts, or is the output structurally similar regardless of data richness?sessions 1, 3, 10
4
Analytics Integrity
Are the numbers honest — and honest about their own confidence?
Charts · Scores · Trend lines · Confidence

Analytics integrity has two parts: accuracy (are the numbers right?) and honesty (do they communicate their own reliability?). A confidence score shown without its data basis is not analytics — it's speculation dressed as data. The audit asks both questions for every metric.

IDQuestionWhat to look forTrigger
D4-Q1Does the readiness confidence level accurately reflect session count?Confidence specification: <3 sessions = Low, 3–9 = Medium, ≥10 = High. Verify the label transitions at the right thresholds. Is the tilde (~) prefix present only during Low confidence? Does the confidence level's color or styling reinforce its meaning?sessions 1, 3, 10
D4-Q2Are trend charts free of data artifacts from anomalous sessions?For any exercise with known anomalous sessions (zero-rep sessions, floor-weight sessions), check the e1RM trend chart. Do these sessions produce data points at 0 or at the floor value? Do they distort the visible trend line? A valid trend should omit or flag sessions that don't represent actual performance.after any zero-rep or floor-weight session
D4-Q3Are comparison windows labeled with their actual timespan?Any chart or card that says "vs X weeks ago" or "over the last Y days" — verify the claim. If the user has less history than the stated window, does the label adjust? If the baseline falls back to first session, is that stated? Mislabeled time windows are a data honesty failure, not a display bug.any progress card with a time comparison
D4-Q4Do EmergentInsights only surface when their prerequisite data exists?For each insight type, identify its data prerequisite (feelRating, HealthKit sleep, weight history, RPE, etc.). If that prerequisite is empty or null, what does the insight show? A specific correlation or state shown with no data is worse than a placeholder — it's a fabrication.every Progress/Home open
D4-Q5Do plateau and momentum signals match the actual e1RM trajectory?Pick any exercise flagged as "plateau" or "best mover." Calculate the e1RM slope manually using the session data. Does the engine's classification match the math? Plateau: slope <0.5 kg/wk over ≥4 weeks with ≥3 sessions. Momentum: positive slope, above threshold. Verify the threshold and window are applied correctly.after ≥3 sessions of same exercise
D4-Q6Is the Composite Strength Score isolated from data quality artifacts?The CSS has three pillars: Level (35%), Momentum (40%), Process (25%). If anomalous sessions (zero-rep, target mismatch) contribute to Momentum, the score is contaminated. Check: does removing the anomalous sessions change the CSS significantly? If so, the score is data-quality sensitive.after ≥5 total sessions
D4-Q7Is volume calculated the same way in every view that displays it?Weekly volume chart, session summary, volume spike modifier in ReadinessEngine, overload scoreboard — do they all use effectiveWeight, or do some use raw weight? A user who sees "3015 kg" in one place and "2800 kg" in another for the same session loses trust in both numbers.any high-volume session
5
Cross-Feature Coherence
Do different parts of the app tell the same story?
Tab-to-tab · Metric consistency · Vocabulary

A user navigates between tabs expecting the same facts in each. If the readiness score on Home differs from the one in Trainer plans, or if "Intermediate" in Progress means something different than "Int" in Insights, the user can't build a mental model. Coherence failures erode trust even when each individual feature is correct in isolation.

The plan intensity and the home score must be internally consistent.every app open
D5-Q2Is strength tier terminology consistent across all views?Find every place in the app that labels a strength tier (Progress, Insights, any chart, any coaching note). D5-Q4Does the e1RM value for an exercise agree between Progress and Insights?Pick any exercise. Note its e1RM in the Progress overload scoreboard. Open the Insights tab and find the same exercise. Progressive Value
Does the app become meaningfully more valuable as sessions accumulate?
Session 1 vs 10 vs 18 · Milestones · Feature unlocks

A fitness app that shows the same interface on session 1 and session 50 has failed its core promise. The data should compound into progressively richer insights. This dimension tests whether the app's depth is visible — and whether users are guided toward it as their data grows.

IDQuestionWhat to look forTrigger
D6-Q1On session 1, is the experience coherent with no prior data?Every metric and insight should either show a valid value (even if low confidence) or an honest "not enough data yet" state. Specifically: does the app try to show trend lines, momentum, or plateau risk with 0 or 1 sessions? Does confidence = Low and is that communicated?session 1
D6-Q2Are confidence thresholds communicated when they are crossed?Confidence transitions at sessions 3 (→Medium) and 10 (→High) represent genuine improvements in data reliability. Does the app acknowledge these crossings? An uncelebrated confidence upgrade is a missed retention moment — the user doesn't know their data just became meaningful.sessions 3, 10
D6-Q3At what session count do each of the major analytics features produce useful output?Map the minimum session count for: e1RM trend line (≥2 sessions of same exercise), plateau detection (≥3 sessions, ≥4 weeks), momentum classification (≥3 sessions), EmergentInsights (varies per insight), Composite Strength Score (≥1 valid session). Are features presented before they have enough data? If yes, they should show honest empty states, not guessed values.any feature-first exposure
D6-Q4Do placeholder states tell the user specifically what's needed to unlock them?For any feature showing an empty or placeholder state, what does the message say? "Collecting data" is not actionable. "Complete 3 sessions of this exercise to see your trend" is. "Connect Apple Health to see sleep impact on your performance" is. Specificity in placeholder states is both a UX and a trust question.any empty state encountered
D6-Q5Is the session 18 experience qualitatively richer than session 5?Open the same features after session 5 and after session 18. Is there more data in the charts? Are more insights available? Is the coaching more specific? If the experience is structurally identical, data accumulation isn't producing compounding value for the user.sessions 5, 18
7
Honesty & Transparency
Does the app tell users what it doesn't know as clearly as what it does?
Data labeling · Terminology · Assumption communication

An app built on data has an obligation to be honest about the limits of that data. Presenting estimates as facts, mislabeling derived values, or drawing conclusions from insufficient data are integrity failures. This dimension is about the gap between what the app knows and what it claims.

IDQuestionWhat to look forTrigger
D7-Q1Are estimates labeled as estimates?The tilde (~) prefix on the readiness score during Low confidence is the designed mechanism. Are there other places in the app where estimated values are presented as measured values? Trend slopes, projected e1RM, EmergentInsight states derived from HRV proxies — all are estimates. Are they labeled as such?any metric displayed
D7-Q2Is terminology accurate and not misleading?Profile field labeled "Muscle Mass %" — is this skeletal muscle mass or lean body mass? These are different values with different normal ranges. Any label that uses a technical term with a commonly understood meaning must use that term correctly, or redefine it in context.any labeled field
D7-Q3When a transformation is applied to data, is it visible?Three transformations apply silently: effectiveWeight doubling for dumbbells, floor weight for barbells/EZ bars, and counterweight inversion for assisted exercises. When these apply, does the user see the transformed value or the raw value? Do they know a transformation occurred?any dumbbell, barbell, or assisted exercise
D7-Q4Are there features that present specific insights based on data that doesn't exist?List every feature that requires data from: HealthKit (sleep, HRV, RHR, VO2 Max), readinessBefore, RPE, weight history, cardio sessions. For each, verify what happens when the data is absent. Does the feature produce a genuine insight, an honest placeholder, or a fabricated one?any insight card
D7-Q5Does the app explain why it's making a recommendation?For any actionable recommendation (increase weight, deload, take rest, switch exercise), can the user trace the recommendation to its data basis? Or is it opaque — "you should increase weight" with no visible reason why? An app that gives recommendations without showing the reasoning is asking for blind trust.any recommendation surfaced

Return & Retention
Does the app earn the next session?
Gap handling · Milestone messaging · Habit loop

The fitness app that succeeds is not the one with the best analytics — it's the one the user comes back to. This dimension asks: what is the experience of returning after time away? Does the app make the next session feel possible, or does it make the absence feel like failure?

IDQuestionWhat to look forTrigger
D8-Q1Does the first thing the user sees after a gap communicate return, not absence?After any gap ≥3 days: what is the first element on the Home screen? Is it the readiness score (forward-looking), a streak count (backward-looking), a gap penalty indicator, or a welcome-back message? The difference between "Readiness: 69 — good day to train" and "You haven't trained in 5 days" is the entire product philosophy.gap ≥3 days
D8-Q2Is any language in the app gap-punishing?Scan every text output after a gap return: coaching notes, HONHabitEngine messages, streak indicators, progress trend language. Does any of it reference the absence negatively? Key test phrases: "you've been away," "you haven't trained in X days," "your streak is broken," "you're losing momentum." Any of these fails the core brand promise.gap ≥3 days
D8-Q3Does the readiness score reflect a gap as recovery, not penalty?After a 5-day rest: the engine applies −4 for the day-gap modifier and potentially +5 for recentFreq ≥4. The net effect should be moderate readiness, not low readiness. More importantly: does the coaching note describe the gap as recovery time ("Your muscles had time to rebuild") or as drift ("Your fitness is declining")?gap ≥4 days
D8-Q4Are there meaningful retention moments beyond streak tracking?The HONHabitEngine generates milestone and pattern messages. Verify: does the app produce any milestone acknowledgment at session 10 (confidence: High)? Does it acknowledge a strong training week? Does it surface the user's best progression (Lat Pulldown slope) as a motivating data point? Or is the only retention mechanism a streak counter?sessions 3, 10, 18+
D8-Q5Does the post-session experience (feel rating, narrative summary) close the loop for the next visit?After completing a session: what is the last thing the user sees before leaving the app? Does it tease the next session ("Next time, try 20 kg on DBP"), contextualize what happened ("Your strongest session in two weeks"), or just close? Grading Framework

Verdict Definitions

  • Pass — The app behaves as designed and as the user would reasonably expect. No follow-up required.
  • Watch — Technically correct but creates a subtle trust, UX, or coherence risk. Not a failure today, but a compounding problem over time. Document and monitor.
  • Fail — The app produces incorrect output, misleads the user, contradicts the data, or violates the product promise. Requires a finding and a fix.
  • N/A — The trigger condition for this question was not present in this session.

Severity Classification (for Fails)

  • P0 — Data integrity: The app stores or displays incorrect data that flows into other features. Requires immediate fix. Affects all downstream calculations.
  • P1 — Trust breach: The app says something that contradicts what the user can observe or know. Coaching that praises a failed session. A number that doesn't add up. A label that's wrong. Erodes trust on each occurrence.
  • P2 — Brand violation: The app's voice or framing contradicts the product identity (cheerleader language, gap-shaming, generic coaching). Cumulative erosion of brand differentiation.
  • P3 — UX friction: The app's design makes it harder than it should be to capture good data or consume insights. Includes discoverability, affordance, and label accuracy issues.

What Makes a Finding

A finding requires three things: (1) the question that was asked, (2) what the app actually did, and (3) what the app should have done. A finding without all three is an observation, not a finding. Every finding must be reproducible — it must be possible to point to the exact user action, the exact screen, and the exact output that produced it.

Data from the workout log can be cited as context but is not itself a finding. "The log shows 0/18 readinessBefore values" is context. "Starting a session, completing one set before answering the readiness prompt causes the prompt to disappear — verified by [specific reproduction steps]" is a finding.
