01 · Executive Brief

What This App Is

H.O.N. is a strength-first training companion that replaces the coach-in-your-pocket model with a data-in-your-pocket model. Its thesis: give the user honest numbers, not motivational fluff, and they will make better decisions.

The value loop is: Log → Analyse → Advise → Return. Everything routes through SeedStore, analytics rebuild on background thread, and coaching language is generated fresh each session. The stack is sophisticated — PCSA-weighted strength scoring, fatigue-adjusted e1RM, physiological decay model, emergent cross-domain insights — but the interface must make this legible to someone who just wants to know if they should go heavy today.

The Central Tension

The app is built for a technically confident, science-interested user who nonetheless has beginner training data (18 sessions). Many of the most powerful features — EmergentInsightEngine, PCSA-weighted composites, allometric scaling — require a data density the user hasn't yet produced.

The risk: a user who opens the Insights tab on session 5 sees sophisticated-looking cards filled with placeholders or misleading low-confidence estimates. This is more damaging to trust than showing nothing. The audit asks: does the app honestly reflect what it knows vs. what it doesn't?

02 · User Profile (Derived from Log)

Who This User Is

Training age: Beginner — 18 sessions over 36 days. No prior log data. First session Apr 20.

Consistency: 0.5 sessions/day on active weeks. Trains in bursts: 3-4 days on, 4-5 days off. Longest gap: 5 days (Apr 27–May 1, and May 21–26).

Equipment access: Full gym. Uses dumbbells, barbells, cable machines, Smith machine, leg press, assisted pull-up machine. Not constrained to home equipment.

Program structure: Loosely PPL. Push days (DBP, shoulder press), Pull days (lat pulldown, seated row), Leg days (squat, leg press). Rarely mixes patterns within a session.

Characteristic behavior: Logs incomplete sessions. 3 sessions with zero-rep completed sets. Double-sessions on May 13 and May 21. Logs Assisted Pull-Up as 2 sets and finishes. Does not fill readinessBefore. Does not enter RPE.

What the Data Tells Us the App Cannot See

Real-world readiness: Feel ratings exist (Tired on Apr 29, May 13; Strong on most others) but readinessBefore = null for all 18 sessions. The engine's most personalized modifier never fires.

Body composition: muscleMassPercent = 79.85 (= 100 − BF%). This is lean mass, not skeletal muscle. The app stores and displays it as "Muscle Mass %" — a mislabeled field. Actual skeletal muscle for a 66.7kg male at 20.2% BF would be ~38–42% (27–28kg).

Equipment convention: User logs "25kg" for Incline DB Press. Whether this means 25 per hand or 25 total is never established. The app silently doubles it. If the user's mental model is "total weight," all dumbbell analytics are 2× inflated.

Session naming: 14 of 18 sessions are "Hon's Routine." Two sessions on May 13 and two on May 21 share names, making history review ambiguous in the Progress tab.

03 · App Mental Model

This section answers: When the user opens the app, what do they believe they are looking at? Where does their mental model diverge from the system's internal model?

User's Assumed Mental Model

  • 📱 Home tab = "How am I doing today?" Dashboard of readiness + what to do
  • 🏋️ Workout tab = "Log my session." Sets, weights, timer.
  • 📊 Progress tab = "Am I getting stronger?" History + charts.
  • 🔍 Insights tab = "What does the app know about each exercise?" Per-exercise deep dive.
  • ⚙️ Settings = "My profile and preferences."

Reality (System's Actual Model)

  • 📱 Home = HomeCache (pre-computed on bg thread): readiness score, progressTrend, todayHints, exerciseNotes. Re-built every time analytics refresh fires. Contains 14+ insight types the user has no visibility into.
  • 🏋️ Workout = ActiveWorkoutView + ReadinessPromptCard (shown before first set, gone after) + WeightSuggestionSheet (on appear if pending suggestions) + narrative engine firing post-set + PR detection comparing live e1RM to exerciseHistoryCache.
  • 📊 Progress = 6 CollapsibleDashSections, ProgressChartsViews (3032 lines), insights computed fresh on every render (not cached), overload scoreboard with <6-wk fallback, plateau risk detection, momentum chips, population percentile.
  • 🔍 Insights = ExerciseInsightsView listing all logged exercises → ExerciseInsightDetailView with per-exercise PCSA-weighted score, e1RM trend, relative strength tier. StrengthLabView (PCSA global composite) accessible from within this tab.
  • ⚙️ Settings = Body composition (mislabeled Muscle Mass %), training location, insight level, fatigue decay override, REST timer, weight unit, coaching nudge frequency, reminder time, backup/restore.

Where Mental Models Diverge

Divergence 1 — Trainer tab is missing from nav

CLAUDE.md and presumably the product roadmap includes a "Trainer" tab generating AI workout plans. In production this is absent from the main TabView. WorkoutPlanEngine exists and TrainerTabView is a complete view — but it is inaccessible from the primary navigation. Users who rely on AI workout suggestions have no clear entry point.

Divergence 2 — Readiness prompt can be bypassed invisibly

The ReadinessPromptCard appears only when readinessBefore == nil && !hasAnyCompletedSets. The moment the user completes any set, the prompt disappears permanently for that session. A user who opens a workout and taps "complete" on the first set before rating has silently bypassed the entire readiness data capture — with no confirmation, no error, no warning. 0/18 sessions captured confirms this is happening.

Divergence 3 — Dumbbell weight convention is implicit

The app doubles all dumbbell entries (effectiveWeight × 2). A user who enters "20" intending "20kg total" (one dumbbell in each hand at 10kg) will have their analytics computed against 40kg. A user entering "20" intending "20kg per hand" will be correct. No label clarifies which convention is assumed. The weight entry field shows no unit hint or per-hand indicator.

Divergence 4 — Plan randomization on every Trainer open

WorkoutPlanEngine.generatePlans() is called inside the TrainerTabView body using randomElement() with no seed. Every time the Trainer view appears, the three plans change. A user who likes plan A, navigates away, then returns will see a different plan A. This makes the "Trainer" feel unreliable and undermines planning intention.

Divergence 5 — muscleMassPercent is lean mass, not muscle

The Settings field is labeled "Muscle Mass %" with placeholder "e.g. 40" and range 10–65. The stored value is 79.85 (= 100 − 20.2% BF). This value is outside the UI's documented valid range yet is stored and displayed. Any screen showing "Muscle Mass: 79.9%" is presenting anatomically impossible data as fact.

04 · Journey Phases

Evaluated from the user's perspective at the time — not with hindsight about what the data would eventually show.

1
Cold Start — Building the Habit Loop
Sessions 1–3 · Apr 20–23 · 3 days · Volume: 842 / 1080 / 430 kg
Confidence: Low Readiness: ~46 / 68 / 80

What the user experiences

The onboarding collects name, body weight, age, and training location over 3 pages. Page 2 is skippable ("Skip for now"). After onboarding, an AppearanceChoiceSheet asks dark/light preference. The user sees the Home tab with a readiness card, likely showing low confidence and a score around 46 — reflecting no prior training data. The coaching note will be generic ("Your readiness estimate is low confidence — build a pattern first").

First session: ReadinessPromptCard appears before sets begin. If the user logs a set before answering, the prompt disappears. After each set, WorkoutNarrativeEngine generates coaching text — for a first-ever session, historyKey = "First Time" which routes to specific onboarding phrases.

By session 3, the user has 3 sessions — Confidence transitions from Low to Medium. The Home tab begins to show more meaningful data. The ExerciseInsightsView starts listing exercises but per-exercise analytics require ≥3 sessions of the same exercise — not yet met for anything except potentially DBP (only 2 sessions by Apr 23).

Consultant Assessment

Onboarding doesn't teach the readiness prompt

The 3-page onboarding collects baseline data but never explains that the pre-workout feel check matters. When the ReadinessPromptCard appears on session 1, users who dismiss it have no context for what they're skipping. Zero of this user's 18 sessions captured it.

First readiness score (≈46) may create a wrong first impression

Score 46 on session 1 is technically correct — no training history means the engine can't verify freshness. But a user opening the app for the first time and seeing "Readiness: 46 / 100" may interpret this as "the app thinks I'm not ready to train." The coaching note must contextualise this precisely. A generic low-confidence note here does damage.

historyKey "First Time" is correct

WorkoutNarrativeEngine routes to "First Time" phrases for all exercises on session 1. This is the right path — no comparison possible, so onboarding phrases fire. Confirm these phrases don't attempt to reference previous performance.

Progress tab: what does a user with 3 sessions see?

After S3 (Apr 23), the user opens the Progress tab. "Where Am I" section (expanded by default): population percentile requires relStrength — which needs bodyWeight AND ≥1 valid e1RM per exercise. With 3 sessions and only 2 DBP sessions, percentile may render with a caveat. "Am I Getting Stronger" starts collapsed. A new user is unlikely to discover it.

2
Habit Formation — Patterns Emerge
Sessions 4–9 · Apr 27 – May 7 · 11 days · Volume: 500 / 1200 / 570 / 978 / 1234 / 350 kg
Confidence: Medium Readiness: ~64–76

What the user experiences

This is the phase where the app should start earning trust. By session 6, the user has trained 6 times — enough for Lat Pulldown and DBP to have 2-3 sessions each. The WorkoutNarrativeEngine begins using historyKey "Normal" (within 7 days) and starts comparing weights. Progressive suggestions begin appearing.

The problem sessions: S6 (Apr 29) feel=Tired after 3 consecutive training days. S9 (May 7) Machine Shoulder Press all-zero reps — 3 completed sets with 0 reps each. This is the first major data quality event. If the engine handles it gracefully, trust is built. If it celebrates a 0-rep session, trust breaks.

5-day gap before S8: User returns May 6 after 5 days. The readiness score should be ≈64 (4-6d gap = −4). But the user reported feel=Strong. The coaching note should acknowledge the gap and welcome return without shame — per the brand principle "Never punish gaps."

Consultant Assessment

Machine Shoulder Press S9 — most critical test of data trust

May 7: 3 sets × 10kg × 0 reps, all isCompleted=true. StrengthAnalyticsEngine's first filter excludes reps<1. But if the first filter returns empty, the fallback we.completedSets.filter { eq.effectiveWeight($0.weight) > 0 } runs — which includes these sets. effectiveWeight(10) = 10 > 0. So they are included via fallback. e1RM formula with reps=0 returns 0. The session contributes e1RM=0 to the trend chart — a visual data error.

WorkoutPlanEngine uses recentlyTrainedRegions from last 2 entries

After S6 (pull: Lat Pulldown + Row) and S7 (push: DBP + Shoulder Press + Cable), the engine's recentlyTrainedRegions = {back, arms, chest, shoulders}. Primary plan for S8 would route to Legs or Core — correct. But randomElement() means the user might see a different plan each time they check. If they open Trainer, navigate away to check something, and come back: plan has changed. Undermines the "today's recommendation" premise.

5-day return framing — brand test

S8 (May 6): user returns after 5-day gap, feel=Strong. Readiness ≈64 (Moderate). The coaching note must not say "you've been away too long." Per USER_RESEARCH.md: "Never punish gaps — the return is celebrated, not the streak." Confirm the 5-day gap note uses shame-free language. This is a core brand promise.

S6 feel=Tired with consecutive days — coaching contradiction risk

Apr 29: 3rd consecutive training day. Readiness ≈68 (1-day gap +8, but consecutive-3 has no penalty until day 4). User reports Tired. If the coaching note says "Go hard today" while the user feels Tired, the user learns not to trust the app. The score (68 = Moderate) doesn't capture that this is day 3 of 3.

3
Analytics Maturation — App Starts Knowing You
Sessions 10–14 · May 11–18 · 8 days · Volume: 890 / 800 / 934 / 1215 / 600 kg
Confidence: High Readiness: ~62–68

What the user experiences

Session 10 (May 11) is the 10th session — Confidence transitions to High. This is a product milestone. Does the app acknowledge it? Per USER_RESEARCH.md: "Which HON message library kinds resonate most over 6 months?" — a session-count milestone message here would validate the habit loop. HONHabitEngine tracks totalSessions and can generate milestone messages.

Same-day double sessions (May 13): S12 (Lat Pulldown + Row, feel=Tired) followed by S13 (Incline DB + Low Row + Cable OTE, feel=Strong) on the same day. Two "Hon's Routine" entries. Analytics must correctly separate these. The readiness for S13 is computed after S12 — the 5-consecutive-day streak penalty (−14) fires, making S13's expected readiness ≈62. But user feels Strong.

DBP crisis (May 18, S14): 17.5kg × 8 reps (Set1), then 0 reps (Set2), 0 reps (Set3). The user stopped after one set. This is qualitatively a failed session. The narrative must reflect this. The weight wasn't increased — all 3 sets at the same 17.5kg. WeightState = held. RepOutcome = underSignificant (8/24 = 33%) if zero-rep sets included, or onTarget (8/8 = 100%) if they're excluded — a binary difference in coaching quality.

Consultant Assessment

Session 10 milestone — does HONHabitEngine fire?

HONHabitEngine.onSessionLogged() checks totalSessions and can generate milestone messages. Verify that crossing into "High" confidence produces a user-visible message — either an in-app banner or a coaching note that references the data density. "10 sessions in — your data is now reliable enough to track real trends" aligns with the brand voice. A missed milestone here is a missed retention moment.

DBP May 18 — the zero-rep verdict determines narrative quality

If zero-rep guard exists: 1 completed set at 8/8 = 100% → onTarget. User receives "Solid session" narrative for what was a 1-of-3 completion. If no guard: 8/24 = 33% → underSignificant. Narrative correctly reflects failure. The guard behavior here is anti-correctness. An app that can't tell the difference between "you completed 1 set and stopped" vs "you completed all 3 sets" is not a coaching app — it's a logging app.

Machine Shoulder Press target-jump pattern (S10)

S10 (May 11): MSP target pre-filled at 10kg (from zero-rep S9 history). User enters 20kg × 8/8/6. WeightState = userIncreasedHit. The narrative should celebrate this — but the jump was a correction of a data error, not genuine progressive overload. The engine has no way to know the difference. Risk: user gets "You crushed it — new weight incoming" praise for recovering from a logging mistake.

Overhead Press floor on May 12 — display coherence

S11 (May 12): OHP logged at 0kg. effectiveWeight floors to 20kg (barbell). e1RM computed against 20kg. The StrengthAnalyticsEngine correctly uses effectiveWeight. But what does the ExerciseInsightDetailView display for this session — "0 kg" (raw) or "20 kg" (floor)? If 0kg, the chart has a misleading data point. If 20kg, the user sees a weight they don't recall entering.

4
Optimization — First Plateau Signal & Volume Peak
Sessions 15–18 · May 19–21 · 3 days · Volume: 3015 / 1410 / 450 / 800(?) kg
Confidence: High Readiness: ~58–76

What the user experiences

Volume spike on May 19 (S15): 3015kg — the highest single-session volume in the log by 2.4×. Smith Machine Squat × 4 sets, Smith RDL × 3 sets, Leg Press × 3 sets, MSP × 3 sets. This is the session most likely to trigger the volume spike modifier (−10) for S16.

Consecutive streak (May 19–21): 3 consecutive training days, with May 21 having two sessions. The consecutive modifier fires: May 21 = day 4 (−8), potentially day 5 if same-day counts. The user is building into a fatigue hole.

Incline DB plateau signal (S17, May 21): 25kg × 3/3/3. This is the 3rd Incline DB session. May 13 = 20kg × 8 (e1RM ≈ 25.3kg/hand). May 21 = 25kg × 3 (e1RM ≈ 27.5kg/hand). The e1RM did increase — but only because the weight went up and reps fell. The actual strength progression is unclear. This is a data richness problem, not a plateau yet.

S18 Assisted Pull-Up (May 21): 2 sets, 1-minute session. effectiveLoad = bodyWeight − assist. This is the app's most complex single calculation — and it happens with the least context.

Consultant Assessment

S16 (May 20) readiness — spike penalty must visibly affect the score

S15 volume = 3015kg. Median of prior sessions ≈ 900kg. Ratio ≈ 3.35× — far above the 1.8× threshold for a −10 penalty. S16 readiness should reflect this: expected ≈58 (68 + 8 (1d gap) − 10 (spike) − 8 (4th consecutive day)). If the displayed score is ≈76 (just 1d gap, spike not applied), the engine is missing the largest training stimulus in the dataset. Verify by checking S16's displayed readiness in the app.

S18 volume calculation is inconsistent with stored value

S18 Assisted Pull-Up: Set1 = 15kg assist × 5 reps, Set2 = 10kg assist × 5 reps. effectiveLoad = bodyWeight − assist = 51.75kg and 56.75kg respectively. Correct volume = 51.75×5 + 56.75×5 = 542kg. Raw weight volume = (15+10)×5 = 125kg. The stored session summary shows "800kg" — this matches neither calculation. A phantom volume number is being stored or displayed.

May 26 — 5-day rest state (today)

Last training was May 21. Today is May 26. The user is 5 days out. Expected readiness ≈69: 68 + (−4, 5-day gap) + (+5, 4 sessions in last 7 days). Intensity suggestion: Moderate. WorkoutPlanEngine should route to a Moderate primary plan. The recentlyTrainedRegions from last 2 sessions (S17=Incline DB/chest, S18=Assisted Pull-Up/back) should steer the plan toward Legs or Push. This is a good "return from rest" scenario to verify.

HONHabitEngine drift detection — 5 days since last session

HONPatternEngine.isDrifting() is called on app foreground and runs at most once per calendar day. After 5 days of inactivity, this should have fired. Verify that the user sees a return-from-gap message — and that it uses shame-free framing per USER_RESEARCH.md. A "you've been away" message that feels like a scolding breaks the core product promise.

06 · Scenario Bank

40 scenarios across 8 categories. Each specifies: trigger, what to observe, pass/fail criterion grounded in real session data.

A · Onboarding & First Use

S-A01First app open — onboarding collects useful dataRetention

Trigger: Fresh install, hasCompletedOnboarding = false.

Observe: 3-page flow: Welcome+Name → Baseline (bodyWeight, age, training location) → Ready. Page 2 has "Skip for now" button.

Pass

Name and bodyWeight captured. The strength tier calculation requires bodyWeight — skipping page 2 means the app operates without percentile ranking for first sessions. The "Skip" option is labeled clearly and the consequence is explained ("strength comparisons need your body weight").

Fail

"Skip for now" exits onboarding with no bodyWeight. First open of Insights tab shows all relStrength cards in a "Set body weight in Settings" placeholder state — with no path back to Settings visible from that screen.

S-A02First readiness score — framing for a beginnerCritical UX

Trigger: S1 (Apr 20). No prior sessions. Expected score ≈46.

Observe: Home readiness card shows "~46" (tilde for Low confidence). Coach note for a score of 46 with confidence=Low and no training history.

Pass

Coach note says something like "Your first few sessions build the baseline — we'll refine this as we learn your pattern." Score of 46 is contextualized as a data-thin estimate, not a verdict on the user's fitness.

Fail

Coach note says "Your readiness is low. Consider resting or light movement." — treating a data-estimation artifact as a coaching directive. User who opens the app for the first time and sees "low readiness" before training session 1 may feel the app is already working against them.

Observe: Does the ReadinessPromptCard disappear? Is readinessBefore = null for the session?

Pass

The prompt persists regardless of set completion, OR it converts to a non-dismissible sheet that requires an answer before the first set can be logged. readinessBefore is captured for the session.

The ReadinessPromptCard should use the .interactiveDismissDisabled(true) pattern like FeelSelectorSheet (AD-008).

08 · Systemic Findings

What the App Gets Right

Data architecture

SeedStore with background analytics, token-based debouncing, and HomeCache pre-computation is correctly designed. Performance caches avoid per-render log scans. The isLoaded gate prevents mid-state flash. These are production-quality patterns.

Strength scoring sophistication

PCSA-weighted composites, allometric scaling, physiological decay model with a 14-day no-decay window, OLS regression for slopes — the analytics engine is methodologically correct and science-cited. For a user who grows into the app, this is a genuine differentiator.

PR detection guard

prevBest > 0 prevents false PRs on first sessions. The PR banner with accessibility announcement is a well-implemented delight moment.

FeelSelectorSheet uses interactiveDismissDisabled

Per AD-008, the feel rating at session end requires explicit action. This is the correct UX pattern. It contrasts badly with ReadinessPromptCard, which uses an inline card that disappears — the inconsistency suggests ReadinessPromptCard was added later without applying the established pattern.

Shame-free return framing in USER_RESEARCH.md

The design principle "Never punish gaps — the return is celebrated, not the streak" is explicitly documented and correctly reflected in ReadinessEngine gap modifiers (5-day gap = −4, not catastrophic). The brand intention is right. The execution needs verification.

What Needs Work

Zero-rep completed sets — root cause unfixed

10 zero-rep completed sets across 6 sessions. No UI validation prevents this. The fix is a single guard in the complete-set action: if reps == 0, show "Did you mean to skip this set?" before isCompleted = true. This single fix resolves A-01, B-01, B-02, B-03, D-02, D-06.

ReadinessPromptCard bypass — architectural mismatch

The prompt disappears after the first set. 0/18 captures. The fix: convert ReadinessPromptCard to a sheet with .interactiveDismissDisabled(true), matching FeelSelectorSheet. This is a 1-hour fix with outsized impact on engine personalization.

WorkoutPlanEngine randomization — no seed

Plans change on every render. Fix: seed with date (Calendar.current.dateComponents of today) so plans are stable per day. A user's "today's plan" should mean something for the whole day.

muscleMassPercent mislabeled and out-of-range

Fix: rename field to "Lean Body Mass %" and update placeholder to "e.g. 78–82" (realistic lean mass range). Add a footer note: "Lean body mass = 100% − body fat %. This differs from skeletal muscle mass (~40–50% of body weight)."

Progress tab section ordering

Fix: set expandedStrong = true by default and expandedWhere = false. The most actionable data ("Am I Getting Stronger") should be the first thing a returning user sees. Population percentile is interesting but not decision-relevant for a 18-session user.
